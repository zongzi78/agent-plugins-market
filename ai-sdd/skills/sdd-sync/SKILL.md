# sdd-sync（文档同步）

## 定位

将 change 中的代码变更智能合并回 `.ai/doc/` 主文档。

工作流程：分析实际代码 diff → 判定每份文档的影响 → 生成变更方案 → 暂停等待人类确认 → 执行。

## 前置检查

1. 无活跃 change → 提示"请先通过 `/sdd-propose` 创建 change"，终止。
2. plan.md 的 `## 文档影响评估` 为空 → **不终止**。提示"plan.md 未声明文档影响，sync 将基于代码 diff 自动分析"，继续执行。

## 执行流程

### 步骤 1：分析

1. **读取 apply 的变更摘要**（plan.md 末尾「变更摘要（apply 生成）」章节）
   - 这是主要输入——apply 知道自己改了什么符号和文件
   - 如果摘要不存在 → 从 plan.md 任务清单的 checkbox 状态 + 任务描述推断变更范围
2. **读取 explore 的影响预测**（plan.md `## 文档影响评估` 表）
3. **读取项目代码验证**（仅在需要确认具体细节时）：
   - 打开变更涉及的源文件，用符号锚点定位具体变更位置
   - 如项目有 git 且 Agent 判断 `git diff` 可辅助快速定位，可作为可选手段使用
4. **符号锚点反向索引** — 对每个变更符号，在 `.ai/doc/` 中搜索引用该符号的位置

### 步骤 2：判定

1. 按文档角色判定每个符号的影响：
   - 涉及模块职责/结构变化 → `00-架构`
   - 涉及用户可见行为 → `01-行为目录`
   - 涉及技术选择/架构模式 → `02-决策记录`
   - 涉及模块内部实现细节 → `03-详细设计`
   - 修复了已知问题条目或者可能引入新问题 → `04-问题与改进`
2. 将判定结果与 explore 的影响预测对比：
   - 一致 → 在提案中标注"✓"
   - explore 未预测到 → 标注"⚠️ 新增发现"
   - explore 预测了但代码中未变更 → 标注"⚠️ 未变更"

### 步骤 2.5：约束影响检查

读取 `.ai/supplement-rules.md` 项目特定约束节（如存在）：

1. **新约束模式**：本次变更是否引入了需要固化为约束的模式？
   → 如有，在提案摘要中提示人类："本次变更引入了以下潜在的约束模式，如需固化为正式约束，请编辑 .ai/supplement-rules.md 项目特定约束节：..."
2. **约束过时**：本次变更是否使现有约束（启用=是）过时？
   → 如有，标记 ⚠️："以下约束可能因本次变更而过时：[编号] — [描述]"
3. **约束违反**：本次变更是否违反了现有约束（启用=是）？
   → 如有，标记为高风险 ⚠️，列举违反的约束编号

### 步骤 3：提案

1. 在 plan.md 中补充/完善 `## 文档变更方案（sync 分析生成）` 章节
2. 格式：

   ```
   ### 00-架构
   · 无变更（或具体变更内容）

   ### 01-行为目录
   · ADDED: B-017 [名称] — 行为列表新增
   · MODIFIED: B-003 [名称] — 入口变更

   ### 02-决策记录
   · ADDED: D-015 [描述] — 设计级

   ### 03-详细设计/[模块名]
   · MODIFIED: [具体变更]

   ### 04-问题与改进
   · DELETED: CD-011（[原因]）

   ### ⚠️ 与 explore 预测的差异
   · explore 未预测到 02-决策记录 需新增条目
   · 原因：[具体原因]
   ```

3. **暂停**，向人类展示摘要：

   ```
   本次代码变更共涉及 X 个符号，映射到 Y 份文档的 Z 处修改：

     01-行为目录    ADDED   B-017 [名称]
     03-[模块名]   MODIFIED [具体变更]
     04-问题与改进  DELETED  CD-011 [原因]

   ⚠️ 与 explore 预测差异 N 处：
     - explore 未预测到 02-决策记录 需新增条目（apply 阶段选择了 GBK 编码方案）

   请确认是否全面，或输入补充/修改指令。
   ```

### 步骤 4：执行

1. 人类回复后执行文档修改（支持："全面" / "补充[具体]" / "修改[具体]"）
2. 逐文件按确认后的方案修改 `.ai/doc/`
3. 质量自检（4 项）：
   - [ ] 没在末尾追加（创建心态：通读后从整体结构考虑插入位置）
   - [ ] 内容属于该文件的角色
   - [ ] 没有"讨论了A和B选了A"（不放实现过程）
   - [ ] 变更方案与 explore 预测的差异已记录
4. **尊重文件角色**：同步前读取目标文档顶部的「📋 本文档角色」区块。
   - 只写入"应该写"范围内的内容
   - 发现越界内容 → 不覆盖，标记为需人工拆分
   - 完整角色定义见各模板顶部

5. 更新 `.ai/project-log.md`

## 联动设计

- **拒绝路径**：无活跃 change → 提示"请先通过 `/sdd-propose` 创建 change"
- **完成后引导**：`✅ 文档已同步。接下来可通过 /sdd-archive 归档本次 change。`
- **如无需同步**：`ℹ️ 本次 change 未涉及文档变更，无需同步。可直接运行 /sdd-archive 归档。`
