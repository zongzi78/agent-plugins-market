# AI-SDD 术语与概念体系

> 本文解决当前体系中概念混乱、中英混杂、Agent 无法正确执行指令的问题。

---

## 一、文档标记体系

AI-SDD 在三个层面使用标记来追踪信息状态。三层分别回答不同问题：

| 层次 | 回答的问题 | 适用范围 | 取值 |
|------|-----------|---------|------|
| **来源** (source) | 这些信息从哪里来的？ | 文档级别 (front matter) | 逆向工程 / 正向设计 |
| **可信度** (confidence) | 这个结论有多可靠？ | 文档级别 (front matter) 或 逐条声明级别 | 已确认 / 推断 / 推测 |
| **生命周期** (lifecycle) | 这个功能还在用吗？ | 仅 01-需求.md 的功能条目 | 活跃 / 疑似废弃 / 已废弃 |

三层独立，互不重叠。一个功能可以同时是"活跃 + 推断"，也可以是"已废弃 + 已确认"。

## 二、逐层定义

### 2.1 来源（source）

| 取值 | 中文显示 | 含义 |
|------|---------|------|
| `reverse-engineering` | 逆向工程 | 文档由 AI 从代码反推生成，描述代码实际做了什么 |
| `human-design` | 正向设计 | 文档由人类或人类+AI 共同设计，描述系统应该做成什么样 |

**互操作规则**：
- 来源在文档创建时确定，一般不随确认流程改变
- 例外：当逆向文档的全部内容经人类确认后，来源可改为 `human-design`（但这不是强制要求——保留 `reverse-engineering` 来源以诚实记录文档产生方式也是合理的）

### 2.2 可信度（confidence）

| 取值 | 中文显示 | 含义 | 判定标准 |
|------|---------|------|----------|
| `confirmed` | 已确认 | 经过人类验证，可作为后续开发基线 | 人类明确确认了该项内容 |
| `inferred` | 推断 | 从代码模式合理推断，但未经人类确认 | 能从代码中找到证据链，但可能误解设计意图 |
| `speculative` | 推测 | 基于有限证据的猜测，准确性低 | 代码证据不完整或存在矛盾 |

**升级规则**：
- `inferred` → `confirmed`：人类在确认环节明确标记为"确认"
- `speculative` → `inferred`：Agent 找到额外代码证据支撑后（在后续会话中），可先升级为 inferred；但最终仍需人类确认
- `speculative` → `confirmed`：不走捷径——必须经过人类确认

**降级规则**：
- 任何已确认内容，如果代码发生了未经文档同步的变更 → 降级为 `inferred`，在 project-log 中记录
- 主动降级由 `ai-sdd-check` 检测触发，Agent 不应自行降级

**过期规则**：
- 标记为 `speculative` 的内容，如果在 3 个月内未被人类确认或升级，应在 project-log 的"待确认"列表中标注为"长期未确认"

### 2.3 生命周期（lifecycle）——仅 01-需求.md 功能条目

| 取值 | 中文显示 | 含义 | 判定标准 |
|------|---------|------|----------|
| `active` | 活跃 | 功能正在被使用 | 有完整实现链路（路由→处理→数据访问），且被调用 |
| `suspected-dead` | 疑似废弃 | 可能已不再使用，但不确定 | 路由注册了但 handler 是空壳；长期无 commit 涉及；配置中关闭 |
| `deprecated` | 已废弃 | 经人类确认已不再使用 | 人类在确认环节明确标记为"废弃"，或代码中有明确的 @Deprecated 标注 |

## 四、文档的 status 状态机

所有 `.ai/doc/` 文件的 front matter 包含 `status` 字段。完整状态机：

| 状态 | 中文含义 | 进入条件 | 退出条件 |
|------|---------|---------|---------|
| `draft` | 草稿 | 文档创建时 | 内容填写完成 |
| `reviewed` | 已审阅 | 至少经过一轮人类确认 | — |
| `baseline` | 基线 | 所有内容 confidence 为 `confirmed`，可作为开发基线 | 代码变更导致内容过时 |
| `deprecated` | 已废弃 | 内容不再适用或已被替代 | — |

区分 `source` 和 `status`：
- `source` 回答"最初怎么来的"（逆向工程 / 正向设计），创建后基本不变
- `status` 回答"当前处于什么状态"（草稿 / 已审阅 / 基线 / 已废弃），随流程推进变化

## 五、术语统一对照

以下为本次改造后的术语规范。所有 skill 和模板统一使用**中文优先**，首次出现时括号备注英文。

| 概念 | 规范用词 | 废弃用词 |
|------|---------|---------|
| confidence | 可信度 | — |
| `confirmed` | 已确认（可信度） | CONFIRMED（全大写英文） |
| `inferred` | 推断（可信度） | INFERRED |
| `speculative` | 推测（可信度） | SPECULATIVE |
| lifecycle | 生命周期 | — |
| `active` | 活跃 | active |
| `suspected-dead` | 疑似废弃 | suspected-dead |
| `deprecated` | 已废弃 | deprecated |
| source | 来源 | — |
| `reverse-engineering` | 逆向工程 | reverse |
| `human-design` | 正向设计 | greenfield / human-design |
| Given/When/Then | 前置条件/操作/预期结果 | GWT / Given/When/Then |
| 版本控制 | 版本控制 | Git / VCS（泛指时用"版本控制"） |

**规则**：
1. 面向 Agent 的指令和文档正文：使用中文规范用词
2. front matter 和模板变量：保留英文 key（兼容已有系统），但值尽量用中文
3. 首次出现时可括号备注英文对照

## 六、设计原则（解决内部矛盾）

### 6.1 "规范是唯一真相来源"的正确解释

原 supplement-rules.md 1.3 节的不变量 #2 和 3.6 节的冲突处理规则存在字面矛盾。修正为：

> **规范是唯一真相来源**意味着：系统开发的决策以规范文档为准。但这不意味着文档永远正确。
>
> - **正向项目**：文档描述"应该是什么" → 代码必须符合文档
> - **逆向项目**：文档描述"代码实际做了什么" → 当代码与文档不一致，意味着代码发生了文档未记录的变更，此时文档过时，需要**以代码为准更新文档**，而非以文档为准修改代码
>
> 这条不变量的正确理解是：**决策权在文档**，文档说了算。但文档需要诚实地反映其自身的可靠性（通过 `confidence` 和 `source` 标记）。

### 6.2 "精确性 > 完整性"的正确解释

- 不确定的内容必须诚实标注（"待确认"，并说明原因），不能为了凑完整性而编造
- 但这**不意味着**可以跳过整个章节——已知部分必须填写完整
- 完整性自检清单的作用是确保"没有遗漏已知信息"，而不是强迫 Agent 编造未知信息

## 七、可信度标记的统一表达规则

所有文档中，可信度的表达遵循统一规则。

### 逐条标记

| 可信度 | 标记 | 含义 |
|--------|------|------|
| 已确认（confirmed） | 不标记 | 默认状态——代码直接可验证或经人类确认 |
| 推断（inferred） | ⚠️ | 从代码模式合理推断，但未经人类确认 |
| 推测（speculative） | ❓ | 基于有限证据的猜测，准确性低 |

### 文档级汇总

front matter 的 `confidence` 值按聚合规则：
- 全部 confirmed → `confirmed`
- 存在 speculative → `speculative`
- 否则 → `inferred`

### 各文档的具体标记位置

| 文档 | 标记位置 | 方式 |
|------|---------|------|
| 00-架构 | 认知边界三子表 | 单元格文字中标注或 ⚠️/❓ 前缀 |
| 01-需求 | 功能列表 `可信度` 列 | 仅标 ⚠️/❓，已确认留空 |
| 02-技术选型 | `选型理由` 列 | "已确认：xxx" 或 "⚠️ 推断：xxx" |
| 03-详细设计 | 数据模型 `置信度` 列 + 接口契约 | `置信度` 列仅标 ⚠️/❓；接口描述中 ⚠️/❓ 前缀 |
| 04-问题与改进 | 各条目描述 | 逆向模式默认 inferred；人类确认过的条目标注"已确认" |
