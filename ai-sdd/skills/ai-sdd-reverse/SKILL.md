# 遗留项目逆向化（ai-sdd-reverse）

## 定位

将没有文档的遗留代码库接入 AI-SDD 体系，从代码反推生成 `.ai/doc/` 规范文档。

**核心原则**：逆向文档 = 现状还原，不是理想设计。回答"系统当前长什么样"，不是"系统应该长什么样"。

**相关概念**：本文涉及的概念（可信度、生命周期、来源、确认标记等）的完整定义见 `skills/ai-sdd-reverse/术语与概念体系.md`（本 skill 目录下的配套参照文档）。

---

## 前置检查

### 检查 1：.ai/ 目录
- 检查 `.ai/` 目录是否存在
- 不存在 → 提示"项目尚未初始化 AI-SDD，请先执行 `/ai-sdd-init`"，终止

### 检查 2：.ai/doc/ 是否已有内容
- 检查 `.ai/doc/` 下是否已有非空的 md 文件
- 有 → 输出警告：

```
⚠️ .ai/doc/ 目录已有文档。逆向化将覆盖现有内容。
   - 已有文件：[列出]
   是否继续？（继续 / 取消 / 增量模式——只补充缺失的文档，不覆盖已有）
```

- 等待人类选择后，按选择执行

### 检查 3：适用范围判断
- 向人类提问：

```
在开始逆向化之前，请评估一下：
未来 3 个月内，这个项目预计有几次实质性改动？
  A) ≥3 次 — 值得做完整逆向化
  B) <3 次 — 建议不做完整逆向，改做一次性探索分析
  C) 不确定 — 但我想做
```

- 人类选 B → 说明"维护模式下成本大于收益，建议用探索性会话做一次性分析"，如人类坚持则继续
- 人类选 A 或 C → 继续

## 文档质量要求

以下要求在整个逆向流程中生效。每完成一个阶段的产出，你必须执行自检。

### 1. 通用规则

**正向规则（必须做到）**：
1. 以真实代码为准——不猜测、不假设，一切以代码实际行为为依据
2. 创建心态——当作首次编写，通读代码后从头组织内容，不在末尾追加
3. 尊重文件角色——每个 doc 文件只写属于它的内容（角色定义见下表）
4. 结构化表述——用表格、列表、Mermaid 图，不用大段叙述
5. 标注可信度——每个关键声明标注 已确认 / 推断 / 推测
6. 提供增量价值——提炼代码中不直观的信息，不复述代码本身

**负向规则（严禁行为）**：
1. 禁止使用 "..." 占位符——填充每一项，或填"待确认"并注明原因
2. 禁止粘贴大段原始代码——用结构化描述代替，附带 `文件:行号` 引用
3. 禁止以"精确性优先"为由跳过整个功能/API/章节——不确定也要列出，标注"推测"
4. 禁止将推断包装为事实——推测内容必须明确标注
5. 禁止猜测业务含义——代码行为与命名不一致时，记录"代码行为与命名不一致"，不自行解释

**"精确性优先"的正确理解**：
- ✅ 正确：不确定的字段标注"待确认" + "为什么不确认" + "建议谁可以补充"
- ❌ 错误：因为一个字段不确定就跳过整个 API
- ✅ 正确：该列出的接口、表、字段穷尽列出——只有其中不确定的细节才标注
- ✅ 正确：简单模块自然产出短文档（因为确实没什么可写）
- ❌ 错误：复杂模块产出几十行的短文档（以"少而准"为借口）

### 2. 文件角色对照表

| 文件 | 角色 | 不该出现的内容 |
|------|------|--------------|
| `00-架构.md` | 模块划分、依赖关系、数据流、外部依赖、技术债务 | 需求细节、API 契约、实现代码 |
| `01-需求.md` | 功能列表（F-XXX）、功能描述、验收标准、行为约束 | 实现细节、API 路径、数据库字段 |
| `02-技术选型.md` | 技术栈总览、关键依赖清单、版本漂移分析、构建/测试/部署 | 需求描述、业务逻辑 |
| `03-详细设计/xxx.md` | 数据模型、API 契约、核心业务流程、异常/并发/性能分析 | 需求动机、全局架构、选型理由 |

### 3. 产出自检清单

各文档的具体产出要求已分散到对应阶段的执行步骤中，Agent 在产出文档后立即执行自检。此处不重复列出。

## 执行流程

### 阶段 0：版本控制探测

**目标**：判断项目是否可访问版本控制历史。

**执行步骤**：
1. 按以下顺序检测版本控制系统：
   - 检测 `.git/` 目录 → 执行 `git log --oneline -1` 验证可用
   - 检测 `.svn/` 目录 → 执行 `svn log -l 1` 验证可用
   - 检测 `.hg/` 目录 → 执行 `hg log -l 1` 验证可用
2. 记录探测结果：
   ```
   版本控制：<类型>（可用 / 不可用）
   - 可用时：记录最新提交信息（摘要 + 日期）
   - 不可用时：记录"本项目无可用版本控制历史"
   ```
3. 不论结果如何，继续执行。后续阶段中，凡是提到"版本控制历史分析"的步骤，仅在探测结果为"可用"时执行。

**产出**：无文件产出。探测结果记入内存，供后续阶段使用。

**不暂停**。

---

### 阶段 1：宏观理解

**目标**：了解项目全貌。

**执行步骤**：

1. 向人类收集 5 个信息：

```
在开始代码分析之前，请帮我了解以下项目背景：
  1. 这个项目是干什么的？（一句话）
  2. 给谁用的？（内部/外部，大概多少用户）
  3. 核心技术栈（如果你知道的话）：语言/框架/数据库？
  4. 当前最头疼的问题是什么？
  5. 最近想在这个项目上做什么？
```

2. 人类回复后，执行以下扫描：
   - 扫描项目根目录及一级子目录，理解目录结构
   - 从构建文件（package.json / pom.xml / requirements.txt / Cargo.toml / go.mod 等）识别技术栈
   - 从目录命名和包结构识别顶层模块划分

3. 将阶段 0 探测结果和本阶段信息汇总为项目概述。

**产出**：将项目概述写入 `.ai/project-log.md` 的变更时间线，作为本次逆向化首条记录。

**不暂停**。

---

### 阶段 2：架构梳理

**目标**：还原系统架构。

**执行步骤**：

1. **分析模块依赖**：扫描 import/include 语句、包结构、模块调用链，识别模块间依赖关系

2. **识别数据流向**：追踪关键业务流程的调用链路

3. **识别外部依赖**：从配置文件、连接代码、docker-compose 等中提取外部系统和中间件

4. **绘制依赖图**：用 Mermaid 绘制模块依赖图
   - 每个模块必须是独立的节点
   - 按层级分组（表示层 / 业务逻辑层 / 数据访问层 / 基础设施层）
   - 箭头方向表示依赖关系（A → B 表示 A 依赖 B）
   - 如存在循环依赖，如实画出并在技术债务中说明

5. **扫描技术债务**：搜索以下模式
   - TODO / FIXME / HACK / XXX / OPTIMIZE 注释
   - 废弃 API 的使用
   - 被注释掉的代码块
   - 硬编码的 URL / IP / 端口 / 凭证
   - 魔数（超时值、缓冲大小、业务常量等不合理硬编码的值）
   - 空 catch 块 / 忽略返回值
   - 每个发现附带具体 `文件:行号`

6. **搜索架构约束违反**：
   - 循环依赖
   - 跨层直接调用
   - 防腐层绕过
   - 描述每个发现：位置(`文件:行号`)、违反了什么、可能的原因、是否有计划修复

7. **填充认知边界地图**（三子表）：
   - 已知已知表：通过阅读具体代码确认的事实（每条带 `文件:行号` 证据）
   - 已知未知表：AI 无法确定、需要人工补充的内容（写具体问题，不写泛泛的"架构不清楚"）
   - 推断结论表：从代码模式推断但未确认的设计决策（标注 推断 或 推测）

**产出**：`.ai/doc/00-架构.md`（基于模板生成）

**产出后自检**（生成完成后立即逐项核对）：
- [ ] 所有第一级模块已列出，每个含职责描述和复杂度概要（代码规模/公开接口数/依赖数）
- [ ] Mermaid 依赖图包含所有模块节点
- [ ] 端到端数据流至少覆盖核心业务闭环（最少 1 条读路径 + 1 条写路径；仅 1 条时说明原因）
- [ ] 全部外部依赖已列出，含故障影响说明
- [ ] 技术债务表已填写（搜索了 TODO/FIXME/HACK/废弃API/硬编码等关键词），无债务时写"当前未识别明显技术债务"并说明判断依据
- [ ] 认知边界地图三子表已填写（已知已知 / 已知未知 / 推断结论）
- [ ] 无 "..." 占位符

**产出后暂停**。输出以下确认清单：

```
📋 阶段 2 确认清单 — 架构梳理

请逐一检查以下内容，如有问题请直接说明（如"模块 X 不对，应该是 Y"或"A 和 B 应该合并"）：

□ 模块列表是否完整？有无遗漏？
□ 依赖关系图是否准确反映了实际的 import/include 关系？
□ 外部依赖是否完整？
□ 认知边界地图中的"已知未知"是否需要补充或修正？

另外，如果有任何补充信息（比如某模块的历史背景、为什么做某个设计决策等），请直接告诉我，我会记录到文档中。
```

**等待人类回复后，按以下分支处理**：

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 | 将文档中所有 `推断`/`推测` 内容更新为 `已确认`；更新 front matter `status: reviewed`、`last_updated` |
| 某内容需要修改 | 按人类指示直接修改文档；修改后的内容标记为 `已确认` |
| 补充了新信息 | 追加到对应章节，标记为 `已确认` |
| 有内容不确定/跳过 | 对应内容保持原可信度标记；更新 front matter `last_updated` |
| 人类指出了遗漏 | 补充分析，补全文档，重新输出确认清单 |
| 人类未回复/跳过 | 所有内容保持原可信度；继续下一阶段 |

**完成后更新 front matter**：
- `last_updated`: 今天日期
- `status`: `reviewed`（如果人类确认）或保持 `draft`（如果未回复）
- `confidence`: 按整体规则计算（全部已确认 → `confirmed`；存在推测 → `speculative`；否则 `inferred`）

### 阶段 3：需求还原

**目标**：从代码反推功能列表。

**执行步骤**：

1. **从 API 路由提取功能**：扫描路由注册代码，列出所有对外接口 → 按业务场景归类为功能组

2. **从数据库 Schema 提取数据域**：列出数据实体，关联到对应功能

3. **从配置文件提取外部依赖**：识别集成的外部服务

4. **版本控制历史辅助分析**（仅阶段 0 确认可用时执行）：
   - 运行 `git log --oneline -n 100`（或 `svn log -l 100` / `hg log -l 100`）
   - 通过提交信息辅助判断：哪些功能是核心功能（高频 feat）、哪些可能已废弃（长期无变更 + 无外部入口）

5. **为每个功能确定生命周期**：
   - 活跃：有完整调用链路（路由 → Handler → Service → DAO）
   - 疑似废弃：路由注册但 handler 是空壳 / 长期无变更 / 配置中关闭
   - 命名但不猜测——不确定时标注"疑似废弃"而非"已废弃"

6. **为每个功能的每个关键声明标注可信度**：
   - 已确认：行为在代码中明确可见，有具体 `文件:行号` 引用
   - 推断：从框架默认值、命名约定推断，但未直接看到配置或显式声明
   - 推测：基于有限证据的猜测，需要人类验证

7. **搜索隐式约束**（以下 4 个维度）：
   - 并发：锁、原子操作、事务隔离级别、幂等性设计
   - 错误处理：错误码体系、重试策略、降级逻辑、超时配置
   - 性能：缓存策略、连接池、N+1 查询风险、资源密集点
   - 边界：输入校验规则、硬编码限制、配置驱动的行为分支
   - 每个发现附带 `文件:行号` 引用

8. **为每个功能写验收标准**：
   - 至少 1 条正常路径 + 至少 1 条错误/边界路径
   - 格式：`给定 [前置条件]，当 [操作]，则 [预期结果]`
   - 逆向模式：从代码中的参数校验、错误处理、状态码推断验收标准

9. **为每个功能写行为约束**：从代码中的校验逻辑、事务边界、中间件守卫提取行为约束（如"订单金额必须 ≥ 0"），标注可信度和代码位置

10. **需求漂移检查**：比较"命名/配置暗示的预期行为"与"代码实际行为"，标记不一致

**产出**：`.ai/doc/01-需求.md`（基于模板生成）

**产出后自检**（生成完成后立即逐项核对）：
- [ ] 每个可识别功能组对应一个 F-XXX 条目
- [ ] 每个功能含验收标准（正常路径 ≥1 条，错误/边界路径 ≥1 条）
- [ ] 每个功能标注了生命周期（活跃 / 疑似废弃 / 已废弃）
- [ ] 每个功能标注了可信度（已确认 / 推断 / 推测）和代码证据
- [ ] 隐式约束章节覆盖了并发/错误处理/性能/边界 4 个子表
- [ ] 需求漂移检查表已填写

**产出后暂停**。输出以下确认清单：

```
📋 阶段 3 确认清单 — 需求还原

请对每个功能标记你的决定。可直接在功能编号后标注，如"F-001 确认"、"F-003 废弃"、"F-005 需澄清——这个功能我不确定还在不在用"

功能列表：
[列出所有 F-XXX 编号 + 功能名称 + 当前生命周期标记]

快捷标记：
  [确认] — 这个功能确实存在且在（或应该）使用
  [废弃] — 这个功能已经不用了
  [需澄清] — 我不确定，需要进一步调查（请说明不确定什么）

另外：
□ 关键业务规则的描述是否符合实际？
□ 隐式约束（并发/幂等/错误处理/性能/边界）是否有遗漏？
□ 标注推测的内容是否需要补充或修正？
```

**等待人类回复后，按以下分支处理**：

| 人类标记 | 你的操作 |
|---------|---------|
| `[确认]` | 该功能可信度 → `confirmed`；生命周期 → `active` |
| `[废弃]` | 该功能生命周期 → `deprecated`；可信度维持原值；不在后续开发中引用 |
| `[需澄清]` | 功能尾部追加 `<!-- 待确认：人类标记需澄清 — YYYY-MM-DD。问题：[人类描述] -->`；可信度维持原值 |
| 人类补充了业务信息 | 追加到对应功能描述，标记为 `已确认` |
| 人类要求修改 | 按指示改文档，改后标记为 `已确认` |
| 人类指出了遗漏功能 | 分析遗漏的代码区域，补充该功能条目，标记为 `推断`，重新输出确认清单 |

**处理完成后**：
- 更新 front matter：`last_updated`、`confidence`（按整体规则计算）、`status: reviewed`（如果人类确认了大部分）
- 如果有 `[需澄清]` 的功能，在 project-log.md 的"待确认"列表中追加记录

### 阶段 4：技术栈确认

**目标**：记录当前技术选型。这是"现状盘点"，不猜测选型动机。

**执行步骤**：

1. **分析依赖**：从包管理器文件中提取所有直接依赖项及其版本
2. **版本漂移评估**：每个关键依赖对比当前版本 vs 包管理器可获取的最新稳定版，标注差距和迁移复杂度
3. **使用模式分析**：每个技术组件标注是标准用法还是定制/非标准用法
4. **构建/测试/部署验证**：从实际配置文件中提取工具链信息，不假设标准工具
5. **技术债务识别**：过时依赖、不一致编码模式、硬编码配置、缺乏测试的模块
6. **提取"缺失决策理由"列表**：列出从代码中无法推断但值得记录的技术选择（如"为什么选 PostgreSQL 而不是 MySQL？代码无法推断"）
7. **技术约束与反模式**：记录代码中隐含的技术禁止事项和已出现的反模式

**产出**：`.ai/doc/02-技术选型.md`（基于模板生成）

**产出后自检**（生成完成后立即逐项核对）：
- [ ] 所有直接依赖及版本已列出，含证据来源
- [ ] 每个关键依赖标注了使用模式（标准用法 / 非标准用法）
- [ ] 构建/测试/部署配置已验证实际文件路径
- [ ] 版本漂移分析表已填写
- [ ] 缺失决策理由列表已填写

**关于可信度**：
- 本阶段产出的信息分为两类：
  - **直接可验证的事实**（如依赖名称、版本号、配置文件路径）→ 直接标记为 `已确认`
  - **推断性内容**（如使用模式评估、版本迁移建议、反模式归类）→ 标记为 `推断`

**产出后暂停**。输出以下确认清单：

### 阶段 5：详细设计还原

**目标**：还原各模块的详细设计。

**核心原则**：深度自适应——每个模块都独立生成详细设计文件。复杂模块自然产出长文档，简单模块自然产出短文档。不强制统一深度。

**执行步骤**：

对每个一级模块（00-架构.md 中已列出的模块），生成一个详细设计文件。

每个文件包含以下章节：

1. **概述**（必填）：
   - 核心职责的一句话精确定义
   - 在系统中的定位
   - 负向限制：本模块绝对不做什么（有多少写多少，通常 2-5 条；如果确实只识别出 1 条就写 1 条并注明"仅从代码中识别出此一条"）

2. **核心数据模型**（必填）：
   - 列出所有持久化字段，穷尽不省略
   - 每个字段标注可信度和代码证据(`文件:行号`)
   - 实体间关系文字说明
   - 关键数据约束

3. **对外接口契约**（必填，每个公开 API）：
   - 方法 + 路径
   - 请求参数（所有参数，含类型/必填/默认/验证规则）
   - 成功响应格式
   - 所有已知错误码及触发条件
   - 副作用说明（读写哪些全局状态）
   - 幂等性设计说明

4. **核心业务流程**：
   - ≥1 个 Mermaid 序列图，使用实际函数名/模块名
   - 每步附带异常处理说明
   - 工具类/纯计算模块如无复杂流程，写明"本模块为无状态工具模块，无复杂流程"并省略图

5. **状态机**（如适用）：如果模块管理有生命周期流转的实体，列出完整状态机

6. **异常处理与容错**（必填）：
   - 已知异常域
   - 处理策略表
   - 降级兜底逻辑
   - 如果模块确实没有异常处理代码，写"未发现异常处理代码，存在安全隐患"并标记为技术债务

7. **并发安全**（必填）：
   - 并发风险点
   - 锁与事务机制
   - 如果模块无共享状态，写"本模块为无状态工具模块，不涉及共享状态，无并发风险"

8. **性能特征**（必填）：
   - 核心路径复杂度
   - 缓存策略（如有）
   - 资源密集点
   - 如果模块非常简单（纯计算、无 I/O），写明原因

9. **已知问题与陷阱**：在本模块代码中发现的具体问题

10. **模块接口图**：对外提供和依赖消费的接口一览

**产出**：`.ai/doc/03-详细设计/<模块中文名>.md`
- **文件名必须优先使用中文**（如"用户认证模块.md"、"订单管理模块.md"），取自模块在系统中的实际人类称呼；专有名词/技术组件名可用英文（如"Redis缓存模块.md"）

**产出后自检**（每个模块文件生成后立即逐项核对）：
- [ ] 概述含核心职责精确定义 + 负向限制（≥2 条；少于 2 条时说明原因）
- [ ] 核心数据模型的关键字段穷尽列出
- [ ] 全部公开 API 的契约完整（含副作用和幂等性说明）
- [ ] ≥1 个 Mermaid 业务流程图（使用实际函数/模块名；工具/纯计算模块如无复杂流程可省略并写明原因）
- [ ] 异常处理/并发安全/性能特征三个专题全部填写（不适用时写"不适用"并原因，不跳过）
- [ ] 文件名使用中文（如"用户认证模块.md"），取自模块在系统中的人类可理解名称

**产出后暂停**。输出以下确认清单：


## 大项目分阶段执行

对于大型项目（模块数 >10），可分 4 个子任务执行：

| 子任务 | 覆盖 | 产出 |
|--------|------|------|
| 子任务 1 | 阶段 0-2 | 版本探测 + 宏观理解 + 00-架构.md |
| 子任务 2 | 阶段 3 | 01-需求.md |

| 子任务 4 | 阶段 6 | 补充完善 |

每个子任务独立会话完成，确认点作为子任务边界。

## 增量模式

当人类指定某个模块/目录（如"逆向文档化 auth 模块"）：

- 跳过阶段 0-1
- 阶段 2：只分析该模块与其余部分的依赖关系
- 阶段 3：只为该模块涉及的功能范围生成/更新 01-需求.md
- 阶段 5：只为指定模块生成详细设计
- 阶段 4 和 6：正常执行（全项目技术栈始终是有用的）
- 不重新创建 .ai/ 目录
