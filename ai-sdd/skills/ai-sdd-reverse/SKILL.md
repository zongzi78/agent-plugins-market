# 遗留项目逆向化

## 定位

将没有文档的遗留代码库接入 AI-SDD 体系，从代码反向生成 `.ai/doc/` 规范文档。

## 前置检查

1. 检查 `.ai/` 是否已存在
   - 如不存在：先执行 `/ai-sdd-init` 流程（创建 .ai/ 目录和基础文件）
2. 检查 `.ai/doc/` 是否已有内容
   - 如已有内容：警告"doc/ 目录已有文档，逆向化可能覆盖"，AskUserQuestion 确认
3. **适用范围判断**：向人类提问——"未来 3 个月内预计有几次实质性改动？"
   - 如果 <3 次：建议不做完整逆向化，用探索会话做一次性分析即可
   - 如果 ≥3 次：继续执行以下流程
   - 如果人类坚持要做（不管几次）：尊重决定，继续

## 执行流程

### 阶段 1：宏观理解

**目标**：了解项目全貌。

**执行**：

先向人类收集 5 个核心信息：
1. 这个项目是干什么的？（一句话）
2. 给谁用的？（内部/外部，用户量级）
3. 核心技术栈（如知道）：语言/框架/数据库
4. 当前主要痛点是什么？
5. 近期目标是什么？

收集完毕后，执行：
- 扫描项目根目录及一级子目录，识别目录结构
- 从 package.json / pom.xml / requirements.txt / Cargo.toml / go.mod 等识别技术栈
- 从目录命名和包结构识别顶层模块划分

**产出**：项目概述，写入 `.ai/project-log.md` 的变更时间线首条记录。

本阶段不需要人类确认。

---

### 阶段 2：架构梳理

**目标**：还原系统架构。

**执行**：
1. 分析模块间依赖关系（import/require 语句、包结构、模块调用链）
2. 识别数据流向
3. 识别外部依赖和集成点
4. 用 Mermaid 绘制模块依赖图
5. 标注技术债务

**产出**：`.ai/doc/00-架构.md`（使用模板）

**产出后暂停**，等待人类确认：模块划分是否正确？有没有遗漏？

---

### 阶段 3：需求还原

**目标**：从代码反向推导功能需求。

**执行**：
1. 从 API 路由推导对外功能
2. 从数据库 Schema 推导数据域
3. 从配置文件推导外部依赖
4. 标注可能废弃的功能
5. 为每个功能使用 RFC 2119 关键词描述行为
6. 为每个功能附带验收标准（Given/When/Then 格式）

**产出**：`.ai/doc/01-需求.md`（使用模板）

**产出后暂停**，等待人类确认：哪些功能还在用？哪些已废弃？

---

### 阶段 4：技术栈确认

**目标**：记录当前技术选型（现状陈述，不是推荐）。

**执行**：
1. 分析所有关键依赖的名称和版本
2. 识别技术债务具体清单
3. 记录构建/测试/部署现状

**产出**：`.ai/doc/02-技术选型.md`（使用模板）

本阶段不需要人类确认。

---

### 阶段 5：详细设计还原

**目标**：还原关键模块的详细设计。

**执行**：
1. 对每个关键模块/业务域：
   - **数据模型**：表格描述
   - **API 契约**：标准格式
   - **核心业务流程**：Mermaid 流程图
   - **状态机**（如有）
2. 按业务域组织（每个域一个文件）
3. 用结构化描述，不用大段代码

**产出**：`.ai/doc/03-详细设计/<模块名>.md`（每个业务域一个文件，基于 `_模板.md`）

**产出后暂停**，等待人类确认：关键设计描述是否准确？

---

### 阶段 6：补充完善

**目标**：完成所有收尾工作。

**执行**：
1. 在 `.ai/project-log.md` 中记录已知问题和技术债务
2. 补充"待同步文档"列表
3. 确保所有 doc 文件有正确的 YAML front matter
4. 如有可从代码推断的决策，记录到 `.ai/changes/`

本阶段不需要人类确认。

---

## 大型项目分阶段执行

| 子任务 | 覆盖阶段 | 产出 |
|--------|---------|------|
| 子任务 1 | 阶段 1-2 | project-log.md 首条记录 + 00-架构.md |
| 子任务 2 | 阶段 3 | 01-需求.md |
| 子任务 3 | 阶段 4-5 | 02-技术选型.md + 各模块详细设计 |
| 子任务 4 | 阶段 6 | 补充完善 |

## 增量模式

当用户指定某个模块/目录时（如"逆向文档化 auth 模块"）：
- 跳过阶段 1（已有宏观理解）
- 只对指定范围执行阶段 2-5
- 阶段 2：只分析该模块与其余部分的依赖关系
- 阶段 5：只为指定模块产出详细设计
- 不重复创建 .ai/ 目录和 project-log.md

```
  .ai/doc/00-架构.md
  .ai/doc/01-需求.md
  .ai/doc/02-技术选型.md
  .ai/doc/03-详细设计/<模块名>.md（×N）
```

## 质量规则

1. **以真实代码为准**——不猜测，不假设，一切以代码实际行为为准
2. **创建心态**——当作首次编写，通读代码后从头组织内容，不在末尾追加
3. **尊重每个文件的角色**——01-需求.md 写"做什么"，02-技术选型.md 写"为什么选这个"，03-详细设计/ 写"怎么做"
4. **不放实现过程**——规范文档是"结果"，不是"讨论记录"
5. **结构化表述**——用表格、列表、Mermaid，不用大段叙述
6. **精确性 > 完整性**——不确定的标注"待确认"，不要瞎编
