# 遗留项目逆向化（ai-sdd-reverse）

## 定位

将没有文档的遗留代码库接入 AI-SDD 体系，从代码反推生成 `.ai/doc/` 规范文档。

**核心原则**：逆向文档 = 现状还原，不是理想设计。回答"系统当前长什么样"，不是"系统应该长什么样"。

---

## 前置检查

### 检查 1：.ai/ 目录
- 检查 `.ai/` 目录是否存在
- 不存在 → 提示"项目尚未初始化 AI-SDD，请先执行 `/ai-sdd-init`"，终止

### 检查 2：.ai/doc/ 是否已有内容
- 检查 `.ai/doc/` 下是否已有非空的 md 文件
- 有 → 输出警告：

```
⚠️ .ai/doc/ 目录已有文档。逆向化将覆盖现有内容。
   - 已有文件：[列出]
   是否继续？（继续 / 取消 / 增量模式——只补充缺失的文档，不覆盖已有）
```

- 等待人类选择后，按选择执行

### 检查 3：适用范围判断
- 向人类提问：

```
在开始逆向化之前，请评估一下：
未来 3 个月内，这个项目预计有几次实质性改动？
  A) ≥3 次 — 值得做完整逆向化
  B) <3 次 — 建议不做完整逆向，改做一次性探索分析
  C) 不确定 — 但我想做
```

- 人类选 B → 说明"维护模式下成本大于收益，建议用探索性会话做一次性分析"，如人类坚持则继续
- 人类选 A 或 C → 继续

## 文档质量要求

以下要求在整个逆向流程中生效。每完成一个阶段的产出，你必须执行自检。

### 1. 通用规则

**正向规则（必须做到）**：
1. 以真实代码为准——不猜测、不假设，一切以代码实际行为为依据
2. 创建心态——当作首次编写，通读代码后从头组织内容，不在末尾追加
3. 尊重文件角色——每个 doc 文件只写属于它的内容（角色定义见下表）
4. 结构化表述——用表格、列表、Mermaid 图，不用大段叙述
5. 标注符号锚点——每个关键声明附带代码符号锚点（ClassName::methodName() 或 filename.cpp → functionName()），有锚点 = 来源明确，无锚点 = 推断待确认
6. 提供增量价值——提炼代码中不直观的信息，不复述代码本身

**负向规则（严禁行为）**：
1. 禁止使用 "..." 占位符——填充每一项，或填"待确认"并注明原因
2. 禁止粘贴大段原始代码——用结构化描述代替，附带符号锚点引用（ClassName::methodName() 或 filename.cpp → functionName() 格式）
3. 禁止以"精确性优先"为由跳过整个功能/API/章节——不确定也要列出，标注"待确认"并说明原因
4. 禁止将推断包装为事实——不确定的内容必须明确标注
5. 禁止猜测业务含义——代码行为与命名不一致时，记录"代码行为与命名不一致"，不自行解释

**"精确性优先"的正确理解**：
- ✅ 正确：不确定的字段标注"待确认" + "为什么不确认" + "建议谁可以补充"
- ❌ 错误：因为一个字段不确定就跳过整个 API
- ✅ 正确：该列出的接口、表、字段穷尽列出——只有其中不确定的细节才标注
- ✅ 正确：简单模块自然产出短文档（因为确实没什么可写）
- ❌ 错误：复杂模块产出几十行的短文档（以"少而准"为借口）

### 2. 文件角色对照表

| 文件 | 角色 | 不该出现的内容 |
|------|------|--------------|
| `00-架构.md` | 模块划分、依赖关系、数据流、外部依赖 | 需求细节、API 契约、技术债务、架构违规 |
| `01-行为目录.md` | 系统可观察行为清单（B-XXX） | 实现细节、数据域、行为约束、需求漂移 |
| `02-决策记录.md` | 技术栈事实 + 所有层级决策理由集（D-XXX） | 需求描述、业务逻辑、反模式、版本风险判断 |
| `03-详细设计/xxx.md` | 数据模型、API 契约、核心业务流程、异常/并发/性能 | 需求动机、全局架构、模块陷阱 |
| `04-问题与改进.md` | 架构违规、需求漂移、功能缺陷、版本与依赖风险、模块陷阱、设计债务、代码债务 | 需求描述、设计理性、实现细节 |

### 3. 产出自检清单

各文档的具体产出要求已分散到对应阶段的 sop/ 子文件中，Agent 在产出文档后立即执行自检。此处不重复列出。

### 阶段 0：版本控制探测

**目标**：判断项目是否可访问版本控制历史。

**执行步骤**：
1. 按以下顺序检测版本控制系统：
   - 检测 `.git/` 目录 → 执行 `git log --oneline -1` 验证可用
   - 检测 `.svn/` 目录 → 执行 `svn log -l 1` 验证可用
   - 检测 `.hg/` 目录 → 执行 `hg log -l 1` 验证可用
2. 记录探测结果：
   ```
   版本控制：<类型>（可用 / 不可用）
   - 可用时：记录最新提交信息（摘要 + 日期）
   - 不可用时：记录"本项目无可用版本控制历史"
   ```
3. 不论结果如何，继续执行。后续阶段中，凡是提到"版本控制历史分析"的步骤，仅在探测结果为"可用"时执行。

**产出**：无文件产出。探测结果记入内存，供后续阶段使用。

**不暂停**。

---

### 阶段 1：宏观理解

**目标**：了解项目全貌。

**执行步骤**：

1. **先自动扫描**（不要先问人类）：
   - 扫描项目根目录及一级子目录，理解目录结构
   - 从构建文件（package.json / pom.xml / requirements.txt / Cargo.toml / go.mod / CMakeLists.txt / *.pro 等）识别技术栈
   - 从 README、项目名、目录命名推断项目用途
   - 从目录命名和包结构识别顶层模块划分

2. **基于扫描结果，用 AskUserQuestion 向人类确认**（一轮一问，四轮收工）：

   **第一轮**（技术栈确认——只有识别不出时才问）：
   - 如果技术栈已从构建文件明确识别 → 直接输出 `✅ 技术栈已识别：<语言> + <框架> + <数据库>`，不提问，跳到下一轮
   - 如果技术栈无法明确识别 → 用 AskUserQuestion 问一个问题："核心技术栈是什么？"；选项示例：A) Java/Spring / B) Python/FastAPI / C) Node.js/Express / D) Go / E) C++/Qt / F) 其他

   **第二轮**（项目用途确认——Agent 先推断再让人类选）：
   - 基于代码扫描结果，提炼一个一句话推断（如"我的推断：这是一个xxx分析工具"）
   - 用 AskUserQuestion 问："关于项目用途，我的推断对吗？"；选项：A) 对，就是这个 / B) 不准确，实际是：[请补充]

   **第三轮**（用户画像——Agent 无法推断，提供典型选项）：
   - 用 AskUserQuestion 问："这个项目的目标用户是？"；选项：A) 内部工程师 / B) 外部客户 / C) 两者都有 / D) 不确定

   **第四轮**（痛点+计划合并——Agent 也无法推断，提供多选降低门槛）：
   - 用 AskUserQuestion 问："当前最头疼的是什么？最近想做什么？"（multiSelect: true）；选项：A) 代码维护困难 / B) 功能扩展慢 / C) 稳定性问题 / D) 文档缺失 / E) 想加新功能 / F) 想重构 / G) 只是了解，暂无计划

   **规则**：
   - 能不问就不问——Agent 能自己确定的不要塞给人类
   - 一问一轮——用 AskUserQuestion 逐轮进行（每轮工具一次调用），不要把所有问题堆在一个文本块里
   - 每轮最多 1 个问题、不超过 6 个选项
   - 上述内容仅用于引导 Agent 生成问题，实际问题文本和选项由 Agent 自行组织

3. 将阶段 0 探测结果和本阶段信息汇总为项目概述。

**产出**：将项目概述写入 `.ai/project-log.md` 的变更时间线，作为本次逆向化首条记录。

**不暂停**。

## 大项目分阶段执行

对于大型项目（模块数 >10），可分 5 个子任务执行：

| 子任务 | 覆盖 | 产出 |
|--------|------|------|
| 子任务 1 | 阶段 0-2 | 版本探测 + 宏观理解 + 00-架构.md |
| 子任务 2 | 阶段 3 | 01-行为目录.md |
| 子任务 3 | 阶段 4-5 | 02-决策记录.md + 03-详细设计/ |
| 子任务 4 | 阶段 6 | 集中诊断 → 04-问题与改进.md |
| 子任务 5 | 阶段 7 | 补充完善 |

每个子任务独立会话完成，确认点作为子任务边界。

---

## 增量模式

当人类指定某个模块/目录（如"逆向文档化 auth 模块"）：

- 跳过阶段 0-1
- 阶段 2：只分析该模块与其余部分的依赖关系
- 阶段 3：只为该模块涉及的功能范围生成/更新 01-行为目录.md
- 阶段 5：只为指定模块生成详细设计
- 阶段 4 和 6：正常执行（全项目技术栈始终是有用的；集中诊断覆盖全项目）
- 阶段 7：正常执行
- 不重新创建 .ai/ 目录
