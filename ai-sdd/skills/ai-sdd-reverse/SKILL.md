# 遗留项目逆向化（ai-sdd-reverse）

## 定位

将没有文档的遗留代码库接入 AI-SDD 体系，从代码反推生成 `.ai/doc/` 规范文档。

**核心原则**：逆向文档 = 现状还原，不是理想设计。回答"系统当前长什么样"，不是"系统应该长什么样"。

## Ref 模式

reverse 支持通过 prompt 指定已有的文档作为参考（ref）。ref 模式有两种使用场景：

### 场景 A：外部参考文档（手动调用 reverse 时指定）
用户手动调用 reverse 时，在 prompt 中说明参考路径。
例如："对项目进行逆向文档化，参考 docs/old-design/ 下的旧设计文档"。

**ref 文档性质**：人类开发者编写的旧文档、设计 Wiki、API 文档等。
**处理方式**：作为理解辅助——帮助 Agent 更快理解系统，但不直接复制内容。
代码仍然是唯一的真相源。

### 场景 B：AI-SDD 内部文档（由 rebuild/update 自动调用）
rebuild（文档腐烂修复）或 update（版本升级）调用 reverse 时，
自动将 `.ai/doc/` 作为 ref 路径传入。

**ref 文档性质**：
- rebuild 场景：腐烂的 AI-SDD 文档——内容可能过时、在错误文件中、缺失
- update 场景：旧版本的 AI-SDD 文档——文件名/角色/结构可能已变更

**处理方式**：
- 作为"待修复/待迁移的材料"，需要 Agent 主动判断哪些保留、哪些移动、哪些丢弃
- 人类确认过的元信息（Confirmed 决策、行为状态裁决）优先保留
- 具体的处理策略由调用方（rebuild/update）通过提示词注入

### Ref 模式的通用约束
无论哪种场景：
- 代码始终是真相源——永远不因为 ref 文档说 X 就忽略代码实际做的 Y
- ref 文档中的内容不直接复制到新文档中
- 所有产出通过 reverse 的标准质量检查（正向 6 条 + 负向 5 条）
- 当被 rebuild/update 调用时，跳过阶段 0-1（项目已有 .ai/ 结构，不需要重新探测 VC 和项目用途）

## 前置检查

### 检查 1：.ai/ 目录
- 检查 `.ai/` 目录是否存在
- 不存在 → 提示"项目尚未初始化 AI-SDD，请先执行 `/ai-sdd-init`"，终止

### 检查 2：.ai/doc/ 是否已有内容
- 检查 `.ai/doc/` 下是否已有非空的 md 文件
- 有 → 输出警告：

```
⚠️ .ai/doc/ 目录已有文档。逆向化将覆盖现有内容。
   - 已有文件：[列出]

  是否继续？
    A) 继续 — 全量重新生成，覆盖现有文档
    B) 取消 — 保留现有文档不变

  如果你希望参考已有文档的内容来辅助本次 reverse，
  建议取消后重新调用时在 prompt 中说明（如"参考 xxx 文档"），
  我会将其作为参考材料。
```

- 用 AskUserQuestion 询问人类，按选择执行

## 文档质量要求

以下要求在整个逆向流程中生效。每完成一个阶段的产出，你必须执行自检。

### 1. 通用规则

遵循 `.ai/supplement-rules.md` 中的行为准则：
- §一（权限边界）— reverse 过程中哪些操作需要人类确认
- §二（强制暂停规则）— 阶段间遇到歧义/矛盾时暂停并报告
- §四（文档操作红线）— 生成的文档不得触犯红线
此外，reverse 特有规则如下：

**正向规则（reverse 特有）**：
1. 以真实代码为准——不猜测、不假设，一切以代码实际行为为依据
2. 结构化表述——用表格、列表、Mermaid 图，不用大段叙述
3. 标注符号锚点——每个关键声明附带代码符号锚点（ClassName::methodName() 或 filename.cpp → functionName()），有锚点 = 来源明确，无锚点 = 推断待确认
4. 提供增量价值——提炼代码中不直观的信息，不复述代码本身

**负向规则（reverse 特有）**：
1. 禁止使用 "..." 占位符——填充每一项，或填"待确认"并注明原因
2. 禁止粘贴大段原始代码——用结构化描述代替，附带符号锚点引用
3. 禁止以"精确性优先"为由跳过整个功能/API/章节——不确定也要列出，标注"待确认"并说明原因
4. 禁止将推断包装为事实——不确定的内容必须明确标注
5. 禁止猜测业务含义——代码行为与命名不一致时，记录"代码行为与命名不一致"，不自行解释

**"精确性优先"的正确理解**：
- ✅ 正确：不确定的字段标注"待确认" + "为什么不确认" + "建议谁可以补充"
- ❌ 错误：因为一个字段不确定就跳过整个 API
- ✅ 正确：该列出的接口、表、字段穷尽列出——只有其中不确定的细节才标注
- ✅ 正确：简单模块自然产出短文档（因为确实没什么可写）
- ❌ 错误：复杂模块产出几十行的短文档（以"少而准"为借口）

### 2. 文件角色对照表

每份文档的完整角色定义（定位、应该写什么、不应该写什么、读者、写入者）
见各模板顶部的「📋 本文档角色」区块。此处仅列出文件与角色的简要映射：

| 文件 | 角色（摘要） | 完整定义见 |
|------|------------|-----------|
| `00-架构.md` | 系统静态结构快照：模块划分、依赖关系、数据流、外部依赖 | 模板顶部角色区块 |
| `01-行为目录.md` | 系统可观察行为清单（B-XXX） | 模板顶部角色区块 |
| `02-决策记录.md` | 技术栈事实 + 所有层级决策理由集（D-XXX） | 模板顶部角色区块 |
| `03-详细设计/xxx.md` | 单模块详细设计：数据模型、API 契约、业务流程等 | 模板顶部角色区块 |
| `04-问题与改进.md` | 当前有效已知问题（AV/DR/FD/VR/MT/DD/CD 七范畴） | 模板顶部角色区块 |

Agent 在产出每份文档前，必须阅读对应模板顶部的完整角色区块。

### 3. 产出自检清单

各文档的具体产出要求已分散到对应阶段的 sop/ 子文件中，Agent 在产出文档后立即执行自检。此处不重复列出。

### 阶段 0：版本控制探测

**目标**：判断项目是否可访问版本控制历史。

**执行步骤**：
1. 按以下顺序检测版本控制系统：
   - 检测 `.git/` 目录 → 执行 `git log --oneline -1` 验证可用
   - 检测 `.svn/` 目录 → 执行 `svn log -l 1` 验证可用
   - 检测 `.hg/` 目录 → 执行 `hg log -l 1` 验证可用
2. 记录探测结果：
   ```
   版本控制：<类型>（可用 / 不可用）
   - 可用时：记录最新提交信息（摘要 + 日期）
   - 不可用时：记录"本项目无可用版本控制历史"
   ```
3. 不论结果如何，继续执行。后续阶段中，凡是提到"版本控制历史分析"的步骤，仅在探测结果为"可用"时执行。

**产出**：无文件产出。探测结果记入历史记忆，供后续阶段使用。

**不暂停**。

---

### 阶段 1：宏观理解

**目标**：了解项目全貌。

**执行步骤**：

1. **自动扫描**（不要先问人类）：
   - 扫描项目根目录及一级子目录，理解目录结构
   - 从构建文件（package.json / pom.xml / Cargo.toml / go.mod / CMakeLists.txt / *.pro / *.sln 等）识别技术栈
   - 从 README、项目名、目录命名推断项目用途
   - 从目录命名和包结构识别顶层模块划分
   - 技术栈直接从构建文件输出结论，不问人类

2. **一句话确认**（仅一问）：
   - 基于扫描结果，提炼一句话项目用途推断（如"这是一个电力系统谐波扫描的桌面工具"）
   - 用 AskUserQuestion 确认："我的推断对吗？"；选项：A) 对 / B) 不准确，实际是：[请补充]

3. 将阶段 0 探测结果和本阶段信息汇总为项目概述。

**产出**：将项目概述写入 `.ai/project-log.md` 的变更时间线，作为本次逆向化首条记录。

**不暂停**。

**目标**：完成收尾工作。

```
  - 部分内容标注为"待确认"，建议在方便时人工确认
```

**不暂停**。
