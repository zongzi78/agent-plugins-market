# 遗留项目逆向化

## 定位

将没有文档的遗留代码库接入 AI-SDD 体系，从代码反向生成 `.ai/doc/` 规范文档。

## 前置检查

1. 检查 `.ai/` 是否已存在
   - 如不存在：先执行 `/ai-sdd-init` 流程（创建 .ai/ 目录和基础文件）
2. 检查 `.ai/doc/` 是否已有内容
   - 如已有内容：警告".ai/doc/ 目录已有文档，逆向化可能覆盖"，AskUserQuestion 确认
3. **适用范围判断**：向人类提问——"未来 3 个月内预计有几次实质性改动？"
   - 如果 <3 次：建议不做完整逆向化，用探索会话做一次性分析即可
   - 如果 ≥3 次：继续执行以下流程
   - 如果人类坚持要做（不管几次）：尊重决定，继续

## 文档质量要求

在开始执行流程之前，你必须理解以下质量要求并严格遵守。

### 1. 文档内容完备性清单

每份文档产出时，必须覆盖以下必备要素：

| 文档 | 必须覆盖 |
|------|----------|
| 00-架构.md | 所有第一级模块的描述（含复杂度概要）、完整 Mermaid 依赖图（所有模块节点不可省略）、≥2 个完整端到端数据流（含模块途经+数据变换+持久化点）、全部外部依赖及故障影响、技术债务项（如实记录，数量由代码质量决定）、认知边界地图 |
| 01-需求.md | 每个可识别功能组 1 个 F-XXX 条目、每个功能 ≥3 条验收标准（含至少 1 条错误路径）、lifecycle 标记（active/suspected-dead/deprecated）、隐式约束章节覆盖并发/错误处理/性能/边界 4 个维度、需求漂移检查 |
| 02-技术选型.md | 全部直接依赖及版本、每个关键依赖的使用模式评估（标准用法 vs 非标准用法）、构建/测试/部署的实际配置路径验证、版本漂移和风险分析、缺失决策理由列表 |
| 03-详细设计/ | 概述含负向限制、核心数据模型的关键字段（穷尽列出，不省略）、全部公开 API 的契约（含副作用和幂等性设计）、≥1 个 Mermaid 业务流程图（使用实际函数/模块名）、异常处理/并发安全/性能特征三专题均填写（如不适用，写明原因而非跳过） |

### 2. "精确性 > 完整性"的正确执行方式

这条规则的真正含义是"不确定的细节要诚实标注"，而非"不确定就什么都不写"：

```
✅ 正确：不确定的字段标注"待确认" + "为什么不确定" + "建议谁可以补充"
❌ 错误：因为一个字段不确定就跳过整个 API
✅ 正确：该列出的接口、表、字段必须穷尽列出——只有其中不确定的细节才标注
❌ 错误：用 "..." 占位符代替实际内容
✅ 正确：简单模块自然产出短文档（因为确实没什么可写）
❌ 错误：复杂模块也产出几十行的短文档（以"少而准"为借口）
```

### 3. 负向限制（禁止事项）

以下行为在任何逆向阶段均严格禁止：

1. ❌ **禁止使用 "..." 占位符**——填充每个必要字段，或标注"待确认"并注明原因
2. ❌ **禁止粘贴大段原始代码**——用结构化描述提炼。文档是代码的"语义层"，提供代码本身之外的价值
3. ❌ **禁止以"精确性优先"跳过整个功能/API/章节**——即使部分内容不确定，也要记录已知部分
4. ❌ **禁止将推断包装为事实**——SPECULATIVE 内容必须有明确标注和理由
5. ❌ **禁止猜测业务含义**——当代码行为与命名/配置暗示不一致时，标记为"代码行为与命名不一致"而非自行解释

## 执行流程

### 阶段 0：版本控制探测

**目标**：判断项目是否可访问版本控制历史，供后续阶段使用。

**执行**：
1. 检查 `.git/` 目录是否存在，`git log` 是否可执行
2. 如果可用 → 记录 commit 总数和最近 commit 日期，供后续阶段辅助分析
3. 如果不可用 → 记录"本项目无可用版本控制历史"，后续阶段跳过 Git 相关分析
4. 不阻塞流程——无论结果如何，继续执行

---

### 阶段 1：宏观理解

**目标**：了解项目全貌。

**执行**：

先向人类收集 5 个核心信息：
1. 这个项目是干什么的？（一句话）
2. 给谁用的？（内部/外部，用户量级）
3. 核心技术栈（如知道）：语言/框架/数据库
4. 当前主要痛点是什么？
5. 近期目标是什么？

收集完毕后，执行：
- 扫描项目根目录及一级子目录，识别目录结构
- 从 package.json / pom.xml / requirements.txt / Cargo.toml / go.mod 等识别技术栈
- 从目录命名和包结构识别顶层模块划分

**产出**：项目概述，写入 `.ai/project-log.md` 的变更时间线首条记录。

本阶段不需要人类确认。

---

### 阶段 2：架构梳理

**目标**：还原系统架构。

**执行**：
1. 分析模块间依赖关系（import/require 语句、包结构、模块调用链）
2. 识别数据流向
3. 识别外部依赖和集成点
4. 用 Mermaid 绘制模块依赖图（所有模块节点不可省略，使用实际模块名）
5. 标注技术债务（搜索 TODO/FIXME/HACK、循环依赖、废弃 API、硬编码配置等）
6. 搜索隐式架构约束：循环依赖（A→B→A）、跨层调用（上层直接调下层之下）、防腐层穿透等。发现后记入"架构约束与已知例外"章节
7. 填充认知边界地图：区分代码验证的事实（已知已知）、分析后仍不理解的点（已知未知）、从模式推断但未确认的设计意图（推断结论）

**产出**：`.ai/doc/00-架构.md`（使用模板）

**产出后暂停**，等待人类确认。输出确认清单：
```
□ 模块列表是否完整？有无遗漏？
□ 依赖关系图是否准确反映了实际的 import/include 关系？
□ 外部依赖是否完整？（数据库、缓存、消息队列、外部服务等）
□ 认知边界地图中的"已知未知"是否需要补充或修正？
□ 模块的复杂度概略评估是否大致合理？
```

**确认后操作**：Agent 必须将已确认内容的 confidence 从 INFERRED/SPECULATIVE 更新为 CONFIRMED，更新文档 front matter 的 status 和 last_updated。

---

### 阶段 3：需求还原

**目标**：从代码反向推导功能需求。

**执行**：
1. 从 API 路由推导对外功能
2. 从数据库 Schema 推导数据域
3. 从配置文件推导外部依赖
4. **Git 历史辅助分析**（仅阶段 0 探测确认版本控制可用时执行）：
   运行 `git log --oneline -n 100`，通过 commit message 关键词辅助判断：
   - 哪些功能是核心功能（高频 feat commit）
   - 哪些功能可能已废弃（长期无变更 + 无外部入口）
   - 是否存在 commit message 描述与代码实际行为不一致的迹象
   如果版本控制不可用，跳过本步骤
5. **功能生命周期标记**：每个 F-XXX 标注 lifecycle（active / suspected-dead / deprecated）
6. **置信度标注**：每个功能的每个关键声明标注 CONFIRMED / INFERRED / SPECULATIVE
7. **隐式约束搜索**：主动扫描代码中的——
   - 并发保护（锁、原子操作、事务隔离级别）
   - 幂等性设计（唯一索引、分布式锁、去重逻辑）
   - 重试与超时（重试策略、超时配置、熔断降级）
   - 限流与配额
   - 事务边界（事务范围、回滚条件）
   - 错误码体系
   - 将发现写入"隐式约束与已知陷阱"章节
8. 标注可能废弃的功能
9. 为每个功能使用 RFC 2119 关键词描述行为（标注置信度来源）
10. 为每个功能附带验收标准（Given/When/Then 格式，逆向时从代码行为还原，标注来源）

**产出**：`.ai/doc/01-需求.md`（使用模板）

**产出后暂停**，等待人类确认。输出确认清单：
```
□ 每个功能请标记：[confirmed] / [discard] / [needs-clarification]
□ 哪些功能判定为废弃？判定是否准确？
□ 关键业务规则的描述是否符合实际？
□ 隐式约束（并发/幂等/错误处理/性能/边界）是否有遗漏？
□ 标注 SPECULATIVE 的内容是否需要补充或修正？
```

**确认后操作**：Agent 必须——
1. 将 CONFIRMED 的功能的 confidence 更新为 CONFIRMED
2. 被标记为 [discard] 的功能在文档中标注为 deprecated
3. 被标记为 [needs-clarification] 的内容保留原 confidence 标记，等待后续补充
4. 更新文档 front matter 的 status 和 last_updated

---

### 阶段 4：技术栈确认

**目标**：记录当前技术选型（现状陈述，不猜测选型理由）。

**执行**：
1. 分析所有关键依赖的名称和版本
2. 识别技术债务具体清单
3. 记录构建/测试/部署现状（验证实际配置文件，不假设标准工具）
4. **版本漂移评估**：每个关键依赖，对比当前版本与包管理器可获取的最新稳定版
5. **使用模式分析**：标注每个技术组件的使用是标准用法还是有非标准/定制用法
6. **提取"缺失的决策理由"列表**：列出从代码中无法推断但值得记录的技术选择决策（如"为什么选 PostgreSQL 而非 MySQL？代码中无法推断"）
7. 搜索代码中隐含的技术约束和反模式（如"某处使用了同步 HTTP 调用但未设置超时"）

**产出**：`.ai/doc/02-技术选型.md`（使用模板）

本阶段不需要人类确认。

---

### 阶段 5：详细设计还原

**目标**：还原各模块的详细设计。

**核心原则：深度自适应——不预设模块层级，每个模块都生成详细设计文件，但深度由模块自身复杂度自然决定。复杂模块自然产出长文档，简单模块自然产出短文档。**

**执行**：
1. 对每个模块：
   - **概述**：一句话职责精确定义 + 在系统中的定位 + 负向限制（本模块绝对不做什么，2-5 条）
   - **数据模型**：列出所有持久化字段（穷尽，不省略），标注置信度
   - **API 契约**：每个公开 API 的完整契约——方法、路径、参数、响应、错误码、副作用、幂等性设计
   - **核心业务流程**：Mermaid 序列图或流程图，使用实际函数/模块名
   - **状态机**（如适用）
   - **异常处理与容错策略**（必填）：已知异常域、捕获策略、降级兜底逻辑。如本模块无异常处理代码，写明"未发现异常处理代码，存在安全隐患"
   - **并发安全与一致性保证**（必填）：并发风险点、锁与事务机制、隔离级别。如本模块无共享状态，写明"本模块为无状态函数/工具模块，不涉及共享状态"
   - **性能特征与资源消耗**（必填）：核心路径复杂度、缓存策略、查询模式、资源密集点。如模块非常简单（纯计算、无 I/O），写明原因
   - **已知问题与陷阱**：文档中记录在本模块代码中发现的具体问题
   - **模块接口图**：对外提供和依赖消费的接口一览
2. 按业务域组织（每个域一个文件）
3. 用结构化描述，不用大段代码

**产出**：`.ai/doc/03-详细设计/<模块名>.md`（每个模块一个文件，基于 `_模板.md`）

**产出后暂停**，等待人类确认。输出确认清单：
```
□ 各模块的设计描述是否准确反映了代码实际？
□ 异常处理/并发安全/性能特征三个专题的分析是否有错误或遗漏？
□ 标注 SPECULATIVE 的内容是否需要人工澄清或补充？
□ 隐式约束是否完善？是否需要补充未发现的设计隐患？
```

**确认后操作**：Agent 必须将已确认内容的 confidence 更新为 CONFIRMED，更新文档 front matter。

---

### 阶段 6：补充完善

**目标**：完成所有收尾工作。

**执行**：
1. 在 `.ai/project-log.md` 中记录已知问题和技术债务
2. 补充"待同步文档"列表
3. 确保所有 doc 文件有正确的 YAML front matter
4. 如有可从代码推断的决策，记录到 `.ai/changes/`
5. 检查所有产出的文档：是否有 "..." 占位符未处理？隐式约束是否至少覆盖 4 个维度？每个关键声明是否标注了置信度？

本阶段不需要人类确认。

## 大型项目分阶段执行

| 子任务 | 覆盖阶段 | 产出 |
|--------|---------|------|
| 子任务 1 | 阶段 0-2 | project-log.md 首条记录 + 00-架构.md |
| 子任务 2 | 阶段 3 | 01-需求.md |
| 子任务 3 | 阶段 4-5 | 02-技术选型.md + 各模块详细设计 |
| 子任务 4 | 阶段 6 | 补充完善 |

## 增量模式

当用户指定某个模块/目录时（如"逆向文档化 auth 模块"）：
- 跳过阶段 0-1（已有宏观理解）
- 只对指定范围执行阶段 2-5
- 阶段 2：只分析该模块与其余部分的依赖关系
- 阶段 5：只为指定模块产出详细设计
- 不重复创建 .ai/ 目录和 project-log.md

## 质量规则

### 正向规则（必须做到）

1. **以真实代码为准**——不猜测，不假设，一切以代码实际行为为准
2. **创建心态**——当作首次编写，通读代码后从头组织内容，不在末尾追加
3. **尊重每个文件的角色**——01-需求.md 写"做什么"，02-技术选型.md 写"用什么"，03-详细设计/ 写"怎么做"
4. **结构化表述**——用表格、列表、Mermaid，不用大段叙述
5. **标注置信度**——每个关键声明标注 CONFIRMED / INFERRED / SPECULATIVE，不得将推断包装为事实
6. **提供增量价值**——文档提炼代码中不直观的信息（设计意图、隐式约束、已知陷阱），不复述代码本身

### 负向规则（严禁行为）

1. ❌ **禁止**使用 "..." 占位符——填充每一项或标注"待确认"并注明原因
2. ❌ **禁止**粘贴大段原始代码——用结构化描述代替
3. ❌ **禁止**以"精确性优先"为由跳过整个功能/API/章节
4. ❌ **禁止**将推断包装为事实——SPECULATIVE 内容必须有标注
5. ❌ **禁止**猜测业务含义——代码行为与命名意图不一致时，标记为"代码行为与命名不一致"而非自行解释

### 深度自检清单（每阶段产出后执行）

- [ ] 文档内容完备性清单是否全部覆盖？
- [ ] 是否有 "..." 占位符未处理？
- [ ] 隐式约束是否至少覆盖了 4 个维度（并发/错误处理/性能/边界）？
- [ ] 每个关键声明是否标注了置信度（CONFIRMED/INFERRED/SPECULATIVE）？
- [ ] 人类确认清单是否已输出？
