# 阶段 4：技术栈确认

**目标**：记录当前技术选型。这是"现状盘点"，不猜测选型动机。

> ⚠️ 执行前提：本阶段涉及 confidence 标记。开始前必须先 Read `skills/ai-sdd-reverse/术语与概念体系.md`，确认理解：
> - confidence 三级定义：`confirmed`（已确认）= 人类确认过 / `inferred`（推断）= 有代码证据但未确认 / `speculative`（推测）= 证据不足
> - front matter 的 `confidence` 字段取整体最高规则：全部 confirmed → `confirmed`；存在 speculative → `speculative`；否则 `inferred`
> - `status` 字段：人类确认后 → `reviewed`；全部 confirmed → `baseline`

## 执行步骤

1. **分析依赖**：从包管理器文件中提取所有直接依赖项及其版本

2. **版本漂移评估**：每个关键依赖对比当前版本 vs 最新稳定版。
   - 版本对比事实写入 `02-技术选型.md` > 版本漂移
   - 风险判断写入 `04-问题与改进.md` > 版本与依赖风险

3. **使用模式分析**：每个技术组件标注是标准用法还是定制/非标准用法

4. **构建/测试/部署验证**：从实际配置文件中提取工具链信息

5. **提取"缺失决策理由"列表**：列出从代码中无法推断但值得记录的技术选择

6. **技术约束提取**：记录代码中隐含的技术禁止事项

7. **反模式识别**：按下方「写入 04-问题与改进.md」中的搜索清单执行，结果写入 `04-问题与改进.md` > 反模式

### 写入观察便签

在 `.ai/project-log.md` 底部的「诊断观察便签」区段中追加「阶段4 观察」子节，仅记录 文件:行号 + 一句话描述：

示例：
```markdown
### 阶段4 观察
- [mainWindow_slot2.cpp:25-30] 7 个 g_ 前缀全局变量跨 4 个 .cpp 共享
- [scanmodel.h:73-76] m_devDetailMap + m_pCompAry 双重数据结构
- [sim_action_adaptor.cpp:16] 混用旧式 SIGNAL()/SLOT() 宏（项目其余使用新式语法）
```

**便签规则**：只记录位置 + 事实，不评级、不分类。不确定的线索也写入，标注"? 需阶段6确认"。

### 模式诊断清单

本阶段在分析代码时，逐项审视以下维度（发现后写入观察便签，不自行写入 04）：

□ 全局可变状态检查：扫描 g_ 前缀变量、extern 声明、static 非 const 变量
  - 跨编译单元共享？是否有同步保护？→ 记录便签

□ 编码约定一致性检查：扫描是否存在混用的语法/命名风格
  - 新旧 Qt connect 语法混用？中英文命名混用？→ 记录便签

□ RAII 合规检查：扫描裸 new/delete
  - 无智能指针包装？手动 lock/unlock 无 RAII 封装？→ 记录便签

□ 职责过载检查：
  - 单类方法数 >20 且涉及 ≥3 种数据域？→ 记录便签
  - 单方法 >80 行且含 ≥3 种操作？→ 记录便签

## 关于可信度

本阶段产出的信息分为两类：
- **直接可验证的事实**（如依赖名称、版本号、配置文件路径）→ 直接标记为已确认
- **推断性内容**（如使用模式评估、版本迁移建议）→ 标记为推断

## 确认清单

产出后暂停，输出以下确认清单：

```
📋 阶段 4 确认清单 — 技术栈确认

□ 技术栈总览：所有主要技术是否都识别到了？有无遗漏或误判？
□ 关键依赖：版本号是否正确？使用模式（标准/非标准）判断是否合理？
□ 版本漂移：需要特别关注的版本差距是否有遗漏？
□ 构建/测试/部署：工具链识别是否准确？
□ 观察便签是否有遗漏的线索需要补充？

如有纠正请直接说明（如"Redis 版本是 6.2 不是 7.0"、"漏了 Nginx"、"XX 不是非标准用法"）。
如需跳过此确认，回复"跳过"即可继续。
```

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新文档内容**：遍历 02-技术选型.md 中所有标记为推断的声明——人类确认的 → 改为已确认
2. **重新统计置信度**：
   a. 重新遍历所有可信度标记（因第1步可能改变了标记）
   b. 输出统计：已确认 N 处、推断 M 处、推测 K 处
   c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
   d. 更新 front matter：`confidence`、`last_updated`（今天日期）、`status`（`reviewed` 如果人类确认）
3. **在 project-log.md 追加记录**：`[日期] 阶段4 经人类确认，N 处推断升级为已确认`

只有完成以上步骤后，才进入阶段 5。

## 人类回复后分支处理

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 / 回复"跳过" | 执行上方「强制更新步骤」后继续 |
| 某信息需要纠正 | 按人类指示直接修改文档；修改后的内容标记为已确认；然后执行「强制更新步骤」 |
| 补充了新信息 | 追加到对应章节，标记为已确认；然后执行「强制更新步骤」 |
| 人类未回复 | 所有内容保持原可信度；继续下一阶段 |
