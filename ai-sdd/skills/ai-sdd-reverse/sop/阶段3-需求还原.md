# 阶段 3：需求还原

**目标**：从代码反推功能列表。

> ⚠️ 执行前提：本阶段涉及 confidence 标记。开始前必须先 Read `skills/ai-sdd-reverse/术语与概念体系.md`，确认理解：
> - confidence 三级定义：`confirmed`（已确认）= 人类确认过 / `inferred`（推断）= 有代码证据但未确认 / `speculative`（推测）= 证据不足
> - front matter 的 `confidence` 字段取整体最高规则：全部 confirmed → `confirmed`；存在 speculative → `speculative`；否则 `inferred`
> - `status` 字段：人类确认后 → `reviewed`；全部 confirmed → `baseline`
> - 生命周期三级定义：`active`（活跃）/ `suspected-dead`（疑似废弃）/ `deprecated`（已废弃）

## 执行步骤

1. **从 API 路由提取功能**：扫描路由注册代码，列出所有对外接口 → 按业务场景归类为功能组

2. **从数据库 Schema 提取数据域**：列出数据实体，关联到对应功能

3. **从配置文件提取外部依赖**：识别集成的外部服务

4. **版本控制历史辅助分析**（仅阶段 0 确认可用时执行）：
   - 运行 `git log --oneline -n 100`（或 `svn log -l 100` / `hg log -l 100`）
   - 通过提交信息辅助判断：哪些功能是核心功能（高频 feat）、哪些可能已废弃（长期无变更 + 无外部入口）

5. **为每个功能确定生命周期**：
   - 活跃：有完整调用链路（路由 → Handler → Service → DAO）
   - 疑似废弃：路由注册但 handler 是空壳 / 长期无变更 / 配置中关闭
   - 命名但不猜测——不确定时标注"疑似废弃"而非"已废弃"

6. **为每个功能的每个关键声明标注可信度**：
   - 已确认：行为在代码中明确可见，有具体 `文件:行号` 引用
   - 推断：从框架默认值、命名约定推断，但未直接看到配置或显式声明
   - 推测：基于有限证据的猜测，需要人类验证

7. **搜索隐式约束**（以下 4 个维度）：
   - 并发：锁、原子操作、事务隔离级别、幂等性设计
   - 错误处理：错误码体系、重试策略、降级逻辑、超时配置
   - 性能：缓存策略、连接池、N+1 查询风险、资源密集点
   - 边界：输入校验规则、硬编码限制、配置驱动的行为分支
   - 每个发现附带 `文件:行号` 引用

8. **为每个功能写验收标准**：
   - 至少 1 条正常路径 + 至少 1 条错误/边界路径
   - 格式：`给定 [前置条件]，当 [操作]，则 [预期结果]`
   - 逆向模式：从代码中的参数校验、错误处理、状态码推断验收标准

9. **为每个功能写行为约束**：从代码中的校验逻辑、事务边界、中间件守卫提取行为约束（如"订单金额必须 ≥ 0"），标注可信度和代码位置

10. **需求漂移检查**：比较"命名/配置暗示的预期行为"与"代码实际行为"，标记不一致

## 产出

`.ai/doc/01-需求.md`（基于模板生成）

## 产出后自检

生成完成后立即逐项核对：
- [ ] 每个可识别功能组对应一个 F-XXX 条目
- [ ] 每个功能含验收标准（正常路径 ≥1 条，错误/边界路径 ≥1 条）
- [ ] 每个功能标注了生命周期（活跃 / 疑似废弃 / 已废弃）
- [ ] 每个功能标注了可信度（已确认 / 推断 / 推测）和代码证据
- [ ] 隐式约束章节覆盖了并发/错误处理/性能/边界 4 个子表
- [ ] 需求漂移检查表已填写
- [ ] **置信度统计**（必须在输出确认清单前完成）：
  a. 逐条列出所有功能条目的可信度标记（F-XXX + 当前等级）
  b. 汇总：已确认 N 处、推断 M 处、推测 K 处
  c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
  d. 将结果写入 front matter 的 `confidence` 字段

产出后暂停，输出以下确认清单：

```
请对每个功能标记你的决定。可直接在功能编号后标注，如"F-001 确认"、"F-003 废弃"、"F-005 需澄清——这个功能我不确定还在不在用"

功能列表：
[列出所有 F-XXX 编号 + 功能名称 + 当前生命周期标记]

  [确认] — 这个功能确实存在且在（或应该）使用
  [废弃] — 这个功能已经不用了
  [需澄清] — 我不确定，需要进一步调查（请说明不确定什么）

□ 隐式约束（并发/幂等/错误处理/性能/边界）是否有遗漏？
□ 标注推测的内容是否需要补充或修正？
```

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新文档内容**：遍历所有功能条目——人类标记 `[确认]` 的 → confidence 改为 `已确认`、lifecycle 改为 `活跃`；标记 `[废弃]` 的 → lifecycle 改为 `已废弃`；标记 `[需澄清]` 的 → 追加 `<!-- 待确认 -->` 注释
2. **重新统计置信度**：
   a. 重新遍历所有功能条目的可信度标记（因第1步可能改变了标记）
   b. 输出统计：已确认 N 处、推断 M 处、推测 K 处
   c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
   d. 更新 front matter：`confidence`、`last_updated`（今天日期）、`status`（`reviewed` 如果人类确认了大部分功能）
3. **在 project-log.md 追加记录**：`[日期] 阶段3 经人类确认，N 个功能标记确认、M 个标记废弃、K 个需澄清`

只有完成以上步骤后，才进入阶段 4。

## 人类回复后分支处理

| 人类标记 | 你的操作 |
|---------|---------|
| `[确认]` | 执行上方「强制更新步骤」后继续 |
| `[废弃]` | 该功能生命周期 → `deprecated`；执行上方「强制更新步骤」后继续 |
| `[需澄清]` | 功能尾部追加 `<!-- 待确认：人类标记需澄清 — YYYY-MM-DD。问题：[人类描述] -->`；执行上方「强制更新步骤」后继续 |
| 人类补充了业务信息 | 追加到对应功能描述，标记为 `已确认`；然后执行「强制更新步骤」 |
| 人类要求修改 | 按指示改文档，改后标记为 `已确认`；然后执行「强制更新步骤」 |
| 人类指出了遗漏功能 | 分析遗漏的代码区域，补充该功能条目，标记为 `推断`，重新输出确认清单（跳过强制更新步骤——等下一轮确认） |

## 处理完成后

- 更新 front matter：`last_updated`、`confidence`（按整体规则计算）、`status: reviewed`（如果人类确认了大部分）
- 如果有 `[需澄清]` 的功能，在 project-log.md 的"待确认"列表中追加记录
