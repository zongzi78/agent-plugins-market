# 阶段 3：需求还原

**目标**：从代码反推功能列表。

> ⚠️ 执行前提：本阶段涉及 confidence 标记。开始前必须先 Read `skills/ai-sdd-reverse/术语与概念体系.md`，确认理解：
> - confidence 三级定义：`confirmed`（已确认）= 人类确认过 / `inferred`（推断）= 有代码证据但未确认 / `speculative`（推测）= 证据不足
> - front matter 的 `confidence` 字段取整体最高规则：全部 confirmed → `confirmed`；存在 speculative → `speculative`；否则 `inferred`
> - `status` 字段：人类确认后 → `reviewed`；全部 confirmed → `baseline`
> - 生命周期三级定义：`active`（活跃）/ `suspected-dead`（疑似废弃）/ `deprecated`（已废弃）

## 执行步骤

1. **从 API 路由提取功能**：扫描路由注册代码，列出所有对外接口 → 按业务场景归类为功能组

2. **从数据库 Schema 提取数据域**：列出数据实体，关联到对应功能（数据域详细描述见 03-详细设计，本阶段仅做关联）

3. **版本控制历史辅助分析**（仅阶段 0 确认可用时执行）：
   - 运行 `git log --oneline -n 100`（或 `svn log -l 100` / `hg log -l 100`）
   - 通过提交信息辅助判断：哪些功能是核心功能（高频 feat）、哪些可能已废弃（长期无变更 + 无外部入口）

4. **为每个功能确定生命周期**：
   - 活跃：有完整调用链路（路由 → Handler → Service → DAO）
   - 疑似废弃：路由注册但 handler 是空壳 / 长期无变更 / 配置中关闭
   - 不确定时标注"疑似废弃"而非"已废弃"

5. **为每个功能标注可信度**：
   - 功能列表级别：标注 ⚠️（推断）或 ❓（推测），已确认不打标记
   - 已确认：行为在代码中明确可见，有具体 `文件:行号` 引用
   - 推断：从框架默认值、命名约定推断
   - 推测：基于有限证据的猜测

6. **为每个功能写验收标准**：
   - 至少 1 条正常路径 + 至少 1 条错误/边界路径
   - 格式：`给定 [前置条件]，当 [操作]，则 [预期结果]`
   - 逆向模式：从代码中的参数校验、错误处理、状态码推断验收标准

7. **提取隐式约束**：
   - 按 `03-详细设计_模板.md` 中并发安全/异常处理/性能特征的引导执行
   - 结果写入各模块的 03-详细设计文件，不写入 01-需求.md

8. **需求漂移检查和功能缺陷识别**：按下方「写入观察便签」中的清单执行，线索写入观察便签

### 写入观察便签

在 `.ai/project-log.md` 底部的「诊断观察便签」区段中追加「阶段3 观察」子节，仅记录 文件:行号 + 一句话描述：

示例：
```markdown
### 阶段3 观察
- [mainWindow_scan.cpp:209] onPauseCalc() 空函数体，函数名暗示"暂停"功能
- [simtaskmanager.cpp:142] scheduleTasks() 弹出任务但不分派
- [mainWindow.cpp:62] openRecentProjects(50) 调用传参 50，但函数体忽略该参数
```

**便签规则**：只记录位置 + 事实，不评级、不分类。不确定的线索也写入，标注"? 需阶段6确认"。

### "待确认"标记的使用限制

在 01-需求.md 的功能详情中标注"待确认"时，必须遵守：

**允许条件**（必须同时满足）：
1. 已阅读了所有相关的代码路径（调用方、被调用方、错误处理分支）
2. 代码中确实不存在对该场景的处理逻辑
3. 无法从代码推断该场景是否会实际发生

**禁止**：
- 只因为没读完相关代码就标注 → 必须先完成代码阅读
- 同一功能中有 ≥3 个"待确认" → 降级该功能的整体 confidence 为 speculative
- 用"⚠️ 待确认"替代具体的分析

当标注"待确认"时，必须附带：
- 已检查的代码路径（文件:行号列表）
- 为什么代码无法回答这个问题
- 要确认需要什么信息或验证方式

### 声明-实现一致性检查

对每个 public/protected 方法：
□ 函数名含动作词（create/send/update/delete/start/stop/pause/open/close/init）→ 检查函数体是否有对应副作用
□ 空函数体 + 动作词函数名 → 记录为观察便签
□ 函数体仅含 return/日志而无实际操作 → 记录为观察便签

对每个调用点：
□ 传入有意义的值但函数签名中参数名被注释（如 foo(int /*unused*/)）→ 记录为观察便签

对控制流：
□ 空 if/else/while/for 块 → 检查是否曾有逻辑被移除 → 记录为观察便签

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新文档内容**：遍历所有功能条目——人类标记 `[确认]` 的 → confidence 改为已确认、lifecycle 改为活跃；标记 `[废弃]` 的 → lifecycle 改为已废弃；标记 `[需澄清]` 的 → 追加 `<!-- 待确认 -->` 注释
2. **重新统计置信度**：
   a. 重新遍历所有功能条目的可信度标记（因第1步可能改变了标记）
   b. 输出统计：已确认 N 处、推断 M 处、推测 K 处
   c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
   d. 更新 front matter：`confidence`、`last_updated`（今天日期）、`status`（`reviewed` 如果人类确认了大部分功能）
3. **在 project-log.md 追加记录**：`[日期] 阶段3 经人类确认，N 个功能标记确认、M 个标记废弃、K 个需澄清`

只有完成以上步骤后，才进入阶段 4。

## 人类回复后分支处理

| 人类标记 | 你的操作 |
|---------|---------|
| `[确认]` | 执行上方「强制更新步骤」后继续 |
| `[废弃]` | 该功能生命周期 → `deprecated`；执行上方「强制更新步骤」后继续 |
| `[需澄清]` | 功能尾部追加 `<!-- 待确认：人类标记需澄清 — YYYY-MM-DD。问题：[人类描述] -->`；执行上方「强制更新步骤」后继续 |
| 人类补充了业务信息 | 追加到对应功能描述，标记为已确认；然后执行「强制更新步骤」 |
| 人类要求修改 | 按指示改文档，改后标记为已确认；然后执行「强制更新步骤」 |
| 人类指出了遗漏功能 | 分析遗漏的代码区域，补充该功能条目，标记为推断，重新输出确认清单（跳过强制更新步骤——等下一轮确认） |

## 处理完成后

- 更新 front matter：`last_updated`、`confidence`（按整体规则计算）、`status: reviewed`（如果人类确认了大部分）
- 如果有 `[需澄清]` 的功能，在 project-log.md 的"待确认"列表中追加记录
