# 阶段 5：详细设计还原

**目标**：还原各模块的详细设计。

**核心原则**：如实描述——模块实际涉及什么就写什么，不涉及就标准声明"不涉及"。章节一个不落，深度自然有深有浅。

> ⚠️ 执行前提：本阶段涉及 confidence 标记。开始前必须先 Read `skills/ai-sdd-reverse/术语与概念体系.md`，确认理解：
> - confidence 三级定义：`confirmed`（已确认）= 人类确认过 / `inferred`（推断）= 有代码证据但未确认 / `speculative`（推测）= 证据不足
> - front matter 的 `confidence` 字段取整体最高规则：全部 confirmed → `confirmed`；存在 speculative → `speculative`；否则 `inferred`
> - `status` 字段：人类确认后 → `reviewed`；全部 confirmed → `baseline`

## 执行步骤

### 章节规则

所有 8 个章节必须全部出现。每章如实描述模块的实际情况：
- 有相关内容 → 按模板 GEN 引导详细编写
- 没有相关内容 → 写标准措辞（见 `03-详细设计_模板.md` 的 `<!-- GEN: 章节规则 -->`），一字不改

禁止以任何理由跳过章节。

### 对每个模块执行

每个文件包含以下章节：

### 章节编写顺序（强制）

不要按模板的线性顺序写。按分析难度倒序，确保最高注意力用于最难分析的内容：
1. **先写**异常处理 + 并发安全（最需要深度分析，消耗最大）
2. **再写**核心业务流程 + 状态机
3. **再写**对外接口契约 + 核心数据模型（最容易从代码提取）
4. **最后写**概述（此时已对整个模块有深入理解，概述质量最高）

### 跨线程交互专项检查

在编写并发安全章节时，必须检查以下跨线程交互：

□ 全局容器（g_ 前缀）的所有读写点 → 标记所在线程
□ QCoreApplication::postEvent 传递的数据 → 检查发送线程和接收线程
□ 信号槽跨线程连接（Qt::QueuedConnection）→ 检查传递的数据是否线程安全
□ QMutexTryLocker 超时路径 → 检查是否有回退/缓存/重试机制
□ weak_ptr::lock() 调用点 → 检查返回值是否判空

每个检查点如有风险 → 记录为观察便签

### 多组件模块组织规则（强制）

如果一个详细设计文档覆盖 N ≥ 2 个独立类/组件：
- 核心数据模型 → 每个组件必须有独立的字段表 + 约束
- 对外接口契约 → 每个组件必须有独立的公开方法表
- 异常处理与容错 → 每个组件必须有独立的异常场景表

**禁止**：在概述中列出某个组件但在后续章节中不再出现。

Agent 自检：概述中列出的每个组件，在数据模型、接口契约、异常处理三章中是否都有独立的子节？如有组件缺失 → 补充分析或从概述中移除该组件。

1. **概述**（必填）：
   - 核心职责的一句话精确定义
   - 在系统中的定位
   - 负向限制：如实列出模块明确不做什么

2. **核心数据模型**（必填）：
   - 列出核心字段
   - 实体间关系文字说明
   - 置信度仅标注 ⚠️/❓

3. **对外接口契约**（必填）：
   - 每个公开 API：签名、参数、返回值、副作用、幂等性

4. **核心业务流程**：
   - 如有 → Mermaid 序列图，使用实际函数名/模块名
   - 图后简短摘要（2-3 行），不逐行重复图中信息
   - 如无 → 标准措辞

5. **状态机**：如有 → 状态表。如无 → 标准措辞

6. **异常处理与容错**：如有 → 异常处理表。如无 → 标准措辞

7. **并发安全**：如有 → 风险点表。如无 → 标准措辞

8. **性能特征**：如有 → 复杂度 + 资源密集点。如无 → 标准措辞

9. **已知问题与陷阱**：→ 写入 `04-问题与改进.md` > 模块陷阱，不在各模块文档中重复

### 写入观察便签

在 `.ai/project-log.md` 底部的「诊断观察便签」区段中追加「阶段5 观察」子节，仅记录 文件:行号 + 一句话描述：

示例：
```markdown
### 阶段5 观察
- [scanmodel.cpp:438-463] m_devDetailMap + m_pCompAry 双存储偏离风险
- [listmodel.cpp:25] append() 被 CMsgThread 跨线程调用，内部无锁
- [scanmodel.cpp:22-27] QMutexTryLocker(500ms) 超时后消息静默丢失
- [simtaskstruct.h:45] SimTask::pComp 是 weak_ptr，lock() 可能悬空
```

**便签规则**：只记录位置 + 事实，不评级、不分类。不确定的线索也写入，标注"? 需阶段6确认"。

## 产出后自检

每个模块文件生成后立即逐项核对：

**Agent 机械自检**：
- [ ] 概述含核心职责精确定义
- [ ] 核心数据模型的关键字段已列出
- [ ] 全部公开 API 的契约完整（含副作用和幂等性）
- [ ] 所有 8 个章节全部出现（无遗漏）
- [ ] 不涉及的章节使用了标准措辞（无展开、无解释）
- [ ] Mermaid 图后有简短摘要（非逐行重复）
- [ ] 观察便签已写入 project-log.md（跨线程交互、数据陷阱、哨兵值等线索）
- [ ] 跨线程交互专项检查已完成（5 个检查点）
- [ ] 多组件模块组织规则已遵守（如有 N≥2 组件，各章均有独立子节）
- [ ] 章节编写顺序已按推荐顺序执行（异常/并发 → 流程/状态机 → 接口/数据 → 概述）
- [ ] **深度检查**：异常处理章节是否仅为标准 N/A 话术？如是，确认代码中确实无任何异常处理——检查每个 public 方法的 return 语句
- [ ] **深度检查**：并发安全章节是否仅为标准 N/A 话术？如是，确认模块确实无共享状态——检查所有成员变量的访问线程
- [ ] 文件名使用中文（取自模块的人类可理解名称）
- [ ] **置信度统计**（必须在输出确认清单前完成）：
  a. 逐条列出各模块文档中所有可信度标记（编号 + 当前等级）
  b. 汇总：已确认 N 处、推断 M 处、推测 K 处
  c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
  d. 将结果写入对应模块 front matter 的 `confidence` 字段

语义质量项（设计描述准确性、可信度标注合理性等）由确认清单覆盖。

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新各模块文档**：遍历各模块文件中所有推断/推测标记——人类确认的 → 改为已确认
2. **重新统计置信度**：
   a. 重新遍历各模块文件中所有可信度标记（因第1步可能改变了标记）
   b. 输出统计：已确认 N 处、推断 M 处、推测 K 处
   c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
   d. 更新各模块 front matter：`confidence`、`last_updated`（今天日期）、`status`（`reviewed` 如果人类确认）
3. **在 project-log.md 追加记录**：`[日期] 阶段5 经人类确认，N 处推断/推测升级为已确认`

只有完成以上步骤后，才进入阶段 6。

## 人类回复后分支处理

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 | 执行上方「强制更新步骤」更新各模块文档和 front matter |
| 某模块内容需要修改 | 按人类指示修改该模块文件；修改后内容标记为已确认；然后执行「强制更新步骤」 |
| 补充了新信息 | 追加到对应模块文件，标记为已确认；然后执行「强制更新步骤」 |
| 有内容不确定 | 对应内容保持原可信度标记；更新 `last_updated` |
| 人类未回复/跳过 | 所有内容保持原可信度；继续下一阶段 |
