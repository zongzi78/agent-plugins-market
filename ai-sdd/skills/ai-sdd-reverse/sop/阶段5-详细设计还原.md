# 阶段 5：详细设计还原

**目标**：还原各模块的详细设计。

**核心原则**：如实描述——模块实际涉及什么就写什么，不涉及就标准声明"不涉及"。章节一个不落，深度自然有深有浅。

> ⚠️ 执行前提：本阶段涉及 confidence 标记。开始前必须先 Read `skills/ai-sdd-reverse/术语与概念体系.md`，确认理解：
> - confidence 三级定义：`confirmed`（已确认）= 人类确认过 / `inferred`（推断）= 有代码证据但未确认 / `speculative`（推测）= 证据不足
> - front matter 的 `confidence` 字段取整体最高规则：全部 confirmed → `confirmed`；存在 speculative → `speculative`；否则 `inferred`
> - `status` 字段：人类确认后 → `reviewed`；全部 confirmed → `baseline`

## 执行步骤

### 章节规则

所有 8 个章节必须全部出现。每章如实描述模块的实际情况：
- 有相关内容 → 按模板 GEN 引导详细编写
- 没有相关内容 → 写标准措辞（见 `03-详细设计_模板.md` 的 `<!-- GEN: 章节规则 -->`），一字不改

禁止以任何理由跳过章节。

### 对每个模块执行

每个文件包含以下章节：

1. **概述**（必填）：
   - 核心职责的一句话精确定义
   - 在系统中的定位
   - 负向限制：如实列出模块明确不做什么

2. **核心数据模型**（必填）：
   - 列出核心字段
   - 实体间关系文字说明
   - 置信度仅标注 ⚠️/❓

3. **对外接口契约**（必填）：
   - 每个公开 API：签名、参数、返回值、副作用、幂等性

4. **核心业务流程**：
   - 如有 → Mermaid 序列图，使用实际函数名/模块名
   - 图后简短摘要（2-3 行），不逐行重复图中信息
   - 如无 → 标准措辞

5. **状态机**：如有 → 状态表。如无 → 标准措辞

6. **异常处理与容错**：如有 → 异常处理表。如无 → 标准措辞

7. **并发安全**：如有 → 风险点表。如无 → 标准措辞

8. **性能特征**：如有 → 复杂度 + 资源密集点。如无 → 标准措辞

9. **已知问题与陷阱**：→ 写入 `04-问题与改进.md` > 模块陷阱，不在各模块文档中重复

### 写入 04-问题与改进.md

> ⚠️ 范围：本阶段写入「模块陷阱」「设计债务」「代码债务」三个子节。
> 不要修改架构层、需求层、技术层的内容。

三个阶段的问题在一次代码扫描中同时识别，分别写入三个子节。判断标准：
- **模块陷阱 (MT)**：使用者需要注意什么？（使用视角）→ 哨兵值、外部进程依赖、调用顺序、资源泄漏
- **设计债务 (DD)**：设计有什么问题，应该怎么改？（演进视角）→ 双重数据结构、双向映射、SRP、命名债务
- **代码债务 (CD)**：具体代码行有什么质量问题？（实现视角）→ TODO、魔数、死代码、注释代码、硬编码

#### 模块陷阱（MT-XXX）

从各模块详细设计中提取（"使用者需要注意什么"）：
- 数据一致性陷阱（双重数据结构、索引依赖不变量）→ 仅记录"使用时的风险"，设计改进方向记录在 DD
- 哨兵值陷阱（如 order=-1 的特殊含义，调用方需区分处理）
- 外部进程依赖陷阱（引擎崩溃/挂起时无超时强制终止）
- 调用顺序依赖（必须先调 A 再调 B，否则状态错乱）
- 资源泄漏风险（线程未正确终止、锁未释放、weak_ptr 悬空）

每条含：模块名、陷阱描述、严重程度、触发场景、当前规避方式。
编号：MT-001, MT-002, ...
无发现时写"各模块未识别明显陷阱"。

#### 设计债务（DD-XXX）

从各模块详细设计中提取（"设计有什么问题，应该怎么改"）：
- 双重数据结构（同一数据在两个独立结构中并行维护，需手动同步）
- 双向映射手动维护（name↔ID、parent↔child 等映射对需人工保证一致）
- 模块级 SRP 违反（单模块承担 ≥3 种不相关职责）
- 命名债务（命名保留了旧系统/旧概念的遗迹）
- 接口设计不一致（同一概念在不同模块用不同方式表达）
- 隐藏耦合（通过 extern 函数/全局变量形成的非显式依赖）

> ⚠️ 与模块陷阱的区别：陷阱是"使用者需要注意什么"（不改设计的前提下的规避），
> 设计债务是"设计者应该改什么"（改进设计的建议）。
> 同一个问题可能同时出现在 MT 和 DD：MT 描述触发场景和当前规避，DD 描述设计缺陷和改进建议。
> 此时两个条目互相引用（MT-XXX 的"当前规避"列写"见 DD-XXX"）。

每条含：描述、类型、严重程度、影响范围、修复建议。
编号：DD-001, DD-002, ...
无发现时写"各模块未识别明显设计债务"。

#### 代码债务（CD-XXX）

从各模块代码中提取（"具体代码行有什么质量问题"）：
- TODO / FIXME / HACK / XXX / OPTIMIZE 注释
- 注释掉的代码块（连续 ≥3 行被注释）
- 魔数（超时值、缓冲大小、业务常量等不应硬编码的裸值）
- 空函数体（方法体仅含 `{ }`）、空 catch 块
- 硬编码 URL / IP / 端口 / 文件路径 / 凭证
- 已声明但从未被调用的函数/变量（死代码）
- 混用新旧 API 语法（如旧式 SIGNAL()/SLOT() 宏和新式 & 语法并存）

每条含：位置（文件:行号）、问题类型、描述、严重程度、修复建议。
编号：CD-001, CD-002, ...
无发现时写"未识别明显代码债务"并简述搜索范围。

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新各模块文档**：遍历各模块文件中所有推断/推测标记——人类确认的 → 改为已确认
2. **重新统计置信度**：
   a. 重新遍历各模块文件中所有可信度标记（因第1步可能改变了标记）
   b. 输出统计：已确认 N 处、推断 M 处、推测 K 处
   c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
   d. 更新各模块 front matter：`confidence`、`last_updated`（今天日期）、`status`（`reviewed` 如果人类确认）
3. **在 project-log.md 追加记录**：`[日期] 阶段5 经人类确认，N 处推断/推测升级为已确认`

只有完成以上步骤后，才进入阶段 6。

## 人类回复后分支处理

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 | 执行上方「强制更新步骤」更新各模块文档和 front matter |
| 某模块内容需要修改 | 按人类指示修改该模块文件；修改后内容标记为已确认；然后执行「强制更新步骤」 |
| 补充了新信息 | 追加到对应模块文件，标记为已确认；然后执行「强制更新步骤」 |
| 有内容不确定 | 对应内容保持原可信度标记；更新 `last_updated` |
| 人类未回复/跳过 | 所有内容保持原可信度；继续下一阶段 |
