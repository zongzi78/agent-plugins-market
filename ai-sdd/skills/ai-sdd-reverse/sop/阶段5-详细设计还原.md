**目标**：还原各模块的详细设计。

**核心原则**：深度与模块权重成正比——重要模块深入分析，简单模块如实简述。章节一个不落，深度自然有深有浅。

> - 质量规则（正向/负向） → `../SKILL.md`「1. 通用规则」

## 综合权重评估规则

七维度定义与评分锚点见 00-架构.md 模板「综合权重评估」引导注释（唯一权威源）。
此处仅保留阶段5 特有的决策流程。

**权重判定**：
- 高权重模块（总分 >=24）-> 建议全章节展开
- 中权重模块（总分 14-23）-> 建议关键章节展开
- 低权重模块（总分 <=13）-> 建议精简

无VCS历史时，变更热度默认3分（中性）。

## 深度决定规则：权重建议 + 内容条件

Agent 不能仅凭权重等级机械决定章节展开。必须执行两步判断：

### 步骤 A：权重评估（得出一级建议）
按「综合权重评估规则」评估7个维度，得到总分。
总分 >=24 → 高权重建议；14-23 → 中权重建议；<=13 → 低权重建议。

### 步骤 B：逐章内容条件检查（得出最终展开决策）
对每个章节，不论权重建议如何，按以下条件独立判断是否展开：

| 章节 | 展开触发条件 |
|------|------------|
| 核心数据模型 | 模块拥有 >=1 个 struct/class 且字段数 >=3 |
| 对外接口契约 | 模块有 >=1 个 public 方法/函数 |
| 核心业务流程 | 模块参与 >=1 条跨模块数据流（见00-架构.md）或其方法调用链深度 >=3 |
| 状态机 | 模块管理 >=1 个有生命周期状态流转的实体（状态数 >=3） |
| 异常处理与容错 | 代码中存在 try-catch/错误回调/降级/重试/超时/熔断逻辑 |
| 并发安全 | 模块有共享状态（成员变量被 >=2 个线程访问 或 有跨线程信号槽连接） |
| 性能特征 | 模块有 O(n²) 或更高复杂度的算法 或 涉及 I/O/网络/大内存分配 |

条件满足 → 展开该章节（即使权重建议精简）。
条件不满足 → 使用标准措辞。

### 步骤 C：深度一致性验证
展开完成后，做最终检查：
展开完成后逐章复核：
- 高权重模块：是否有内容条件满足但被跳过的章节？
- 低权重模块：展开的章节是否都有充分的内容支撑（而非大量低质量填充）？
- 如果某个权重偏低但内容条件触发了大量章节展开 → 确认条件判断是否有误

**所有章节必须全部出现，但各章节的展开深度由以上条件决定。**
禁止以任何理由跳过章节。不涉及的章节使用标准措辞（见 `03-详细设计_模板.md` 的 `<!-- GEN: 章节规则 -->`），一字不改。

## 步骤 0：模块分组方案（强制，在编写任何详细设计文件之前执行）

**目标**：将 00-架构.md 中的 N 个模块映射到 M 个详细设计文件，产出分组方案。
Agent 自行检查合并合理性，不需要人类确认。

### 默认策略：一模块一文件

默认情况下，00-架构.md 中的每个模块对应一个详细设计文件。合并是例外，需要论证。

### 合并条件（三条件必须全部满足才允许合并）

两个或多个模块合并为一个详细设计文件，必须同时满足：

1. **共享接口定义**：各模块实现同一抽象接口的不同变体
   - 可验证：存在虚基类/抽象类/接口类被这些模块共同继承
   - 反例：仅因为"都用数据库"不是共享接口

2. **同库编译**：各模块的源文件编译为同一个 .dll/.so/.a 目标
   - 可验证：检查 CMakeLists.txt / *.pro 中的 target 定义
   - 反例：编译为不同可执行程序 → 不满足

3. **编译期互锁**：任一模块的移除会导致其他模块编译失败（不是链接失败）
   - 可验证：头文件层面的依赖——A 的 .h 包含 B 的 .h 中的类型定义
   - 反例：仅运行时通过信号槽连接 → 不满足

**不满足任一条件 → 必须拆分为独立文件。**

### 合并补充约束（即使三条件满足，以下情况也应拆分）

- 任一模块权重为高（>= 24）→ 不得合并，必须独立成文件
- 合并后文档预估 > 15KB（~400行） → 拆分为独立文件

### 文件名命名规则

文件名不得使用"与/和/及"连接不相关的概念。

例外条件（与合并条件一致，三条件必须全部满足才允许使用连接词）：
1. 共享接口定义 — 各子模块实现同一抽象接口（有明确的虚基类定义）
2. 同库编译 — 编译为同一个二进制目标
3. 编译期互锁 — 头文件层面不可分离（移除A导致B编译失败）

不满足任一条件 → 必须拆分为独立文件并使用各自的文件名。
满足全部三个条件 → 可使用连接词，但需在概述中注明三条件满足的证据。

### 分组方案格式

产出以下分组方案表：

| 详细设计文件 | 涵盖的架构模块 | 合并理由（引用合并条件编号） | 自检结果 |
|-------------|--------------|--------------------------|---------|
| 通信抽象层.md | bus(comm_newservices/comm_rtb/comm_mock) | 满足条件1(IServiceBus接口)+条件2(同lib)+条件3(头文件互锁) | ✅ 充分 |
| 数据库抽象层.md | db(db_odbc) | 默认一模块一文件 | ✅ 充分 |

### 自检（必须逐项通过）

- [ ] 每个 00-架构.md 中的模块至少在分组方案中出现一次（全覆盖）
- [ ] 每个合并决策满足全部三个合并条件，且不触发任何补充约束
- [ ] 文件名不含"与""和""及"（除非三条件全部满足 + 补充约束全部不触发），且文件名使用中文
- [ ] 被合并的模块在概述中逐个列出，标注各自的权重

### 倾向性原则

不确定是否应该合并 → **不合并**。默认策略是一模块一文件，合并是例外且需要充分论证。

## 执行步骤

### 章节编写顺序（强制）

不要按模板的线性顺序写。按分析难度倒序，确保最高注意力用于最难分析的内容：
1. **先写**异常处理 + 并发安全（最需要深度分析，消耗最大）
2. **再写**核心业务流程 + 状态机
3. **再写**对外接口契约 + 核心数据模型（最容易从代码提取）
4. **最后写**概述（此时已对整个模块有深入理解，概述质量最高）

### 对每个模块执行

每个文件包含以下章节：

1. **概述**（必填）：
   - 核心职责的一句话精确定义
   - 在系统中的定位
   - 权重评估结果（如"高权重：被 12 个模块依赖，2200 行，外部可见性3，核心业务引擎"）
   - 负向限制：如实列出模块明确不做什么
   - **设计决策关联**：列出本模块涉及的 D-XXX 编号（一行一句话简介）。
     如 02-决策记录.md 中缺少对应的决策条目 → 先补充到 02，再在此处关联。
     禁止重复展开决策理由——理由只在 02 中存在一份

2. **核心数据模型**（必填）：
   - 列出核心字段
   - 实体间关系文字说明

3. **对外接口契约**（必填）：
   - 每个公开 API：签名、参数、返回值、副作用、幂等性

4. **核心业务流程**：
   - 如有 -> Mermaid 序列图，使用实际函数名/模块名
   - 图后简短摘要（2-3 行），不逐行重复图中信息
   - 如无 -> 标准措辞

5. **状态机**：如有 -> 状态表。如无 -> 标准措辞

6. **异常处理与容错**：如有 -> 异常处理表。如无 -> 标准措辞

7. **并发安全**：如有 -> 风险点表。如无 -> 标准措辞

8. **性能特征**：如有 -> 复杂度 + 资源密集点。如无 -> 标准措辞

9. **已知问题与陷阱**：-> 写入 `04-问题与改进.md` > 模块陷阱，不在各模块文档中重复

### 跨线程交互专项检查

在编写并发安全章节时，必须检查以下跨线程交互：

- [ ] 全局容器（g_ 前缀）的所有读写点 -> 标记所在线程
- [ ] QCoreApplication::postEvent 传递的数据 -> 检查发送线程和接收线程
- [ ] 信号槽跨线程连接（Qt::QueuedConnection）-> 检查传递的数据是否线程安全
- [ ] QMutexTryLocker 超时路径 -> 检查是否有回退/缓存/重试机制
- [ ] weak_ptr::lock() 调用点 -> 检查返回值是否判空

每个检查点如有风险 -> 记录为观察便签

### 多组件模块组织规则（强制）

如果一个详细设计文档覆盖 N >= 2 个独立类/组件：
- 核心数据模型 -> 每个组件必须有独立的字段表 + 约束
- 对外接口契约 -> 每个组件必须有独立的公开方法表
- 异常处理与容错 -> 每个组件必须有独立的异常场景表

**禁止**：在概述中列出某个组件但在后续章节中不再出现。

Agent 自检：概述中列出的每个组件，在数据模型、接口契约、异常处理三章中是否都有独立的子节？如有组件缺失 -> 补充分析或从概述中移除该组件。

### 写入观察便签

→ 格式与规则见 project-log.md 模板「诊断观察便签」区段（唯一权威源）。

示例：
```markdown
### 阶段5 观察
- [DD] [scanmodel.cpp → ScanModel::updateDevDetailMap()] m_devDetailMap + m_pCompAry 双存储偏离风险
- [MT] [listmodel.cpp → ListModel::append()] 被 CMsgThread 跨线程调用，内部无锁
- [MT] [scanmodel.cpp → ScanModel::processScanResult()] QMutexTryLocker(500ms) 超时后消息静默丢失
- [MT] [simtaskstruct.h → SimTask::pComp] weak_ptr，lock() 可能悬空
```

**疑似约束模式标注**：深入阅读代码时如发现重复出现的规则/模式，标注为疑似约束（格式同阶段2「写入观察便签」→ 疑似约束模式标注）。供人类确认后填入 `.ai/supplement-rules.md` 项目特定约束节。

## 常见错误（反例对照）

### 错误1：模块过度合并
❌ 错误：bidataproxy + emthmiproxy + emtsimplugin + gtsolhmi 合并为
    "HMI数据代理与图表插件.md"——它们只有调用关系，不满足三个合并条件
✅ 正确：一模块一文件。仅在满足全部三个合并条件（共享接口+同库编译+编译期互锁）
    时才合并。

### 错误2：低权重模块一律用标准措辞
❌ 错误：数据库模块代码量小但运维价值极高（独立部署、专有配置），
    仅因权重低就用标准措辞，丢失 SQL DDL、部署步骤等运维关键信息
✅ 正确：对每个章节独立执行内容条件检查。数据库模块虽然代码量小，
    但有数据模型（>=3字段）、有接口契约（public方法）、有异常处理——逐章判断，
    有则如实展开，无则标准措辞。不因综合权重低而跳过条件检查。

### 错误3：已有文档时只改元数据不重写内容
❌ 错误：检测到已有详细设计文件后，保留全部内容，只更新 front matter 日期
✅ 正确：无论是全新生成还是已有文档，都遵循"创建心态"——从头分析代码，
    按当前模板重新组织内容
