# 阶段 5：详细设计还原

**目标**：还原各模块的详细设计。

**核心原则**：深度自适应——每个模块都独立生成详细设计文件。复杂模块自然产出长文档，简单模块自然产出短文档。不强制统一深度。

> ⚠️ 执行前提：本阶段涉及 confidence 标记。开始前必须先 Read `skills/ai-sdd-reverse/术语与概念体系.md`，确认理解：
> - confidence 三级定义：`confirmed`（已确认）= 人类确认过 / `inferred`（推断）= 有代码证据但未确认 / `speculative`（推测）= 证据不足
> - front matter 的 `confidence` 字段取整体最高规则：全部 confirmed → `confirmed`；存在 speculative → `speculative`；否则 `inferred`
> - `status` 字段：人类确认后 → `reviewed`；全部 confirmed → `baseline`

## 执行步骤

对每个一级模块（00-架构.md 中已列出的模块），生成一个详细设计文件。

每个文件包含以下章节：

1. **概述**（必填）：
   - 核心职责的一句话精确定义
   - 在系统中的定位
   - 负向限制：本模块绝对不做什么（有多少写多少，通常 2-5 条；如果确实只识别出 1 条就写 1 条并注明"仅从代码中识别出此一条"）

2. **核心数据模型**（必填）：
   - 列出所有持久化字段，穷尽不省略
   - 每个字段标注可信度和代码证据(`文件:行号`)
   - 实体间关系文字说明
   - 关键数据约束

3. **对外接口契约**（必填，每个公开 API）：
   - 方法 + 路径
   - 请求参数（所有参数，含类型/必填/默认/验证规则）
   - 成功响应格式
   - 所有已知错误码及触发条件
   - 副作用说明（读写哪些全局状态）
   - 幂等性设计说明

4. **核心业务流程**：
   - ≥1 个 Mermaid 序列图，使用实际函数名/模块名
   - 每步附带异常处理说明
   - 工具类/纯计算模块如无复杂流程，写明"本模块为无状态工具模块，无复杂流程"并省略图

5. **状态机**（如适用）：如果模块管理有生命周期流转的实体，列出完整状态机

6. **异常处理与容错**（必填）：
   - 已知异常域
   - 处理策略表
   - 降级兜底逻辑
   - 如果模块确实没有异常处理代码，写"未发现异常处理代码，存在安全隐患"并标记为技术债务

7. **并发安全**（必填）：
   - 并发风险点
   - 锁与事务机制
   - 如果模块无共享状态，写"本模块为无状态工具模块，不涉及共享状态，无并发风险"

8. **性能特征**（必填）：
   - 核心路径复杂度
   - 缓存策略（如有）
   - 资源密集点
   - 如果模块非常简单（纯计算、无 I/O），写明原因

9. **已知问题与陷阱**：在本模块代码中发现的具体问题

10. **模块接口图**：对外提供和依赖消费的接口一览

## 产出

`.ai/doc/03-详细设计/<模块中文名>.md`
- **文件名必须优先使用中文**（如"用户认证模块.md"、"订单管理模块.md"），取自模块在系统中的实际人类称呼；专有名词/技术组件名可用英文（如"Redis缓存模块.md"）

## 产出后自检

每个模块文件生成后立即逐项核对：
- [ ] 概述含核心职责精确定义 + 负向限制（≥2 条；少于 2 条时说明原因）
- [ ] 核心数据模型的关键字段穷尽列出
- [ ] 全部公开 API 的契约完整（含副作用和幂等性说明）
- [ ] ≥1 个 Mermaid 业务流程图（使用实际函数/模块名；工具/纯计算模块如无复杂流程可省略并写明原因）
- [ ] 异常处理/并发安全/性能特征三个专题全部填写（不适用时写"不适用"并原因，不跳过）
- [ ] 文件名使用中文（如"用户认证模块.md"），取自模块在系统中的人类可理解名称

## 确认清单

产出后暂停，输出以下确认清单：

```
📋 阶段 5 确认清单 — 详细设计还原

以下是各模块的设计描述摘要：
[每个模块列出：模块名、API 数量、数据实体数量、关键设计点摘要]

请检查：
□ 各模块的设计描述是否准确反映了代码实际？
□ 异常处理/并发安全/性能特征三个专题的分析是否有错误或遗漏？
□ 标注推测的内容是否需要人工澄清或补充？
□ 隐式约束是否完善？是否有未发现的设计隐患？

如有问题请直接说明（如"用户模块的 API 描述不对，XX 接口实际上是做 YY 的"）。
```

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新各模块文档**：遍历各模块文件中所有标记为 `推断` 或 `推测` 的声明——人类确认的 → 改为 `已确认`
2. **更新各模块 front matter**：
   - `last_updated`: 今天日期
   - `status`: `reviewed`（人类确认后）/ 保持 `draft`（未回复或跳过）
   - `confidence`: 按整体规则计算（全部已确认 → `confirmed`；存在推测 → `speculative`；否则 `inferred`）
3. **在 project-log.md 追加记录**：`[日期] 阶段5 经人类确认，N 处推断/推测升级为已确认`

只有完成以上步骤后，才进入阶段 6。

## 人类回复后分支处理

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 | 执行上方「强制更新步骤」更新各模块文档和 front matter |
| 某模块内容需要修改 | 按人类指示修改该模块文件；修改后内容标记为 `已确认`；然后执行「强制更新步骤」 |
| 补充了新信息 | 追加到对应模块文件，标记为 `已确认`；然后执行「强制更新步骤」 |
| 有内容不确定 | 对应内容保持原可信度标记；更新 `last_updated` |
| 人类未回复/跳过 | 所有内容保持原可信度；继续下一阶段 |
