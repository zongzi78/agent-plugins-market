# 阶段 2：架构梳理

**目标**：还原系统架构。

> **本 SOP 的上下文**：
> - 文件角色定义 → `../SKILL.md`「2. 文件角色对照表」
> - 质量规则（正向/负向） → `../SKILL.md`「1. 通用规则」
> - 产出模板路径见 SKILL.md「各阶段资源映射」表
> - 全局约束：代码证据使用符号锚点格式（ClassName::methodName() 或 filename.cpp -> functionName()）

## 执行步骤

1. **分析模块依赖**：扫描 import/include 语句、包结构、模块调用链，识别模块间依赖关系

2. **识别数据流向**：追踪关键业务流程的调用链路。
   **L1/L2边界规则（强制）**：
   - L1（本文档 00-架构.md）：以模块为粒度描述跨模块数据传递，
     每个步骤 = "模块A → 模块B，传递什么数据、什么形态变化"
   - L2（03-详细设计）：模块内部的函数调用链、条件分支、循环逻辑
   - 判断标准：描述的是"模块内部如何实现"（如"创建XX实例""取最后N个采样点平均值"）
     → 属于L2，不应出现在此处
   - 每条数据流至少涉及2个不同模块
   - **数据流数量**：由项目的端到端业务场景自然决定，不设最少/最多限制。
     从系统启动流程、每条写路径、每条读路径、系统间交互路径、定时/后台任务路径逐一排查

3. **识别外部依赖**：从配置文件、连接代码中提取外部系统和中间件

4. **绘制依赖图**：用 Mermaid 绘制模块依赖图
   - 每个模块必须是独立的节点
   - 按层级分组（表示层 / 业务逻辑层 / 数据访问层 / 基础设施层）
   - 箭头方向表示依赖关系（A -> B 表示 A 依赖 B）
   - 如存在循环依赖，如实画出——循环依赖条目写入观察便签

5. **扫描技术债务**：搜索 TODO/FIXME/HACK/XXX 注释、注释掉代码块（>=3行）、魔数、空catch块/空函数体、硬编码URL/IP/端口/路径/凭证、死代码。结果写入观察便签（见下方「写入观察便签」）。

6. **搜索架构违规**：搜索循环依赖、跨层直接调用、防腐层绕过、共享内核泄漏、God Class（单类>=3种职责）。结果写入观察便签（见下方「写入观察便签」）。

7. **填充认知边界地图**（三子表）：
   - 已知已知表：通过阅读具体代码确认的事实（每条带符号锚点证据，如 ClassName::methodName() 或 filename.cpp -> functionName()）
   - 已知未知表：AI 无法确定、需要人工补充的内容（写具体问题）
   - 推断结论表：从代码模式推断但未确认的设计决策

### 写入观察便签

在 `.ai/project-log.md` 底部追加「诊断观察便签」临时区段，仅记录 符号锚点 + 一句话描述，供阶段 6（集中诊断）消费：

示例：
```markdown
## 诊断观察便签（阶段2-5 临时记录，阶段6 消费后删除）

### 阶段2 观察
- [mainWindow.h -> MainWindow class] 疑似 God Class — ~40 成员 + ~30 方法，>=7 种职责
- [simtaskstruct.h -> SimTask::timeout] 硬编码 timeout(3000)，无常量定义
- [hs_structdefine.h -> HsStructDefine] 硬编码 etime=10; step=50E-6，无注释说明物理含义
```

**便签规则**：
- 只记录位置 + 一句话事实，不评级、不分类、不写修复建议
- 不重复记录同一位置（去重）
- 你无法确定的线索也写入，标注"? 需阶段6确认"
- 便签由阶段 6 的 ai-sdd-diagnose Skill 消费，消费后删除区段

## 常见错误（反例对照）

### 错误1：数据流步骤混入模块内部实现 [来源: v1.6.9 example]
❌ 错误：DF1 步骤5 "创建GTS设备模板实例(GtsDeviceTemplate)"——这是模块内部的实例化操作
✅ 正确：DF1 步骤应该是跨模块粒度的，如"计算节点 → 模型数据整合：传递设备模板参数"

### 错误2：架构文档使用推测性措辞 [来源: v1.6.8 跨版本对比]
❌ 错误："Agent推断该模块可能是用于..." "笔者推测这里的设计意图是..."
✅ 正确：所有结论以代码证据支撑。不确定的归入"认知边界地图→已知未知"，
    使用断言式语言（"该模块负责..."而非"该模块可能负责..."）

### 错误3：Mermaid图含外部库节点 [来源: v1.6.9 example]
❌ 错误：在模块依赖图中画出 Qt5::Widgets、SQLite 等外部库节点
✅ 正确：Mermaid图仅含项目内部模块，外部依赖在"外部依赖"表格中描述

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新文档内容**：按人类回复修改文档（修改/追加/删除）
2. **更新 front matter**：`last_updated`（今天日期）、`last_code_verified`（今天日期）、`status`（`reviewed`）
3. **在 project-log.md 追加记录**：`[日期] 阶段2 经人类确认`

只有完成以上步骤后，才进入阶段 3。

## 人类回复后分支处理

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 | 执行上方「强制更新步骤」更新文档和 front matter |
| 某内容需要修改 | 按人类指示直接修改文档；修改后的内容直接更新；然后执行「强制更新步骤」 |
| 补充了新信息 | 追加到对应章节；然后执行「强制更新步骤」 |
| 有内容不确定/跳过 | 对应内容保持待确认标记；更新 front matter `last_updated` |
| 人类指出了遗漏 | 补充分析，补全文档，重新输出确认清单 |
| 人类未回复/跳过 | 内容保持不变；继续下一阶段 |
