# 阶段 2：架构梳理

**目标**：还原系统架构。

> ⚠️ 执行前提：本阶段涉及 confidence 标记。开始前必须先 Read `skills/ai-sdd-reverse/术语与概念体系.md`，确认理解：
> - confidence 三级定义：`confirmed`（已确认）= 人类确认过 / `inferred`（推断）= 有代码证据但未确认 / `speculative`（推测）= 证据不足
> - front matter 的 `confidence` 字段取整体最高规则：全部 confirmed → `confirmed`；存在 speculative → `speculative`；否则 `inferred`
> - `status` 字段：人类确认后 → `reviewed`；全部 confirmed → `baseline`

## 执行步骤

1. **分析模块依赖**：扫描 import/include 语句、包结构、模块调用链，识别模块间依赖关系

2. **识别数据流向**：追踪关键业务流程的调用链路

3. **识别外部依赖**：从配置文件、连接代码中提取外部系统和中间件

4. **绘制依赖图**：用 Mermaid 绘制模块依赖图
   - 每个模块必须是独立的节点
   - 按层级分组（表示层 / 业务逻辑层 / 数据访问层 / 基础设施层）
   - 箭头方向表示依赖关系（A → B 表示 A 依赖 B）
   - 如存在循环依赖，如实画出——循环依赖条目写入 04-问题与改进.md > 架构违规

5. **扫描技术债务**：按下方「写入 04-问题与改进.md」中的搜索清单执行，结果写入 `04-问题与改进.md` > 技术债务

6. **搜索架构违规**：按下方「写入 04-问题与改进.md」中的搜索清单执行，结果写入 `04-问题与改进.md` > 架构违规

7. **填充认知边界地图**（三子表）：
   - 已知已知表：通过阅读具体代码确认的事实（每条带 `文件:行号` 证据）
   - 已知未知表：AI 无法确定、需要人工补充的内容（写具体问题）
   - 推断结论表：从代码模式推断但未确认的设计决策

### 写入 04-问题与改进.md

> ⚠️ 范围：本阶段只写入「技术债务」和「架构违规」两个子节。
> 不要修改需求层、技术层、设计层的任何内容。即使你在代码中发现了相关迹象（如空函数实现、需求名实不符等），留给阶段3/4/5处理。

#### 技术债务（TD-XXX）

搜索：
- TODO / FIXME / HACK / XXX / OPTIMIZE 注释
- 注释掉的代码块（连续 ≥3 行被注释）
- 魔数（超时值、缓冲大小、业务常量等不应硬编码的裸值）
- 空 catch 块、空函数体（方法体仅含 `{ }`）
- 硬编码 URL / IP / 端口 / 文件路径 / 凭证
- 已声明但未被调用的函数（死代码）

每条必须带具体 `文件:行号` 引用。表格格式见 04-问题与改进.md 模板。
无发现时写"未识别明显技术债务"并简述搜索范围。
编号：TD-001, TD-002, ...

#### 架构违规（AV-XXX）

搜索：
- 循环依赖（A→B→A 或 A→B→C→A）
- 跨层直接调用（表示层跳过业务逻辑层直接访问数据层）
- 防腐层绕过（直接调用不应直调的外部系统内部 API）
- 共享内核泄漏（本应隔离的模块间共享了内部实现细节）
- God Class（单类承担 ≥3 种不同职责）

每条必须说明：位置（文件:行号）、违反了什么原则、原因、修复计划。
无发现时写"当前未识别架构违规"。
编号：AV-001, AV-002, ...

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新文档内容**：遍历 00-架构.md 中所有标记为 `推断` 或 `推测` 的声明——人类确认的 → 改为 `已确认`
2. **重新统计置信度**：
   a. 重新遍历所有可信度标记（因第1步可能改变了标记）
   b. 输出统计：已确认 N 处、推断 M 处、推测 K 处
   c. 代入规则：存在推测 → `speculative`；全部已确认 → `confirmed`；否则 → `inferred`
   d. 更新 front matter：`confidence`、`last_updated`（今天日期）、`status`（`reviewed` 如果人类确认）
3. **在 project-log.md 追加记录**：`[日期] 阶段2 经人类确认，N 处推断/推测升级为已确认`

只有完成以上步骤后，才进入阶段 3。

## 人类回复后分支处理

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 | 执行上方「强制更新步骤」更新文档和 front matter |
| 某内容需要修改 | 按人类指示直接修改文档；修改后的内容标记为 `已确认`；然后执行「强制更新步骤」 |
| 补充了新信息 | 追加到对应章节，标记为 `已确认`；然后执行「强制更新步骤」 |
| 有内容不确定/跳过 | 对应内容保持原可信度标记；更新 front matter `last_updated` |
| 人类指出了遗漏 | 补充分析，补全文档，重新输出确认清单 |
| 人类未回复/跳过 | 所有内容保持原可信度；继续下一阶段 |

## 完成后更新 front matter

- `last_updated`: 今天日期
- `status`: `reviewed`（如果人类确认）或保持 `draft`（如果未回复）
- `confidence`: 按整体规则计算（全部已确认 → `confirmed`；存在推测 → `speculative`；否则 `inferred`）
