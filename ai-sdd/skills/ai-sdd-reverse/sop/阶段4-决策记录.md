# 阶段 4：决策记录

**目标**：技术栈盘点 + 从代码观察决策迹象 + Agent 必须给出推断理由（不能留空）。

> **本 SOP 的上下文**：
> - 文件角色定义 → `../SKILL.md`「2. 文件角色对照表」
> - 质量规则（正向/负向） → `../SKILL.md`「1. 通用规则」
> - 产出模板路径见 SKILL.md「各阶段资源映射」表

## 执行步骤

### 1. 技术栈分析

**分析依赖**：从包管理器文件中提取所有直接依赖项及其版本

**版本漂移评估**：每个关键依赖对比当前版本 vs 最新稳定版。
- 版本对比事实写入 `02-决策记录.md` > 版本漂移
- 风险判断写入 `04-问题与改进.md` > 版本与依赖风险

**构建/测试/部署验证**：从实际配置文件中提取工具链信息

**技术约束提取**：记录代码中隐含的技术禁止事项

### 2. 决策迹象观察

扫描代码，寻找可观察到的设计选择痕迹。分组记录：

**架构级决策**（影响多个模块或全系统）：
- 模块划分方式（如按功能/按层次/按领域）
- 通信方式选择（如 HTTP/消息队列/共享内存）
- 数据存储策略（如单库/多库、读写分离）
- 部署架构选择（如单体/微服务、容器化）
- 跨切面决策（如统一认证方式、日志策略、错误处理模型）

**设计级决策**（影响单个或少数模块）：
- 设计模式采用（如工厂/观察者/策略等，从代码结构识别）
- 数据结构选择（如为何用 map 而非 vector、为何自定义容器）
- 线程模型选择（如单线程事件循环/线程池/手动管理）
- 资源管理策略（如连接池大小、缓存策略、超时策略）

### 3. 推断理由（强制）

对每个观察到的决策迹象，Agent **必须**给出推断理由。格式：

```
Agent推断：[理由]。证据：[符号锚点]
```

状态标记为 `Inferred`。

**禁止**：对观察到的决策仅记录事实而不写推断理由。

### 4. 关于决策状态

决策记录使用4种状态，详细定义及Agent消费指引见 `02-决策记录.md` 模板的「状态说明」表。
Agent 必须遵守以下规则：
- 对每个观察到的决策迹象，**必须**给出推断理由，不允许留空
- 状态标记为 `Inferred` 的条目需在确认清单中请人类逐条验证
- 人类确认后状态从 `Inferred` 升级为 `Confirmed`

**状态流转规则**：
- 新发现的决策 → 状态 = `Inferred`（Agent有推断理由）或 `Observed`（Agent无法推断）
- 人类确认推断理由正确 → `Inferred` → `Confirmed`
- 人类纠正推断理由 → 更新理由，`Inferred` → `Confirmed`
- 人类补充了理由 → `Observed` → `Confirmed`
- 经人类确认理由已永久不可追溯 → `Untraceable`

### 5. 写入观察便签

→ 格式与规则见 project-log.md 模板「诊断观察便签」区段（唯一权威源）。

示例：
```markdown
### 阶段4 观察
- [DD] [mainWindow_slot2.cpp → g_globals] 7 个 g_ 前缀全局变量跨 4 个 .cpp 共享
- [DD] [scanmodel.h → ScanModel::m_devDetailMap/m_pCompAry] 双重数据结构
- [CD] [sim_action_adaptor.cpp → SimActionAdaptor::connect()] 混用旧式 SIGNAL()/SLOT() 宏（项目其余使用新式语法）
```

### 6. 模式诊断清单

本阶段在分析代码时，逐项审视以下维度（发现后写入观察便签，不自行写入 04）：

- [ ] 全局可变状态检查：扫描 g_ 前缀变量、extern 声明、static 非 const 变量
  - 跨编译单元共享？是否有同步保护？-> 记录便签

- [ ] 编码约定一致性检查：扫描是否存在混用的语法/命名风格
  - 新旧 Qt connect 语法混用？中英文命名混用？-> 记录便签

- [ ] RAII 合规检查：扫描裸 new/delete
  - 无智能指针包装？手动 lock/unlock 无 RAII 封装？-> 记录便签

- [ ] 职责过载检查：
  - 单类方法数 >20 且涉及 >=3 种数据域？-> 记录便签
  - 单方法 >80 行且含 >=3 种操作？-> 记录便签

## 模板 → 输出转换规则

从模板生成最终文档时，以下规则是机械操作：

✅ **原样保留**（从模板复制到输出，一字不改）：
- front matter（填充占位符后）
- 所有 `> **📋 ...**` 引用块及其全部嵌套内容（「本文档角色」「决策状态与规则」等）。这些块在 Markdown 渲染中可见，是跨 skill 共享知识，必须持久化在生成文档中
- 一级标题 `# ...`

❌ **删除**（这些是 Agent 引导注释，不出现在最终文档中）：
- 所有 `<!-- GEN: ... -->` 或 `<!-- ...引导 -->` HTML 注释块
- 自检清单 HTML 注释块（`<!-- Agent 格式自检... -->`）
- 任何包含"待填写"字样的行

📝 **填充**：
- 所有 `待填写` 占位符替换为实际内容
- 所有 `YYY-MM-DD` 替换为实际日期

⚠️ **禁止**：
- 在生成的文档中添加模板中不存在的章节或段落
- 修改 `📋` 引用块中的任何文字
- 在 `📋` 引用块和正文之间插入额外内容

## 常见错误（反例对照）

### 错误1：Observed/Inferred 状态标注不一致 [来源: v1.6.9 example D-112]
❌ 错误：D-112 标记为 Observed，但内容包含"可能因为...""推测是..."等推断表述
✅ 正确：Observed = 代码中直接可见的取舍结果（如"此处用QHash而非QMap"），
    无额外推断。有推断理由 → 状态应为 Inferred

### 错误2：决策条目全部标记为 Inferred 而不寻求人类确认 [来源: v1.6.9 example2 原始改造]
❌ 错误：21条决策全部 Inferred，确认清单被跳过
✅ 正确：阶段4暂停时必须列出所有Inferred条目，请人类逐条确认或纠正

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **升级决策状态**：人类确认了推断理由正确的 -> `Inferred` -> `Confirmed`；人类纠正了理由的 -> 更新理由并标记 `Confirmed`
2. **更新 front matter**：`last_updated`（今天日期）、`last_code_verified`（今天日期）、`status`（`reviewed`）
3. **在 project-log.md 追加记录**：`[日期] 阶段4 经人类确认，N 条决策升级为 Confirmed`

只有完成以上步骤后，才进入阶段 5。

## 人类回复后分支处理

| 人类反馈 | 你的操作 |
|---------|---------|
| 确认全部正确 / 回复"跳过" | 执行上方「强制更新步骤」后继续 |
| 某技术信息需要纠正 | 按人类指示直接修改文档；然后执行「强制更新步骤」 |
| 某推断理由需要修正 | 更新推断理由；标记为 Confirmed；然后执行「强制更新步骤」 |
| 补充了新决策或背景 | 追加到对应章节，标记为 Confirmed；然后执行「强制更新步骤」 |
| 人类将某 Inferred 升级 | 对应决策状态 -> Confirmed；然后执行「强制更新步骤」 |
| 人类未回复 | 所有内容保持原状态；继续下一阶段 |
