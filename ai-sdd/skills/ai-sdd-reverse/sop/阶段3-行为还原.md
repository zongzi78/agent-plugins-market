# 阶段 3：行为还原

**目标**：从 API/UI 入口提取系统可观察行为清单。每个行为标注状态（活跃 / 不完整 / 疑似废弃）。

> **本 SOP 的上下文**：
> - 文件角色定义 → `../SKILL.md`「2. 文件角色对照表」
> - 质量规则（正向/负向） → `../SKILL.md`「1. 通用规则」
> - 产出模板路径见 SKILL.md「各阶段资源映射」表
> - 全局约束：代码证据使用符号锚点格式（ClassName::methodName() 或 filename.cpp -> functionName()）

## 执行步骤

1. **扫描公开 API 入口**：扫描路由注册代码、控制器、RPC 服务定义、REST 端点等，列出所有对外接口
2. **扫描 UI 事件处理器**：扫描 UI 文件（.xaml/.xml/.html 等）中的事件绑定、按钮回调、菜单项处理函数
3. **为每个入口追踪实现**：从入口点向下追踪调用链，确定：
   - 该行为是否完整实现了用户可观察的效果（活跃）
   - 该行为是否有部分实现但缺失关键步骤（不完整）
   - 入口存在但实现体为空壳/已注释/长期无变更（疑似废弃）
4. **版本控制历史辅助分析**（仅阶段 0 确认可用时执行）：
   - 运行 `git log --oneline -n 100`（或 `svn log -l 100` / `hg log -l 100`）
   - 通过提交信息辅助判断：哪些行为是核心行为（高频 feat）、哪些可能已废弃（长期无变更 + 无外部入口）
5. **写入 B-XXX 行为条目**：为每个可观察行为创建条目，包含：
   - 行为名称
   - 用户可观察的效果
   - 入口（API 端点 / UI 控件 / 命令行）
   - 代码证据（符号锚点，如 ClassName::methodName()）
   - 当前状态（活跃 / 不完整 / 疑似废弃）
6. **声明-实现一致性检查**：
   - 对每个 public/protected 方法：函数名含动作词（create/send/update/delete/start/stop/pause/open/close/init）-> 检查函数体是否有对应副作用
   - 空函数体 + 动作词函数名 -> 该行为状态降为"不完整"
   - 函数体仅含 return/日志而无实际操作 -> 该行为状态降为"不完整"
   - 传入有意义的值但函数签名中参数名被注释（如 foo(int /\*unused\*/)）-> 记录为观察便签
   - 空 if/else/while/for 块 -> 检查是否曾有逻辑被移除 -> 记录为观察便签
7. **写入观察便签**：记录供阶段 6 诊断使用的线索（见下方「写入观察便签」）

### 写入观察便签

在 `.ai/project-log.md` 底部的「诊断观察便签」区段中追加「阶段3 观察」子节，仅记录 符号锚点 + 一句话描述：

示例：
```markdown
### 阶段3 观察
- [mainWindow_scan.cpp -> onPauseCalc()] 空函数体，函数名暗示"暂停"功能
- [simtaskmanager.cpp -> scheduleTasks()] 弹出任务但不分派
- [mainWindow.cpp -> openRecentProjects()] 调用传参 50，但函数体忽略该参数
```

**便签规则**：只记录位置 + 事实，不评级、不分类。不确定的线索也写入，标注"? 需阶段6确认"。

### "待确认"标记的使用限制

在行为条目中标注"待确认"时，必须遵守：

**允许条件**（必须同时满足）：
1. 已阅读了所有相关的代码路径（调用方、被调用方、错误处理分支）
2. 代码中确实不存在对该场景的处理逻辑
3. 无法从代码推断该场景是否会实际发生

**禁止**：
- 只因为没读完相关代码就标注 -> 必须先完成代码阅读
- 同一行为条目中有 >=3 个"待确认" -> 该行为的整体状态需重新评估
- 用"待确认"替代具体的分析

当标注"待确认"时，必须附带：
- 已检查的代码路径（符号锚点列表）
- 为什么代码无法回答这个问题
- 要确认需要什么信息或验证方式

- [ ] 每个可识别的行为对应一个 B-XXX 条目
- [ ] 每个行为含：行为名称、用户可观察的效果、入口、代码证据（符号锚点）、当前状态
- [ ] 每个行为标注了状态（活跃 / 不完整 / 疑似废弃）
- [ ] 声明-实现一致性检查已完成
- [ ] 观察便签已写入 project-log.md（声明-实现不一致、空函数体、命名偏离等线索）
- [ ] "待确认"标记符合使用限制（同一行为 <3 个；每个附带代码路径和原因）
- [ ] **符号锚点核查**：所有代码证据是否使用了符号锚点格式（ClassName::methodName() 或 filename.cpp -> functionName()）？行号仅作为附加缓存，不作为独立证据引用

## 确认清单

产出后暂停，输出以下确认清单：

```
📋 阶段 3 确认清单 — 行为还原

请对每个行为标记你的决定。可直接在行为编号后标注，如"B-001 确认"、"B-003 已废弃"、"B-005 需澄清——这个功能我不确定还在不在用"

行为列表：
[列出所有 B-XXX 编号 + 行为名称 + 当前状态]

快捷标记：
  [确认] — 这个行为确实存在且在（或应该）使用，状态判断准确
  [已废弃] — 这个行为已经不再使用
  [状态更正] — 状态标记不准确，应该是 [活跃/不完整/疑似废弃]
  [需澄清] — 我不确定，需要进一步调查（请说明不确定什么）

另外：
□ 行为描述是否符合实际？（用户可观察的效果是否准确？）
□ 观察便签是否有遗漏的线索需要补充？
□ 是否有遗漏的行为需要补充？
```

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新行为条目**：遍历所有行为条目——人类标记 `[确认]` 的 -> 确认当前状态标记；标记 `[已废弃]` 的 -> 状态改为"已废弃"；标记 `[状态更正]` 的 -> 按人类指示更新状态；标记 `[需澄清]` 的 -> 追加 `<!-- 待确认 -->` 注释
2. **更新 front matter**：`last_updated`（今天日期）、`last_code_verified`（今天日期）、`status`（`reviewed`）
3. **在 project-log.md 追加记录**：`[日期] 阶段3 经人类确认，N 个行为标记确认、M 个标记废弃、K 个需澄清`

只有完成以上步骤后，才进入阶段 4。

## 人类回复后分支处理

| 人类标记 | 你的操作 |
|---------|---------|
| `[确认]` | 执行上方「强制更新步骤」后继续 |
| `[已废弃]` | 该行为状态 -> `已废弃`；执行上方「强制更新步骤」后继续 |
| `[状态更正]` | 按人类指示更新状态；执行上方「强制更新步骤」后继续 |
| `[需澄清]` | 行为尾部追加 `<!-- 待确认：人类标记需澄清 — YYYY-MM-DD。问题：[人类描述] -->`；执行上方「强制更新步骤」后继续 |
| 人类补充了业务信息 | 追加到对应行为描述；然后执行「强制更新步骤」 |
| 人类要求修改 | 按指示改文档；然后执行「强制更新步骤」 |
| 人类指出了遗漏行为 | 分析遗漏的代码区域，补充该行为条目，重新输出确认清单（跳过强制更新步骤——等下一轮确认） |

## 处理完成后

- 更新 front matter：`last_updated`、`status: reviewed`（如果人类确认了大部分）、`last_code_verified`（今天日期）
- 如果有 `[需澄清]` 的行为，在 project-log.md 的"待确认"列表中追加记录
