**目标**：从 API/UI 入口提取系统可观察行为清单。每个行为标注状态（活跃 / 不完整 / 疑似废弃）。

## 行为粒度标准（强制）

在扫描和创建行为条目之前，必须先理解"一个行为"的定义。

### 一个行为的四个强制条件

一个行为条目必须同时满足以下四个条件：
1. **可观察效果**：用户或外部系统能感知到某个效果发生了（状态变化、输出产生、副作用执行）
2. **可识别入口**：存在明确的触发点（UI控件/API端点/CLI命令/定时器/系统事件）
3. **可独立触发**：该行为可以不依赖其他行为的内部状态而被单独触发
4. **完成状态可验证**：从代码中可以判断该行为是否已执行完成（有明确的结束条件或返回值）

### 反例：什么不算一个行为

- ~~"系统启动"~~ → 应当拆分为具体的启动步骤：加载配置(B-001)、建立连接(B-002)、初始化UI(B-003)
- ~~"数据管理"~~ → 太模糊，无法对应到具体入口
- ~~"MainWindow::onTimer() 内部调用 updateUI()"~~ → 无独立入口，是其他行为的内部步骤

### 行为分组（可选）

如果多个行为服务于同一用户目标（如"项目管理"下的"新建/打开/保存/关闭"），
可在行为名称中使用前缀进行逻辑分组，如"项目管理-新建项目""项目管理-打开项目"。
这不是必须的层级结构，仅用于提高可读性。

### 粒度边界

- 如果一个UI操作触发了一条完整的处理链路（如"点击扫描按钮" → 读取配置 → 执行计算 → 显示结果），这是一个行为，不要拆成3个
- 如果一个API有多个独立模式（如 query(type=JSON) vs query(type=XML) 走不同代码路径），这是多个行为
- 决策依据：不同的入口类型或不同的代码路径 → 不同行为；同一个入口同一个代码路径 → 同一个行为

## 执行步骤

1. **扫描公开 API 入口**：扫描路由注册代码、控制器、RPC 服务定义、REST 端点等，列出所有对外接口
2. **扫描 UI 事件处理器**：扫描 UI 文件（.xaml/.xml/.html 等）中的事件绑定、按钮回调、菜单项处理函数
3. **为每个入口追踪实现**：从入口点向下追踪调用链，确定：
   - 该行为是否完整实现了用户可观察的效果（活跃）
   - 该行为是否有部分实现但缺失关键步骤（不完整）
   - 入口存在但实现体为空壳/已注释/长期无变更（疑似废弃）
4. **版本控制历史辅助分析**（仅阶段 0 确认可用时执行）：
   - 运行 `git log --oneline -n 100`（或 `svn log -l 100` / `hg log -l 100`）
   - 通过提交信息辅助判断：哪些行为是核心行为（高频 feat）、哪些可能已废弃（长期无变更 + 无外部入口）
5. **写入 B-XXX 行为条目**：按模板表格列填写每个行为条目。
   模板列定义见 `01-行为目录.md`。
   - **写入后立即检查**：该行为是否满足「行为粒度标准」的四个强制条件？
     不满足 → 拆分或合并，直到每个行为都满足
6. **声明-实现一致性检查**：
   - 对每个 public/protected 方法：函数名含动作词（create/send/update/delete/start/stop/pause/open/close/init）-> 检查函数体是否有对应副作用
   - 空函数体 + 动作词函数名 -> 该行为状态降为"不完整"
   - 函数体仅含 return/日志而无实际操作 -> 该行为状态降为"不完整"
   - 传入有意义的值但函数签名中参数名被注释（如 foo(int /\*unused\*/)）-> 记录为观察便签
   - 空 if/else/while/for 块 -> 检查是否曾有逻辑被移除 -> 记录为观察便签
7. **写入观察便签**：记录供阶段 6 诊断使用的线索（见下方「写入观察便签」）

### 写入观察便签

→ 格式与规则见 project-log.md 模板「诊断观察便签」区段（唯一权威源）。

示例：
```markdown
### 阶段3 观察
- [FD] [mainWindow_scan.cpp → onPauseCalc()] 空函数体，函数名暗示"暂停"功能
- [FD] [simtaskmanager.cpp → scheduleTasks()] 弹出任务但不分派
- [DR] [mainWindow.cpp → openRecentProjects()] 调用传参 50，但函数体忽略该参数
```

### "待确认"标记的使用限制

→ 完整定义见 ai-sdd-diagnose SKILL.md 步骤4「待确认标记的使用限制」。

当标注"待确认"时，必须附带：
- 已检查的代码路径（符号锚点列表）
- 为什么代码无法回答这个问题
- 要确认需要什么信息或验证方式

- [ ] **符号锚点核查**：所有代码证据是否使用了符号锚点格式（ClassName::methodName() 或 filename.cpp -> functionName()）？行号仅作为附加缓存，不作为独立证据引用
- [ ] **粒度一致性检查**：每个B-XXX条目是否满足四个强制条件（可观察效果/可识别入口/可独立触发/完成状态可验证）？抽查行为验证粒度一致性：优先选择边界案例——最复杂的（疑似应拆分）、最简单的（疑似应合并）、以及状态为"不完整"或"疑似废弃"的（最容易被误判粒度的）。覆盖全部三种状态（活跃/不完整/疑似废弃）各至少一个
- [ ] **粒度均匀性检查**：是否存在一个行为描述端到端流程而另一个行为描述单个按钮点击？如存在 → 粒度不一致，需要调整
- [ ] **行为-代码映射已维护**：映射表是否已记录所有「不完整」「疑似废弃」的行为及其代码位置？

## 常见错误（反例对照）

### 错误1：行为粒度混用
❌ 错误：B-001 "项目打开"（UI原子操作）与 B-014 "批量谐波扫描"（端到端流程）混在同一表格
✅ 正确：统一粒度——每个行为对应一个可独立触发的、用户可感知的状态变化。
    "批量谐波扫描"应拆为：设置扫描参数 → 启动扫描 → 查看扫描进度 → 查看扫描结果

### 错误2：行为状态与代码事实不符
❌ 错误：onPauseCalc() 函数体为空，但 B-014 标记为"活跃"
✅ 正确：空函数体 + 函数名含动作词 → 状态应为"不完整"，并记录观察便签

## 人类确认后的强制更新步骤

收到人类回复后，在执行分支处理之前，**必须**按顺序完成：

1. **逐条更新行为条目**：遍历所有行为条目——人类标记 `[确认]` 的 -> 确认当前状态标记；标记 `[已废弃]` 的 -> 状态改为"已废弃"；标记 `[状态更正]` 的 -> 按人类指示更新状态；标记 `[需澄清]` 的 -> 追加 `<!-- 待确认 -->` 注释
2. **更新 front matter**：`last_updated`（今天日期）、`last_code_verified`（今天日期）、`status`（`reviewed`）
3. **在 project-log.md 追加记录**：`[日期] 阶段3 经人类确认，N 个行为标记确认、M 个标记废弃、K 个需澄清`

只有完成以上步骤后，才进入阶段 4。

## 人类回复后分支处理

| 人类标记 | 你的操作 |
|---------|---------|
| `[确认]` | 执行上方「强制更新步骤」后继续 |
| `[已废弃]` | 该行为状态 -> `已废弃`；执行上方「强制更新步骤」后继续 |
| `[状态更正]` | 按人类指示更新状态；执行上方「强制更新步骤」后继续 |
| `[需澄清]` | 行为尾部追加 `<!-- 待确认：人类标记需澄清 — YYYY-MM-DD。问题：[人类描述] -->`；执行上方「强制更新步骤」后继续 |
| 人类补充了业务信息 | 追加到对应行为描述；然后执行「强制更新步骤」 |
| 人类要求修改 | 按指示改文档；然后执行「强制更新步骤」 |
| 人类指出了遗漏行为 | 分析遗漏的代码区域，补充该行为条目，重新输出确认清单（跳过强制更新步骤——等下一轮确认） |

## 处理完成后

- 更新 front matter：`last_updated`、`status: reviewed`（如果人类确认了大部分）、`last_code_verified`（今天日期）
- 如果有 `[需澄清]` 的行为，在 project-log.md 的"待确认"列表中追加记录
