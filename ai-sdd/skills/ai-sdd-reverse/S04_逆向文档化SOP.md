# S04 · 逆向文档化 SOP

> 本文是 ai-sdd-reverse 技能的产品说明书。完整操作指令见 SKILL.md，本文提供流程概览和 Prompt 模板。

## 4.1 核心理念

旧工程最大的问题是：**代码是唯一的真相来源。**

逆向文档化就是：让 AI 阅读理解遗留代码，反向生成规范文档。

核心原则：**逆向文档 = 现状还原，不等于理想设计。** 回答"系统当前长什么样"，不是"系统应该长什么样"。

---

## 4.2 七阶段标准流程

```
阶段 0：版本控制探测
═══════════════
检测 Git/SVN/Hg 是否可用，记录探测结果供后续阶段使用。不阻塞流程。

阶段 1：宏观理解
═══════════════
Agent 先自动扫描目录结构和构建文件识别技术栈，再基于扫描结果用 AskUserQuestion 选择题方式向人类确认项目用途、用户画像、痛点和计划。
技术栈能从构建文件确认的就不问人类。
产出：项目概述 → .ai/project-log.md 首条记录
    │
    ▼
阶段 2：架构梳理
═══════════════
识别模块划分和依赖关系，绘制 Mermaid 架构图，扫描架构违规，填充认知边界地图
产出：架构文档 → .ai/doc/00-架构.md
    │ 🔴 人类确认（模块划分、依赖关系、认知边界）
    │
    ▼
阶段 3：行为还原
═══════════════
从 API/UI 入口提取行为清单，标注状态（活跃/不完整/疑似废弃）
产出：行为目录 → .ai/doc/01-行为目录.md
    │ 🔴 人类确认（行为状态准确性）
    │
    ▼
阶段 4：决策记录
═══════════════
技术栈盘点 + 从代码观察决策迹象 + Agent 推断理由（不能留空）
产出：决策记录 → .ai/doc/02-决策记录.md
    │
    ▼
阶段 5：详细设计还原
═══════════════
每个模块深入还原：数据模型/API 契约/核心流程/异常处理/并发安全/性能特征。
综合权重评估 + 深度与权重成正比。
发现问题线索写入观察便签（project-log.md），不直接写入 04。
产出：详细设计文档 → .ai/doc/03-详细设计/<模块中文名>.md
    │ 🔴 人类确认关键设计描述是否准确
    │
    ▼
阶段 6：集中诊断
═══════════════
调用 ai-sdd-diagnose(layer=ALL, depth=L2)，基于完整 00-03 文档和观察便签，
从架构层→需求层→技术层→设计层逐层系统性发现问题，一次性填充 04 文档。
产出：问题与改进 → .ai/doc/04-问题与改进.md
    │ 🔴 人类确认问题清单
    │
    ▼
阶段 7：补充完善
═══════════════
更新 project-log.md，终检所有产出文档，输出完成摘要
```

## 4.3 各阶段 Prompt 模板

### 阶段 1：宏观理解

```
我正在对一个遗留项目进行 AI-SDD 逆向文档化（阶段 1/7：宏观理解）。

请执行：
1. 先自动扫描：目录结构、构建文件、README、包结构——识别技术栈和顶层模块
2. 基于扫描结果，用 AskUserQuestion 向人类逐轮确认（能自己确定的不问）：
   - 技术栈已识别则直接输出结论，不问
   - 项目用途：先给出推断，让人类确认对/不对
   - 目标用户：提供选项（内部工程师/外部客户/两者/不确定）
   - 痛点+计划：合并一问，多选（维护困难/扩展慢/稳定性/文档缺失/想加功能/想重构/暂无计划）
3. 输出项目概述，写入 .ai/project-log.md
```

### 阶段 2：架构梳理

```
请继续逆向文档化阶段 2/7：架构梳理。

分析当前工程的架构：
1. 分析模块间依赖关系（import/include 语句、包结构、调用链）
2. 识别数据流向
3. 识别外部依赖和集成点
4. 用 Mermaid 绘制模块依赖图（所有模块不可省略）
5. 搜索架构违规线索（循环依赖/跨层调用/防腐层绕过/God Class/全局可变状态）→ 写入观察便签
6. 填充认知边界地图（已知已知/已知未知/推断结论）

输出架构文档到 .ai/doc/00-架构.md（基于模板生成）。
问题线索写入 .ai/project-log.md 的「诊断观察便签」区段，由阶段 6 统一处理。

完成后暂停——我需要确认模块划分是否准确。
```

### 阶段 3：行为还原

```
请继续逆向文档化阶段 3/7：行为还原。

从 API/UI 入口提取行为清单：
1. 从 API 路由和 UI 入口遍历所有可观测行为
2. 按业务场景归类为行为组
3. 每个行为标注状态（活跃/不完整/疑似废弃），使用 B-XXX 编号
4. 从数据库 Schema 提取数据域
5. 从配置文件提取外部依赖
6. 版本控制历史辅助分析（如可用）
7. 搜索隐式约束（并发/错误处理/性能/边界 4 个维度）
8. 行为漂移检查 → 线索写入观察便签

输出行为目录到 .ai/doc/01-行为目录.md（基于模板生成）。
问题线索写入 .ai/project-log.md 的「诊断观察便签」区段，由阶段 6 统一处理。

完成后暂停——我需要确认行为状态是否准确。请对每个行为标注状态。
```

### 阶段 4：决策记录

```
请继续逆向文档化阶段 4/7：决策记录。

从代码中提取技术决策证据：
1. 技术栈盘点：所有关键依赖的名称和版本
2. 版本漂移评估（当前版本 vs 最新稳定版）
3. 从代码观察决策迹象（架构级/设计级），使用 D-XXX 编号
4. 对每个观察到的决策，Agent 必须推断理由（状态标记为 Inferred），不能留空
5. 验证构建/测试/部署的实际配置路径
6. 识别版本风险 → 线索写入观察便签

输出决策记录到 .ai/doc/02-决策记录.md（基于模板生成）。
问题线索写入 .ai/project-log.md 的「诊断观察便签」区段，由阶段 6 统一处理。

注意：这是现状陈述 + 推断理由。直接可验证的信息标记"已确认"，Agent 推断的理由标记"推断"。
```

### 阶段 5：详细设计还原

```
请继续逆向文档化阶段 5/7：详细设计还原。

对每个模块深入还原：
1. 按综合权重确定模块深度，深度与权重成正比
2. God Class 必须单独成文
3. 模块名不能包含"与/和/及"等连接词，每个模块独立命名
4. 使用符号锚点标注关键设计元素的代码位置

每个模块一个文件，文件名使用中文。全部 8 个章节必须出现：
1. 概述：核心职责 + 负向限制
2. 核心数据模型：穷尽列出字段
3. 对外接口契约：每个 API 的完整契约（含副作用和幂等性）
4. 核心业务流程：如有 → Mermaid 序列图（使用实际函数名）。如无 → 标准措辞
5. 异常处理与容错：如有 → 异常处理表。如无 → 标准措辞
6. 并发安全与一致性：如有 → 风险点表。如无 → 标准措辞。**必须检查跨线程交互**
7. 性能特征与资源消耗：如有 → 复杂度 + 资源密集点。如无 → 标准措辞
8. 已知问题与陷阱 → 写入观察便签（project-log.md），不直接写入 04

章节一个不落，如实描述。严格按推荐顺序编写：异常/并发 → 流程/状态机 → 接口/数据 → 概述。
多组件模块（N≥2）必须为每个组件提供独立的子节。

输出到 .ai/doc/03-详细设计/<模块中文名>.md。

完成后暂停——我需要确认关键设计描述是否准确。
```

### 阶段 6：集中诊断

```
请完成逆向文档化阶段 6/7：集中诊断。

调用 ai-sdd-diagnose(layer=ALL, depth=L2)：
1. 读取 .ai/doc/00-03 全部文档（物化的系统理解）
2. 读取 .ai/project-log.md 的「诊断观察便签」区段（阶段2-5 的问题线索）
3. 执行全部 9 个范畴的搜索清单
4. 从架构层→需求层→技术层→设计层逐层梳理问题
5. 关联跨层级的根因和表现
6. 全局标准化严重程度
7. 去重、合并
8. 写入 .ai/doc/04-问题与改进.md
9. 删除观察便签区段

完成后暂停——我需要确认问题清单是否准确、是否有遗漏。
```

### 阶段 7：补充完善

```
请完成逆向文档化阶段 7/7：补充完善。

1. 更新 .ai/project-log.md（变更时间线 + 已知问题 + 待确认列表）
2. 终检所有产出文档（补全遗漏、修复 front matter、移除占位文本）
3. 输出完成摘要

完成后，项目即具备完整规范文档，后续改造可基于文档进行。
```

## 4.4 与 .ai/doc/ 结构的集成

逆向文档化的产出天然对应 AI-SDD 的 `.ai/doc/` 目录结构：

| 逆向阶段 | 产出 | 路径 |
|---------|------|------|
| 阶段 2 | 架构文档 | `.ai/doc/00-架构.md` |
| 阶段 3 | 行为目录 | `.ai/doc/01-行为目录.md` |
| 阶段 4 | 决策记录 | `.ai/doc/02-决策记录.md` |
| 阶段 5 | 详细设计 | `.ai/doc/03-详细设计/<模块中文名>.md` |
| 阶段 6 | 问题诊断 | `.ai/doc/04-问题与改进.md` |
| 阶段 7 | 状态/决策 | `.ai/project-log.md`、`.ai/changes/` |

## 4.5 注意事项

### 人类确认频率

| 阶段 | 是否确认 | 确认内容 |
|------|---------|---------|
| 阶段 0 | 否 | — |
| 阶段 1 | 否 | — |
| 阶段 2 | 是 | 模块划分、依赖关系、认知边界 |
| 阶段 3 | 是（最重要） | 行为状态准确性 |
| 阶段 4 | 是（可跳过） | 推断理由可靠性 |
| 阶段 5 | 是 | 关键设计描述是否准确 |
| 阶段 6 | 是 | 问题清单是否准确、是否有遗漏 |
| 阶段 7 | 否 | — |

### 分阶段执行

大型遗留系统按子任务拆分，每个子任务独立会话：

- 子任务 1：阶段 0-2（探测 + 宏观 + 架构）
- 子任务 2：阶段 3（行为还原）
- 子任务 3：阶段 4-5（决策记录 + 详细设计）
- 子任务 4：阶段 6（集中诊断 → 04-问题与改进.md）
- 子任务 5：阶段 7（补充完善）

确认点是子任务之间的自然边界。

### 何时值得做

未来 3 个月内预计 ≥3 次实质性改动 → 值得做完整逆向化。
维护模式 → 成本大于收益，改用一次性探索分析。
