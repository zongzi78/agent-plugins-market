# S04 · 逆向文档化 SOP

> 本文是 ai-sdd-reverse 技能的产品说明书。完整操作指令见 SKILL.md，本文提供流程概览和 Prompt 模板。

## 4.1 核心理念

旧工程最大的问题是：**代码是唯一的真相来源。**

逆向文档化就是：让 AI 阅读理解遗留代码，反向生成规范文档。

核心原则：**逆向文档 = 现状还原，不等于理想设计。** 回答"系统当前长什么样"，不是"系统应该长什么样"。

---

## 4.2 七阶段标准流程

```
阶段 0：版本控制探测
═══════════════
检测 Git/SVN/Hg 是否可用，记录探测结果供后续阶段使用。不阻塞流程。

阶段 1：宏观理解
═══════════════
向人类收集 5 个核心信息 + 扫描目录结构 + 识别技术栈
产出：项目概述 → .ai/project-log.md 首条记录
    │
    ▼
阶段 2：架构梳理
═══════════════
识别模块划分和依赖关系，绘制 Mermaid 架构图，扫描技术债务，填充认知边界地图
产出：架构文档 → .ai/doc/00-架构.md
    │ 🔴 人类确认（模块划分、依赖关系、认知边界）
    │
    ▼
阶段 3：需求还原
═══════════════
从 API 路由/数据库 Schema/配置文件反推功能列表，标注生命周期和可信度，搜索隐式约束
产出：需求文档 → .ai/doc/01-需求.md
    │ 🔴 人类确认（每个功能标记 确认/废弃/需澄清）
    │
    ▼
阶段 4：技术栈确认
═══════════════
分析版本和关键依赖，版本漂移评估，使用模式分析，列出缺失决策理由
产出：技术选型报告 → .ai/doc/02-技术选型.md
    │
    ▼
阶段 5：详细设计还原
═══════════════
每个模块深入还原：数据模型/API 契约/核心流程/异常处理/并发安全/性能特征
产出：详细设计文档 → .ai/doc/03-详细设计/<模块中文名>.md
    │ 🔴 人类确认关键设计描述是否准确
    │
    ▼
阶段 6：补充完善
═══════════════
更新 project-log.md，终检所有产出文档，输出完成摘要
```

## 4.3 各阶段 Prompt 模板

### 阶段 1：宏观理解

```
我正在对一个遗留项目进行 AI-SDD 逆向文档化（阶段 1/7：宏观理解）。

请执行：
- 扫描项目目录结构，了解顶层模块划分
- 从构建配置文件（package.json/pom.xml/go.mod 等）识别技术栈
- 输出项目概述，写入 .ai/project-log.md 的首条变更记录
- 不需要做更多——后续阶段会逐步推进
```

### 阶段 2：架构梳理

```
请继续逆向文档化阶段 2/7：架构梳理。

分析当前工程的架构：
1. 分析模块间依赖关系（import/include 语句、包结构、调用链）
2. 识别数据流向
3. 识别外部依赖和集成点
4. 用 Mermaid 绘制模块依赖图（所有模块不可省略）
5. 搜索技术债务（TODO/FIXME/HACK/废弃 API/硬编码等）
6. 搜索架构约束违反（循环依赖/跨层调用/防腐层绕过）
7. 填充认知边界地图（已知已知/已知未知/推断结论）

输出架构文档到 .ai/doc/00-架构.md（基于模板生成）。

完成后暂停——我需要确认模块划分是否准确。
```

### 阶段 3：需求还原

```
请继续逆向文档化阶段 3/7：需求还原。

从代码中反推功能列表：
1. 从 API 路由提取对外功能，按业务场景归类为功能组
2. 从数据库 Schema 提取数据域
3. 从配置文件提取外部依赖
4. 版本控制历史辅助分析（如可用）
5. 每个功能标注生命周期（活跃/疑似废弃/已废弃）和可信度（已确认/推断/推测）
6. 搜索隐式约束（并发/错误处理/性能/边界 4 个维度）
7. 每个功能写验收标准（正常路径 ≥1 + 错误路径 ≥1）
8. 需求漂移检查

输出需求文档到 .ai/doc/01-需求.md（基于模板生成）。

完成后暂停——我需要确认哪些功能还在用、哪些已废弃。请对每个功能标记 [确认]/[废弃]/[需澄清]。
```

### 阶段 4：技术栈确认

```
请继续逆向文档化阶段 4/7：技术栈确认。

分析技术栈现状：
1. 分析所有关键依赖的名称和版本
2. 版本漂移评估（当前版本 vs 最新稳定版）
3. 标注每个依赖的使用模式（标准用法/非标准用法）
4. 验证构建/测试/部署的实际配置路径
5. 识别技术债务
6. 提取"缺失的决策理由"列表

输出技术选型报告到 .ai/doc/02-技术选型.md（基于模板生成）。

注意：这是现状陈述，不猜测选型动机。直接可验证的信息标记"已确认"，推断性评估标记"推断"。
```

### 阶段 5：详细设计还原

```
请继续逆向文档化阶段 5/7：详细设计还原。

对每个模块深入还原。每个模块一个文件，文件名使用中文。包含：
1. 概述：核心职责 + 负向限制
2. 核心数据模型：穷尽列出字段，标注可信度
3. 对外接口契约：每个 API 的完整契约（含副作用和幂等性）
4. 核心业务流程：Mermaid 序列图（使用实际函数名）
5. 异常处理与容错（必填）
6. 并发安全与一致性（必填）
7. 性能特征与资源消耗（必填）
8. 已知问题与陷阱

深度自适应：复杂模块自然长，简单模块自然短。

输出到 .ai/doc/03-详细设计/<模块中文名>.md。

完成后暂停——我需要确认关键设计描述是否准确。
```

### 阶段 6：补充完善

```
请完成逆向文档化阶段 6/7：补充完善。

1. 更新 .ai/project-log.md（变更时间线 + 已知问题 + 待确认列表）
2. 终检所有产出文档（补全遗漏、修复 front matter、移除占位文本）
3. 输出完成摘要

完成后，项目即具备完整规范文档，后续改造可基于文档进行。
```

## 4.4 与 .ai/doc/ 结构的集成

逆向文档化的产出天然对应 AI-SDD 的 `.ai/doc/` 目录结构：

| 逆向阶段 | 产出 | 路径 |
|---------|------|------|
| 阶段 2 | 架构文档 | `.ai/doc/00-架构.md` |
| 阶段 3 | 需求文档 | `.ai/doc/01-需求.md` |
| 阶段 4 | 技术选型 | `.ai/doc/02-技术选型.md` |
| 阶段 5 | 详细设计 | `.ai/doc/03-详细设计/<模块中文名>.md` |
| 阶段 6 | 状态/决策 | `.ai/project-log.md`、`.ai/changes/` |

## 4.5 注意事项

### 人类确认频率

| 阶段 | 是否确认 | 确认内容 |
|------|---------|---------|
| 阶段 0 | 否 | — |
| 阶段 1 | 否 | — |
| 阶段 2 | 是 | 模块划分、依赖关系、认知边界 |
| 阶段 3 | 是（最重要） | 每个功能标记 确认/废弃/需澄清 |
| 阶段 4 | 否 | 直接可验证的事实不需要确认 |
| 阶段 5 | 是 | 关键设计描述是否准确 |
| 阶段 6 | 否 | — |

### 分阶段执行

大型遗留系统按子任务拆分，每个子任务独立会话：

- 子任务 1：阶段 0-2（探测 + 宏观 + 架构）
- 子任务 2：阶段 3（需求还原）
- 子任务 3：阶段 4-5（技术栈 + 详细设计）
- 子任务 4：阶段 6（补充完善）

确认点是子任务之间的自然边界。

### 何时值得做

未来 3 个月内预计 ≥3 次实质性改动 → 值得做完整逆向化。
维护模式 → 成本大于收益，改用一次性探索分析。
