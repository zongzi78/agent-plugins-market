# AI-SDD 漂移检测

## 定位

只读操作，不修改任何文件。检测 `.ai/doc/` 中的规范文档与实际代码之间的一致性。

## 前置检查

1. 检查 `.ai/` 是否存在 → 如不存在，提示先运行 `/ai-sdd-init`
2. 检查 `.ai/doc/` 下是否有 doc 文件 → 如为空，提示无文档可检查
3. 检查 `.ai/doc/` 下 04-问题与改进.md 是否存在 → 如不存在，提示"建议运行 /ai-sdd-reverse 生成或手动创建"

## 执行流程

### 步骤 1：读取所有规范文档

扫描 `.ai/doc/` 下的所有文件，记录：
- 模块列表（从 `00-架构.md`）
- 功能列表（从 `01-行为目录.md`）
- API 契约（从 `03-详细设计/*.md`）
- 文档结构（是否符合角色定义）

### 步骤 2：读取对应代码

根据 doc 文件中描述的模块和功能，读取对应的源代码。

### 步骤 3：逐文件对比

按以下维度检测：

**维度 1：架构漂移**
- `00-架构.md` 中的模块列表 vs 代码中的实际模块目录
- 检测：模块新增、删除、重命名

**维度 2：行为漂移**
- `01-行为目录.md` 中的功能列表 vs 代码中的 API 路由/UI 组件
- 检测：功能新增、功能废弃

**维度 3：设计漂移**
- `03-详细设计/<模块名>.md` 中的 API 契约 vs 代码中的实际实现
- 检测：参数变更、返回值变更、错误码变更

**维度 4：结构漂移**
- doc 文件的组织方式是否符合角色定义
- 检测：追加式更新（内容堆在末尾）、内容混乱（实现细节混入需求文档）、大量原始代码

**维度 5：文档角色一致性检查**

对每份 doc 文档，读取其顶部的「📋 本文档角色」区块中的 **不应该写** 列表。
逐项执行对应的信号检测：

| 文档 | 不应出现的信号 | 检测方法 |
|------|--------------|---------|
| 00-架构.md | API 契约细节 | grep 表格头含 `\| 参数 \| 类型 \| 必填` 或 `\| 返回值 \|` |
| 00-架构.md | 架构违规条目 | grep `AV-\d{3}` 编号模式 |
| 00-架构.md | 推测性措辞 | grep `(可能是\|笔者推测\|Agent推断)` |
| 00-架构.md | 模块内部实现混入数据流 | 检查数据流步骤是否描述单个类/函数的内部操作（非跨模块交互） |
| 01-行为目录.md | 实现细节 | grep `(mutex\|Mutex\|QHash\|std::map\|std::vector\|智能指针\|RAII)` |
| 01-行为目录.md | 数据域定义 | grep 含字段定义表格（`\| 字段 \| 类型 \| 约束`） |
| 01-行为目录.md | 需求漂移判断 | grep `(应该废弃\|不再需要\|已过时)` |
| 02-决策记录.md | 版本风险判断 | grep `(EOL\|不再维护\|安全漏洞\|CVE)` — VR 应归入04 |
| 02-决策记录.md | API 细节 | grep 含 `\| 参数 \| 类型 \| 必填` 的接口表格 |
| 02-决策记录.md | 问题后果描述 | grep `(会导致\|可能造成\|影响范围)` — 属于04 |
| 03-详细设计/*.md | 需求动机 | grep `(用户需要\|业务需求\|产品要求\|为了满足)` |
| 03-详细设计/*.md | 全局架构判断 | grep `(应该拆分为\|应该合并\|架构不合理)` |
| 03-详细设计/*.md | 模块陷阱条目 | grep `MT-\d{3}` 编号 — MT 归入04 |
| 04-问题与改进.md | 设计理性 | grep `(设计意图\|权衡\|选择.*因为\|之所以)` — 归入02 |
| 04-问题与改进.md | 需求描述段落 | 检查是否有长篇功能描述（非问题描述格式） |

每项检测到信号 → 标记为潜在越界，列出具体位置和匹配内容。
建议对标记结果进行人工复核后决定是否需要拆分。

**维度 6：交叉引用完整性检查**
- 检查 `02-决策记录.md` 中的决策条目是否在 `04-问题与改进.md` 中有对应的交叉引用（如决策导致了已知问题）
- 检查 `04-问题与改进.md` 中的条目是否引用了 `02-决策记录.md` 中对应的决策
- 检测：单向引用（A 引用 B 但 B 未回引）、孤立引用（引用目标不存在）、断链

**维度 7：内容完备性检查**（仅对 `source: reverse-engineering` 的文档执行）

对照 ai-sdd-reverse SKILL.md 中的"各文档产出必备内容"清单逐项检查每份文档。检查项以 SKILL.md 中的定义为唯一权威，此处不重复维护。

输出等级：
- ✅ 完备 — 所有要素覆盖
- ⚠️ 偏浅 — 缺 1-2 项，列出具体缺失项
- ❌ 不足 — 缺 3 项以上，列出具体缺失项，建议运行 `/ai-sdd-reverse` 补充

**维度 8：符号锚点自愈**

对每条代码证据的符号锚点执行自愈流程：
1. 提取符号锚点（类名、方法名、函数签名等）
2. grep 代码库确认符号是否存在
3. 有效 → grep -n 自动更新文档中的行号缓存
4. 失效 → 标记为死引用 ⚠️，记录到 project-log 待人工复查

### 步骤 4：生成报告

```
=== AI-SDD 健康检查报告 ===

[文件路径]
  [状态] [描述]（失效类型：[类型]）

状态说明：
  ✅ 一致     — 文档与代码匹配
  ⚠️ 轻微漂移 — 小的过时（如缺少新增参数）
  ❌ 严重漂移 — 文档与代码根本对不上
  📭 缺失     — 代码中有模块但没有对应 doc
  🗂️ 结构问题 — 追加式更新、内容混乱、角色不符
  💀 死引用   — 符号锚点失效，文档引用的代码符号不存在

失效类型（对应 S06 失效模式）：
  文档腐烂 — 内容与事实不符
  文档质量退化 — 内容正确但组织混乱
  规范过度设计 — 文档比实现还重
  上下文泄漏 — 实现偏离了最终设计
  符号锚点失效 — 文档引用的代码符号已不存在

覆盖范围说明：
  本 skill 覆盖 S06 七种失效模式中可自动检测的 4 种（上表）。
  另外 3 种属于人类行为模式，需要开发者自评：
  - 模式僵化 → 每季度反思"流程本身是否需要调整"
  - AI 盲从过时文档 → 通过"行为漂移"维度间接发现
  - 人类橡皮图章化 → 每 5-10 个任务做一次深度审查

总结：X 个一致，Y 个轻微漂移，Z 个严重漂移，W 个缺失，V 个结构问题，D 个内容偏浅/不足，S 个符号锚点失效

[如有漂移] 建议：运行 /ai-sdd-rebuild 修复（自动调用 reverse ref 模式从代码重建）
[如有内容深度不足] 建议：运行 /ai-sdd-reverse 对偏浅模块重新逆向
[如项目 supplement-rules.md 版本过旧] 建议：运行 /ai-sdd-update 检查是否需要升级文档体系
[如有符号锚点失效] 建议：查看 project-log 中的死引用记录，人工复查后更新文档
```

### 三级审计模式

用户可通过触发词选择审计深度：

| 模式 | 触发词（举例） | 检查范围 |
|------|--------------|---------|
| **快速检查** | "快速检查文档" | 仅检查架构漂移（维度 1），5 分钟 |
| **标准检查**（默认） | "检查文档漂移" | 全部维度 |
| **深度审计** | "深度审计文档" | 全部维度 + 结构漂移逐文件详细分析 + 生成修复建议清单 + 检测 supplement-rules.md 版本是否与当前 AI-SDD 版本一致 |

## 保鲜检查操作（供功能完成后的例行保鲜使用）

1. **重新读取**相关 doc 文件全文，理解其结构和当前内容
2. **对照代码**实际行为逐条核对文档描述是否仍然准确
3. **记录差异**：将不一致处写入 `.ai/project-log.md` 的「待同步文档」区段
4. 发现文档与代码不符时：立即标记 → 报告人类 → 不要假装文档是对的

## 注意事项

- 本 skill 不修改任何文件（符号锚点行号更新除外——仅更新行号缓存，不改变内容）
- 对于"轻微漂移"，可以在报告中简要说明修复建议
- 对于"严重漂移"和"结构问题"，建议用户运行 `/ai-sdd-rebuild`
