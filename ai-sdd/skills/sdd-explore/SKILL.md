# sdd-explore（探索 + 规划）

## 定位

深度探索代码 + 思维风暴，产出详尽的 plan.md。这是整个工作流中最重的一步。

## 前置检查（Hard Gate）

1. `.ai/` 不存在 → 提示"请先执行 `/ai-sdd-init`"，终止
2. 无活跃 change → 拒绝，提示"请先通过 `/sdd-propose` 创建一个 change"，终止
3. 多个活跃 change → 让用户选择目标 change
4. plan.md 非空（已有内容）→ 警告"plan.md 已有内容"，AskUserQuestion：覆盖 / 从当前状态继续 / 取消

## 执行流程

### 阶段 1：全面上下文收集

聚焦 `.ai/` 体系内的所有文档，按依赖顺序读取：

| 序号 | 文件 | 用途 | 缺失处理 |
|------|------|------|----------|
| 1 | `.ai/changes/<target>/proposal.md` | 探索起点 | 不应发生，中止 |
| 2 | `.ai/project-log.md` | 历史脉络、已知问题、待同步文档、技术债务 | 警告，以空上下文继续 |
| 3 | `.ai/doc/00-架构.md` | 模块划分、依赖关系、数据流 | 警告，标记风险 |
| 4 | `.ai/doc/01-需求.md` | 功能需求（F-XXX）、验收标准 | 警告，标记风险 |
| 5 | `.ai/doc/02-技术选型.md` | 技术栈、依赖、构建/测试/部署 | 警告，标记风险 |
| 6 | `.ai/doc/03-详细设计/*.md`（glob 所有文件） | 各模块详细设计、API 契约、数据模型 | 警告，标记风险 |

**上下文过大时的降级策略**：
- 必读：proposal + project-log + 00-架构.md + 01-需求.md
- 按需读取 03-详细设计/：仅读取 proposal "影响范围分析" 涉及的模块对应文件
- 02-技术选型.md：仅读"技术栈总览"和"关键依赖"章节

### 阶段 2：项目理解构建

在读完所有文档后，不急于进入方案设计，先构建系统性项目理解。

**步骤 1：综合理解**

从读取的文档中提取 6 个维度的信息（内部工作上下文，不写入文件）：

| 维度 | 提取内容 | 来源 |
|------|----------|------|
| 系统架构 | 模块列表、依赖图、数据流 | 00-架构.md |
| 现有需求 | F-XXX 功能、状态、验收标准 | 01-需求.md |
| 技术约束 | 语言/框架/DB/构建工具/关键依赖 | 02-技术选型.md |
| 模块设计 | 数据模型、API 契约、业务流程 | 03-详细设计/*.md |
| 项目历史 | 变更时间线、决策记录 | project-log.md |
| 已知问题 | 技术债务、待同步文档 | project-log.md |

**步骤 2：文档内部一致性检查**

检查文档之间的一致性（代码比对属于 `/ai-sdd-check` 职责）：

| 检查项 | 比较对象 | 发现不一致时 |
|--------|----------|-------------|
| 需求-架构对齐 | 01-需求.md 涉及的模块 vs 00-架构.md 模块列表 | 标记，警告用户 |
| 设计-架构对齐 | 03-详细设计/ 的模块 vs 00-架构.md 依赖关系 | 标记，警告用户 |
| 待同步文档 | project-log.md「待同步文档」区段 | 提示用户注意 |
| 技术债务 | project-log.md「已知问题」区段 | 纳入方案考量 |

检查发现**不是阻断项**，而是上下文增强。向用户报告：
```
📊 上下文收集发现以下需要注意的事项：
- [warning 列表]
这些已纳入方案设计的考量范围。
```

**步骤 3：置信度感知读取**

在读取 `.ai/doc/` 构建项目理解时，同步解析文档中的 confidence 标记：

- 对每份文档（尤其是 `mode: reverse` 的文档），统计 CONFIRMED / INFERRED / SPECULATIVE 三类标记的分布
- 不同置信度的内容在后续使用中的处理方式：
  - CONFIRMED → 可直接作为方案设计依据
  - INFERRED → 可参考，但在方案中需注明"基于代码推断，需验证"
  - SPECULATIVE → 不可直接依赖，在方案中作为风险标注，建议编码前人工确认

**步骤 4：影响范围映射**

基于 proposal 的"影响范围分析" + 架构依赖图：
1. 识别直接受影响的模块
2. 追踪一层传递依赖（被影响模块的上下游）
3. 标注涉及的详细设计文档、需求编号、技术债务
4. 输出影响范围映射供阶段 3 使用：

```
📍 影响范围映射：
- 直接影响：[module A, module B]
- 间接影响（依赖链）：[module C]
- 涉及需求：[F-001, F-003]
- 涉及设计文档：[auth.md, user.md]
- 涉及技术债务：[TD-001]
📊 置信度分布：
- CONFIRMED 依赖：[auth.md §API契约, user.md §数据模型]
- INFERRED 依赖：[payment.md §并发策略, F-003 §验收标准]
- SPECULATIVE 依赖：[auth模块的幂等性设计]
- 建议：在编码前人工确认 SPECULATIVE 项，方案中标注需验证的 INFERRED 假设
```

### 阶段 3：思维风暴

按 change 实际复杂度灵活调整：

| 复杂度 | 澄清问题数 | 候选方案数 | 设计展示 |
|--------|-----------|-----------|----------|
| 中等 | 2-3 | 2 | 逐节 |
| 复杂 | 3-5 | 2-3 | 逐节确认 |

1. **数据驱动的范围确认**：展示阶段 2 的影响范围映射，让人类确认范围（替代主观判断的"评估范围"）
2. **逐步澄清**：基于 proposal，逐步提出澄清问题（一次一个问题，多选优先）
3. **探索代码**：用结构图展示架构关系（当前结构 + 变更后结构）
4. **约束感知的候选方案**：提出候选方案 + 优劣矩阵 + **明确推荐**，方案中显式引用发现的约束：
   - "根据 02-技术选型.md，项目使用 [X]，因此..."
   - "01-需求.md 的 F-003 要求 [Y]，所以方案必须..."
   - "03-详细设计/auth.md 定义了当前认证流程 [Z]，集成点在..."
   - "project-log.md 的技术债务 TD-001 指出 [W]，本方案应避免加重..."
5. **逐节确认**：逐节展示设计，每节获得人类确认
6. **Hard Gate**：方案必须经人类批准才能继续
7. **设计自检**：占位符扫描、内部一致性、范围检查、歧义检查

### 阶段 4：产出 plan.md

1. 将确认后的方案写入 change 的 plan.md
2. **No Placeholders Rule**：禁止 TBD、TODO、"类似任务N"、模糊指令
3. 每个任务必须包含的内容：
   - ✅ 完整文件路径、函数签名、数据结构定义
   - ✅ 伪代码级算法描述、精确的 CLI 命令、预期输出
   - ❌ 不必有：生产级完整实现代码（那是 `/sdd-apply` 的事）
4. **必须包含四个章节**：

   **`## 风险与注意事项`**：
   ```markdown
   ## 风险与注意事项

   | 风险 | 来源 | 缓解措施 |
   |------|------|----------|
   | [风险描述] | [文档一致性检查发现/技术债务编号] | [缓解措施] |
   ```

   **`## 文档变更`**（为 `/sdd-sync` 提供结构化输入）：
   ```markdown
   ## 文档变更

   | 操作 | 目标文件 | 章节 | 变更摘要 |
   |------|----------|------|----------|
   | ADDED | .ai/doc/01-需求.md | F-005 | 新增导出CSV功能需求 |
   | MODIFIED | .ai/doc/03-详细设计/auth.md | §API契约 | 新增 /api/export 端点 |
   ```

   **`## 架构决策`**（决策化石）：
   ```markdown
   ### 决策：[决策名称]
   - **背景**：[为什么需要做这个决策]
   - **方案对比**：A) ... B) ...
   - **选择**：[方案X]，理由：[...]
   - **放弃的方案**：[方案Y]，原因：[...]
   - **影响范围**：[哪些模块/文件受影响]
   ```

   **`## 备选方案（已否决）`**（探索过程存档）：
   - 记录被否决方案及其否决原因，设计讨论的产物不丢失

5. **覆盖性自检**（写入 plan.md 前的强制检查门）：
   - [ ] proposal.md 的所有验收标准在任务清单中有对应实现步骤
   - [ ] 影响范围映射中的所有模块在任务清单中被覆盖
   - [ ] 01-需求.md 中受影响的 F-XXX 在文档变更表中有记录
   - [ ] 03-详细设计/ 中受影响的模块在文档变更表中有记录
   - [ ] 00-架构.md 如有新增模块/依赖变更，在文档变更表中有记录
   - [ ] project-log.md 中相关的技术债务已在方案中考虑或标注"不在本次范围"
   - [ ] 所有占位符扫描通过（无 TBD/TODO/模糊指令）
   - [ ] 内部一致性检查通过（任务间无矛盾依赖）
6. 提交人类审阅

### 阶段 5：更新状态

1. 更新 proposal.md 的 status 为 `planned`
2. 更新 project-log.md
