# AI-SDD 行为准则

## 启动时读取的文件（按优先级）

1. `.ai/project-log.md` — 项目状态和变更时间线
2. **本文件（.ai/supplement-rules.md）** — 行为准则和约束
3. `.ai/doc/00-架构.md` — 系统架构
4. `.ai/doc/01-行为目录.md` — 可观察行为清单（B-XXX）
5. `.ai/doc/02-决策记录.md` — 技术栈事实与决策理由（D-XXX）
6. `.ai/doc/03-详细设计/<模块名>.md` — 各模块详细设计
7. `.ai/doc/04-问题与改进.md` — 当前有效问题汇总
8. 活跃 change 的 plan.md — 当前任务的实施方案
9. `.ai/project-rules.md`（如存在）— 项目私有规则

# 一、权限边界

## 1.1 你能自主决定 / 必须确认 / 无权决定

| 自主决定 | 必须征求人类确认 | 无权决定 |
|---------|-----------------|----------|
| 代码风格、变量命名 | 技术方案方向 | 需求范围 |
| 具体算法选择（在规范框架内） | 引入新技术依赖 | 架构设计 |
| 测试用例编写 | API 对外契约变更 | 产品方向 |
| 文档格式和结构 | 设计文档的重大变更 | 功能取舍 |
| 实现细节（在规范范围内） | 遇到规范矛盾时的处理方案 | 发布/部署决策 |

## 1.2 决策点分类

| 标记 | 含义 | 你的行为 |
|------|------|----------|
| Hard Gate | 人类必须决策 | **STOP**，等待确认 |
| Soft Gate | 提出问题，人类选方向 | 提 2-3 个选项 + 推荐 |
| AI Autonomy | 自主决定 | 在规范框架内自行决定 |

## 1.3 五条不变量

| # | 原则 | 含义 |
|---|------|------|
| 1 | **意图归人类，实现归机器** | 人类定 What & Why，你定 How |
| 2 | **规范是唯一真相来源** | `.ai/doc/` 是决策权威。代码与文档冲突时，检查 `last_code_verified` 判断漂移方向 |
| 3 | **可逆的决策，不可逆的记录** | 决策可以推翻，记录不能抹去 |
| 4 | **会话隔离是上下文卫生** | 每次执行从干净的上下文开始，状态通过文件传递 |
| 5 | **文档与代码双向闭环** | 文档驱动代码，代码验证文档。偏离时必须主动提出 |

# 二、强制暂停规则

## 2.1 回滚契约

规范文档是你与人类的"合同"。发现合同问题，不能自行偏离：

1. 暂停实现
2. 明确指出规范中的矛盾或不可行之处（引用具体章节）
3. 提出 2-3 个可选方案 + 推荐
4. 等待人类裁决
5. 人类裁决后，**先更新规范文档，再继续实现**

## 2.2 强制暂停场景

以下任一场景发生 → 立即停止，报告人类，不自行裁决：

1. **需求歧义**：需求文档描述有 ≥2 种合理理解
2. **设计不可行**：设计方案不可行或有严重缺陷
3. **无决策标准**：需要做技术取舍但无明确决策标准
4. **意图不明**：需要改动你不理解其原始设计意图的模块
5. **文档代码不一致**：代码实际行为与 `.ai/doc/` 中的描述不符

# 三、代码注释规则

- **解释 Why，而非 What**：注释解释设计意图、边界条件、性能考量、已知限制
- **必写注释的场景**：
  - 非显而易见的设计或算法选择
  - 边界条件处理（空值/溢出/并发）
  - 性能敏感代码的优化理由
  - 已知限制或 TODO
  - 公共 API / 导出函数 / 模块入口
- **禁止写在注释中**：
  - 文档文件路径引用（`.ai/doc/`、`.ai/changes/`）— 由 plan.md 管理
  - 需求编号（F-XXX）— 正向溯源由 plan.md「涉及文件」表承载
  - 与代码逻辑重复的冗余描述

# 四、文档操作规则

## 4.1 操作红线

以下行为**禁止**：

1. **禁止追加式更新**：不得在文档末尾追加内容而不融入既有结构
2. **禁止越界写入**：不得在 00~03 中写入应属于 04 的内容（架构违规、需求漂移、功能缺陷、版本与依赖风险、模块陷阱、设计债务、代码债务）
3. **禁止隐瞒冲突**：文档与代码不符时必须立即标记 → 报告人类 → 不假装文档是对的
4. **禁止忽略已知问题**：explore 和 apply 开始前必须读取 04，确认已知问题与本次变更的关系
5. **禁止保留已解决问题**：04 是当前有效问题快照——解决了就删，不保留历史
6. **禁止遗漏取舍记录**：explore 阶段做有代价的取舍（选 A 弃 B），必须产出条目写入 04

## 4.2 文档创建原则

更新文档时当作首次编写。通读全文，确保新增内容融入既有结构，而非在末尾追加。

## 4.3 文档与代码冲突

代码与文档冲突时：
1. 检查 `last_code_verified` 时间戳——如果远早于 `last_updated`，文档可能已漂移
2. 运行 `/ai-sdd-check` 诊断
3. 或运行 `/ai-sdd-rebuild` 从代码重建

## 4.4 文档角色边界

每份 `.ai/doc/` 文档有明确的角色边界。完整定义见各文档顶部的「📋 本文档角色」区块。写入前检查：内容是否属于该文档角色区块中"应该写"的范围？不属于 → 归入正确的文档。

# 五、会话与启动规则

## 5.1 会话即事务

- 会话有且只有唯一的明确产出目标
- 开始时明确 → 过程中不偏离 → 结束时提交或回滚
- 跨任务状态通过文件传递，不通过聊天历史

# 项目特定约束

> 本节记录**本项目特有的**约束规则。这些规则无法从代码自动推导——需要人工显式声明。
>
> **约束分类**：
> | 类别 | 编号前缀 | 约束对象 | 检测方式 |
> |------|---------|---------|---------|
> | 架构约束 | AC-XXX | 模块边界、分层、数据所有权 | sdd-apply reviewer 上下文判断 + diagnose 扫描 |
> | 设计约束 | DP-XXX | 设计模式、接口契约、继承规则 | sdd-apply reviewer 上下文判断 |
> | 代码约束 | CI-XXX | 命名规范、禁止语法、资源管理 | sdd-apply reviewer 自动 grep |
> | 环境约束 | ENV-XXX | 工具链、命令规范、运行方式 | sdd-apply 注入 implementer system prompt + reviewer 审查工具调用记录 |
>
> **填写指引**：
> - 约束用"必须"或"禁止"，不用"应该""建议"
> - 适用域写文件 glob 或模块名（如 `*.h, *.cpp` 或 `src/db/`），无明确范围写"全局"
> - 阻塞=是 → 违反则 reviewer 直接拒绝（不计入审查循环次数）；阻塞=否 → 标记警告但放行
> - 启用=否 → 所有 downstream skill 跳过该条约束（临时禁用，无需删除）
> - 建议总数控制在 20 条以内
> - **不要写能从代码自动推导的规则**（如"项目用 Qt 5.15"——从 CMakeLists.txt 就能读到）
>
> **与 downstream skill 的关系**：
> - `sdd-apply`：reviewer 逐条检查 CI 约束；implementer 加载 ENV 约束到 system prompt
> - `ai-sdd-diagnose`：将 AC/DP/CI 约束作为判定标准，违反则写入 04-问题与改进.md（引用约束编号）
> - `sdd-explore`：设计方案前读取全部约束，plan.md 中写合规声明
> - `ai-sdd-check`：检查约束适用域是否引用已不存在的模块/文件
> - `sdd-sync`：变更完成后检查是否引入新约束模式或使现有约束过时
> - `ai-sdd-reverse`：阶段 2-5 扫描代码时标注 [XX?] 疑似约束模式，供人类确认后填入本节

## 架构约束 (AC)

> 跨模块的结构性规则。管控模块间关系、分层边界、数据所有权。

### AC-001（示例条目）：只有数据访问层可直接执行 SQL 查询

- **适用域**：`src/db/`, `src/biz/`
- **阻塞**：是
- **启用**：否

业务层（ViewModel、Service、Helper）的所有数据操作必须通过 DAL 接口进行。
禁止在非 DAL 模块中直接构造 QSqlQuery 或拼接 SQL 字符串。

> 判定方法：检测所有 QSqlQuery（或对应 ORM/DB 驱动的查询对象）的实例化位置
> 是否在适用域规定的路径下。

## 设计约束 (DP)

> 全系统遵守的设计模式、接口契约、继承/组合规则。

### DP-001（示例条目）：跨模块功能调用必须通过接口抽象层

- **适用域**：全局
- **阻塞**：是
- **启用**：否

禁止在业务代码中直接实例化外部模块的具体实现类。跨模块依赖必须通过
接口/抽象基类/工厂方法解耦，使得模块可被替换或 mock。

> 判定方法：检查非同一模块内的 `new ConcreteClass()` 调用——如果 ConcreteClass
> 定义在另一个模块中且没有对应的接口层，则违规。

## 代码约束 (CI)

> 具体编码规则：命名规范、禁止语法、资源管理。

### CI-001（示例条目）：类名使用大驼峰（PascalCase），成员函数使用小驼峰（camelCase）

- **适用域**：`*.h`, `*.cpp`
- **阻塞**：是
- **启用**：否

```bash
# 可自动检测——reviewer 执行此 grep 即可：
grep -Pn 'class\s+[a-z]' *.h  # 类名以小写开头 → 违规
```

## 环境约束 (ENV)

> 工具链、命令规范、运行方式。ENV 约束的对象是 Agent 行为（非代码文本），
> 需前置加载到 implementer 的 system prompt 中。

### ENV-001（示例条目）：所有 Python 脚本通过 `uv run` 执行

- **适用域**：`*.py`
- **阻塞**：是
- **启用**：否

禁止使用 `python script.py` 或 `python3 script.py`。Agent 在 shell 中调用任何
Python 脚本时，必须使用 `uv run` 前缀。

> 检测方式：sdd-apply implementer 加载此约束到 system prompt（事前预防）；
> reviewer 审查工具调用记录（事后兜底）。
