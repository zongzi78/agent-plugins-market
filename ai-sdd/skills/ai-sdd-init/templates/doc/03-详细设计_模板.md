<!--
  章节展开由"权重建议 + 内容条件"双因素决定。

  一、权重评估（来自阶段5 SOP的7维度评分）：
  - 高权重（>=24分）→ 建议全章节展开
  - 中权重（14-23分）→ 建议关键章节展开
  - 低权重（<=13分）→ 建议精简

  二、内容条件（逐章判断，优先级高于权重建议）：
  各章节的展开触发条件如下。满足条件 → 展开该章节（即使权重建议精简）；
  不满足 → 使用标准措辞。

  | 章节 | 展开触发条件 |
  |------|------------|
  | 核心数据模型 | 模块拥有 >=1 个 struct/class 且字段数 >=3 |
  | 对外接口契约 | 模块有 >=1 个 public 方法/函数 |
  | 核心业务流程 | 模块参与 >=1 条跨模块数据流（见00-架构.md）或其方法调用链深度 >=3 |
  | 状态机 | 模块管理 >=1 个有生命周期状态流转的实体（状态数 >=3） |
  | 异常处理与容错 | 代码中存在 try-catch/错误回调/降级/重试/超时/熔断逻辑 |
  | 并发安全 | 模块有共享状态（成员变量被 >=2 个线程访问 或 有跨线程信号槽连接） |
  | 性能特征 | 模块有 O(n²) 或更高复杂度的算法 或 涉及 I/O/网络/大内存分配 |

  三、决策流程：
  1. 先用7维度评估权重，得到建议级别
  2. 逐章检查内容条件：
     - 条件满足 → 展开该章节，不受权重建议限制
     - 条件不满足 → 使用标准措辞
  3. 如果最终展开章节数 >=5 → 视为高深度；3-4 → 中深度；<=2 → 低深度
  4. 检查深度与实际权重是否匹配：高权重模块低深度 → 重新检查各章节条件判断是否有遗漏

  四、God Class 特殊规则：MainWindow 等承担 >=4 种职责的类必须单独成文。
  文档内部按职责拆分子节（3-5 个），每个子节有自己的流程描述。

  五、章节编写顺序（强制）：不要按模板的线性顺序写。按分析难度倒序：
  1. 先写异常处理 + 并发安全（最需要深度分析，消耗最大注意力）
  2. 再写核心业务流程 + 状态机
  3. 再写对外接口契约 + 核心数据模型（最容易从代码提取）
  4. 最后写概述（此时已对整个模块有深入理解，概述质量最高）

  六、各章节"不涉及"时的标准措辞：
    核心业务流程 → "本模块无多步骤业务流程。"
    状态机 → "本模块不管理有生命周期状态流转的实体。"
    异常处理与容错 → "本模块无异常处理逻辑。"
    并发安全 → "本模块无共享状态，无并发风险。"
    性能特征 → "本模块无性能敏感路径。"
-->
