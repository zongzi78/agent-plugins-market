# sdd-explore（探索 + 规划）

## 定位

深度探索代码 + 思维风暴，产出详尽的 plan.md。这是整个工作流中最重的一步。

## 前置检查（Hard Gate）

1. `.ai/` 不存在 → 提示"请先执行 `/ai-sdd:init`"，终止
2. 无活跃 change → 拒绝，提示"请先通过 `/ai-sdd:propose` 创建一个 change"，终止
3. 多个活跃 change → 让用户选择目标 change
4. plan.md 非空（已有内容）→ 警告"plan.md 已有内容"，AskUserQuestion：覆盖 / 从当前状态继续 / 取消

## 执行流程

### 阶段 1：上下文收集

1. 读取目标 change 的 proposal.md
2. 读取 `.ai/doc/00-架构.md` 了解项目架构
3. 读取 `.ai/project-log.md` 了解历史脉络
4. 扫描项目代码结构

### 阶段 2：思维风暴

按 change 实际复杂度灵活调整：

| 复杂度 | 澄清问题数 | 候选方案数 | 设计展示 |
|--------|-----------|-----------|----------|
| 中等 | 2-3 | 2 | 逐节 |
| 复杂 | 3-5 | 2-3 | 逐节确认 |

1. **评估范围**：如果涉及多个独立子系统，先帮用户拆分
2. **逐步澄清**：基于 proposal，逐步提出澄清问题（一次一个问题，多选优先）
3. **探索代码**：用结构图展示架构关系（当前结构 + 变更后结构）
4. **候选方案**：提出候选方案 + 优劣矩阵 + **明确推荐**
5. **逐节确认**：逐节展示设计，每节获得人类确认
6. **Hard Gate**：方案必须经人类批准才能继续
7. **设计自检**：占位符扫描、内部一致性、范围检查、歧义检查

### 阶段 3：产出 plan.md

1. 将确认后的方案写入 change 的 plan.md
2. **No Placeholders Rule**：禁止 TBD、TODO、"类似任务N"、模糊指令
3. 每个任务必须包含的内容：
   - ✅ 完整文件路径、函数签名、数据结构定义
   - ✅ 伪代码级算法描述、精确的 CLI 命令、预期输出
   - ❌ 不必有：生产级完整实现代码（那是 `/ai-sdd:apply` 的事）
4. **必须包含三个章节**：

   **`## 文档变更`**（为 `/ai-sdd:sync` 提供结构化输入）：
   ```markdown
   ## 文档变更

   | 操作 | 目标文件 | 章节 | 变更摘要 |
   |------|----------|------|----------|
   | ADDED | .ai/doc/01-需求.md | F-005 | 新增导出CSV功能需求 |
   | MODIFIED | .ai/doc/03-详细设计/auth.md | §API契约 | 新增 /api/export 端点 |
   ```

   **`## 架构决策`**（决策化石）：
   ```markdown
   ### 决策：[决策名称]
   - **背景**：[为什么需要做这个决策]
   - **方案对比**：A) ... B) ...
   - **选择**：[方案X]，理由：[...]
   - **放弃的方案**：[方案Y]，原因：[...]
   - **影响范围**：[哪些模块/文件受影响]
   ```

   **`## 备选方案（已否决）`**（探索过程存档）：
   - 记录被否决方案及其否决原因，设计讨论的产物不丢失

5. **自检**：占位符扫描、覆盖性检查、内部一致性、文档变更完整性
6. 提交人类审阅

### 阶段 4：更新状态

1. 更新 proposal.md 的 status 为 `planned`
2. 更新 project-log.md

## 联动设计

- **拒绝路径**：无活跃 change → 提示运行 `/ai-sdd:propose`
- **完成后引导**：`✅ 实施方案已写入 .ai/changes/NNN-xxx/plan.md。接下来可通过 /ai-sdd:apply 开始执行编码。`
- **如发现 proposal 需调整**：`⚠️ 探索过程中发现提案需要调整，建议先更新 proposal.md 再继续。`
