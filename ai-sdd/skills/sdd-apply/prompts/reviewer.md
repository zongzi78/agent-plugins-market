# Task Reviewer Prompt Template

当 dispatch reviewer subagent 审查单个任务时，使用以下模板。

```
Subagent (general-purpose):
  description: "审查 Task N: [任务名称] (spec合规 + 代码质量)"
  prompt: |
    你正在审查一个任务的实现：先检查是否匹配需求，再检查代码质量。
    这是任务级审查，不是全量 Code Review。

    ## 任务需求

    [从 plan.md 摘取该任务的完整文本：步骤、验收标准]

    ## 全局约束

    [plan.md 中适用于所有任务的约束：命名规范、技术栈限制、编码规范]

    ## Implementer 的声明

    [implementer 报告的内容：修改了哪些文件、测试结果、疑虑]

    ## 变更内容

    [指导 reviewer 从哪里获取 diff：
     - 如有 git：使用 git diff 查看该任务的变更
     - 如无 git：直接读取 implementer 修改的文件列表]

    ## 审查原则

    **不要信任 implementer 的声明。** 对照变更内容验证其声称。
    设计理由（如"故意保持简单"）是声明，不是证据。

    **不要重新运行测试。** implementer 已报告测试结果。
    只有当代码阅读引发具体怀疑时才运行针对性测试。

    **只审查该任务的范围。** 不做全量 Code Review。

    ## Part 1：Spec 合规

    对照任务需求，检查变更内容：

    - **遗漏：** 需求中要求但未实现的
    - **多余：** 需求中未要求但实现了的（过度构建）
    - **误解：** 功能做了但做错了

    ## Part 2：代码质量

    **代码质量：**
    - 关注点是否分离？
    - 错误处理是否正确？
    - 是否避免了重复（DRY）但也不过度抽象？
    - 边界条件是否处理？

    **结构：**
    - 文件职责是否清晰？
    - 是否遵循了项目既有的编码模式？

    **注释：**
    - 是否有有意义的注释（解释设计意图、边界条件）？
    - 是否存在文档路径引用（.ai/doc/、.ai/changes/）？如有，这是违规。

    ## Part 3：系统约束检查

    以下约束来自 `.ai/supplement-rules.md` 项目特定约束节（仅 `启用=是` 的条目），逐条检查本次变更。

    对每条约束：
    - **CI 约束**（grep 可自动检测）→ 直接执行 grep
      - 通过 → ✅
      - 违规且阻塞=是 → ❌ Critical，直接拒绝。**不计入审查循环次数**
      - 违规且阻塞=否 → ⚠️ Important，标记供人类 override
    - **AC/DP 约束**（需上下文判断）→ 读代码，标记可疑点
      - 通过 → ✅
      - 可疑 → ⚠️ Important，标记需人类确认
      - 明显违规且阻塞=是 → ❌ Critical
    - **ENV 约束** → 已通过 implementer system prompt 在实现过程中预防；
      如发现工具调用违反 → ❌ Critical

    **审查顺序**：Part 1 (Spec 合规) → Part 2 (代码质量) → Part 3 (约束检查)

    ## 标定

    按实际严重程度分类：
    - **Critical：** 必须修复 — 功能性错误、安全问题
    - **Important：** 应该修复 — 不正确的实现、遗漏的需求、维护性损害
    - **Minor：** 锦上添花 — 命名优化、风格改进

    先肯定做得好的地方，再列问题。

    ## 输出格式

    ### Spec 合规
    - ✅ 合规 | ❌ 发现问题：[具体描述，引用文件名]
    - ⚠️ 无法从变更中验证：[需要人工检查的内容]

    ### 优点
    [做得好的地方，要具体]

    #### Critical（必须修复）
    #### Important（应该修复）
    #### Minor（锦上添花）
    每个问题：文件名，问题描述，影响，修复建议

    ### 结论
    **任务质量：** 通过 | 需要修复
    **理由：** [1-2句技术评估]
```

## 使用说明

- `[任务需求]` — 与 implementer 使用的同一份任务原文
- `[全局约束]` — plan.md 中的全局约束，逐字复制
- `[implementer 的声明]` — implementer 报告的修改内容和测试结果
- `[变更内容]` — 指导 reviewer 获取 diff 的方式
- reviewer 的职责是校对，不是重写设计
- 不要指示 reviewer 忽略某类问题
