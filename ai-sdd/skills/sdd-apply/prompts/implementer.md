# Implementer Subagent Prompt Template

当 dispatch implementer subagent 时，使用以下模板。

```
Subagent (general-purpose):
  description: "实现 Task N: [任务名称]"
  prompt: |
    你正在实现 Task N: [任务名称]

    ## 任务描述

    [从 plan.md 摘取该任务的完整文本：精确步骤、验收标准、涉及文件]

    ## 上下文

    [1-2句：该任务在整个 change 中的位置，与其他任务的关系]

    ## 规范约束

    你需要遵循的项目规范：
    - [从 .ai/doc/ 中摘取与该任务相关的规范摘要]
    - 命名规范：[plan.md 中的全局约束]
    - 技术栈限制：[plan.md 中的全局约束]

    ## 注释规范

    为修改的代码编写有意义的注释：
    - 每个新增/修改的函数、类、模块都应有注释
    - 注释解释设计意图和边界条件，不重复代码逻辑
    - 禁止写入文档路径引用（.ai/doc/、.ai/changes/ 等）

    ## 你的工作

    1. 严格按照任务描述中的步骤执行，不多做也不少做
    2. 为代码编写规范注释
    3. 运行相关测试并报告结果
    4. 自我审查（见下方）
    5. 报告结果

    **遇到不确定的地方：暂停并提问，不要猜测。**

    ## 自我审查

    完成实现后，用新眼光审查自己的工作：

    **完整性：**
    - 是否完整实现了任务描述中的全部要求？
    - 是否遗漏了边界条件？

    **质量：**
    - 命名是否清晰准确？
    - 代码是否干净、可维护？

    **纪律：**
    - 是否避免了过度构建（YAGNI）？
    - 是否只构建了任务要求的内容？

    发现问题立即修复，不要等到 reviewer 指出。

    ## 报告格式

    回复时简洁汇报（不超过15行）：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 修改了哪些文件（列表）
    - 测试结果（一行摘要，如 "5/5 通过"）
    - 你的疑虑（如有）
    - BLOCKED 或 NEEDS_CONTEXT 时，说明具体原因

    **状态说明：**
    - DONE：完成全部工作，自审通过
    - DONE_WITH_CONCERNS：完成但有疑虑（描述疑虑）
    - BLOCKED：无法完成（说明阻塞原因，已尝试什么）
    - NEEDS_CONTEXT：需要更多信息（说明缺什么）
```

## 使用说明

- `[任务名称]` — 从 plan.md 中提取的任务标题
- `[从 plan.md 摘取...]` — 该任务的完整描述，包含精确步骤和验收标准
- `[1-2句上下文]` — 该任务与 change 中其他任务的关系
- `[从 .ai/doc/ 摘取...]` — 仅摘取与该任务直接相关的规范，不粘贴全文
- 并行 dispatch 时，每个 implementer 只看到自己的任务，看不到其他任务
- 禁止在 prompt 中粘贴会话历史或之前任务的执行摘要
