## 决策记录

### 状态说明

| 状态 | 含义 | 理由来源 | Agent 消费指引 |
|------|------|---------|---------------|
| `Confirmed` | 决策理由经人类确认或补充，可靠性最高 | 人类在确认环节填写 | **可依赖**：后续设计可直接引用此决策理由作为约束。修改此决策涉及的代码时，须在 plan.md 中说明为何可以违背已有决策 |
| `Inferred` | Agent 基于代码模式、命名约定、上下文推断的理由，未经人类验证 | Agent 在 reverse 阶段自动生成，或由 sdd-explore 初步提出 | **可参考但保持怀疑**：若后续设计与此理由矛盾，优先相信自己判断并标记冲突。若修改涉及此决策的代码，应尝试从变更中获取更多信息，并在 sync 时升级为 Confirmed |
| `Observed` | Agent 观察到了某个技术选择的结果，但无法推断原因 | Agent 在 reverse 阶段标记 | **仅知晓结果，不做归因**：后续设计不应依赖任何关于"为什么"的假设。若修改涉及此选择，这是引入人类确认的适当时机 |
| `Untraceable` | 经人类确认，决策理由已永久不可追溯 | 人类在确认环节标记 | **接受信息丢失**：不要再追问"为什么"，当作既成事实处理。修改时按当前最佳实践重新决策 |

**Agent 读取决策记录时的快速策略**：
1. 先扫描"技术栈与依赖" → 了解项目技术边界、版本约束、构建工具链
2. 按需要决策的层级定位：做架构决策 → 看"架构级"表；做模块设计 → 看"设计级"表
3. 逐条消费：
   - 状态=Confirmed → 记录下来作为硬约束
   - 状态=Inferred → 记录下来作为软约束，保持怀疑态度
   - 状态=Observed/Untraceable → 接受既成事实，不深究原因
4. 若修改的代码与某条 Inferred 决策相关：
   - 先尝试从代码变更中确认或推翻推断理由
   - 若确认 → sync 时升级为 Confirmed
   - 若推翻 → sync 时追加新的 Confirmed 条目，旧条目标记为 Untraceable

## 关键规则

> **Agent 必须给出"推断理由"**（不能留空）。Agent 必须基于代码模式、命名约定、上下文给出一条推断理由并标记为 `Inferred`。人类有机会修正或确认，但不是必须的。
>
> **状态流转**：
> ```
> Observed → Inferred → Confirmed
>     │                      │
>     └──────────────────────┼──→ Untraceable（经人类确认理由已不可追溯）
>                            └──→ 若确认该决策导致了问题 → 04 追加 DD/AV 条目，02 的"关联问题"列补上引用
> ```
