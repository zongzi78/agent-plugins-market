### Ref 文档的性质

你要处理的 ref 文档（`.ai/doc/` 下的所有文件）**不是**首次 reverse 时的
"外部参考文档"。它们是：

- **腐烂的 AI-SDD 文档**：内容可能过时、与代码事实不符
- **角色污染的材料**：某些内容出现在了错误的文档中
  （如 00-架构.md 中出现了 API 细节，01-行为目录.md 中出现了实现细节）
- **结构退化的文件**：存在追加式更新（内容堆在末尾）、组织混乱
- **有保留价值的碎片**：其中人类明确确认过的元信息需要保留

### 你的任务（按优先级排序）

#### 任务 1：重组（Reorganize）
逐份阅读 ref 文档，判断每条内容当前是否在其**正确的文档**中。
判断标准：各模板顶部的「📋 本文档角色」区块中的"应该写"/"不应该写"列表。

具体步骤：
1. 逐文档、逐章节阅读 ref 文档
2. 对每条内容判断：这段内容属于哪个文档的角色范围？
3. 内容在正确的文档 + 正确的章节 → 保留原位
4. 内容在错误的文档中 → 移动到正确文档的正确章节
5. 内容属于多个文档都不该有的（如实现过程记录、讨论记录）→ 删除

重组后，每份文档只包含属于它角色范围的内容。
不需要的内容不保留、不挪到末尾、不创建"杂项"章节。

#### 任务 2：修正（Correct）
逐条对照 check 诊断报告中的漂移项：
- 架构漂移 → 更新 `00-架构.md` 的模块列表、依赖关系（以代码实际目录结构为准）
- 行为漂移 → 更新 `01-行为目录.md` 的行为清单（新增/废弃，以代码实际 API/UI 入口为准）
- 设计漂移 → 更新 `03-详细设计/` 的 API 契约（以代码实际签名为准）
- 角色越界 → 已在任务 1 中处理

修正时的铁律：**以代码为准**。不因为 ref 文档说 X 就保留 X——
如果代码实际做的是 Y，文档必须写 Y。

#### 任务 3：补全（Complete）
像首次 reverse 一样遍历代码，发现 ref 文档中**完全没有覆盖**的内容：
- `00-架构.md`：代码中有但文档中没有的模块 → 按阶段 2 SOP 生成
- `01-行为目录.md`：代码中有但文档中没有的可观察行为 → 按阶段 3 SOP 生成
- `02-决策记录.md`：代码中可观察到的但文档中没有记录的技术栈事实和设计决策 → 按阶段 4 SOP 生成
- `03-详细设计/`：代码中有但文档中没有的模块详细设计 → 按阶段 5 SOP 生成
- `04-问题与改进.md`：从代码中新发现的问题 → 调用 ai-sdd-diagnose 生成

对新发现的内容，按 reverse 各阶段 SOP 的标准生成并填充到正确的文档中。

保留规则：

  但如果代码事实已经改变（如原来用 QHash 现在改用 std::map），

### 与首次 reverse 的关键区别

| 维度 | 首次 reverse | 本次 rebuild |
|------|-------------|-------------|
| 起点 | 空白项目，无从参考 | 有腐烂的文档作为"待修复材料" |
| 阶段 0-1 | 需要（VC 探测、项目用途确认） | **跳过**——项目信息已在 project-log.md 中 |
| 对 ref 文档的态度 | 理解辅助，看完就放一边 | **待修复的主体**——需要逐条判断保留/移动/丢弃 |
| 核心动作 | 生成 | **重组 + 修正 + 补全 + 保留** |
| 元信息 | 无历史负担 | **必须保留人类确认过的内容** |

### 质量要求

遵循 reverse 的全部质量规则：
- **正向 6 条**：以真实代码为准、创建心态、尊重文件角色、结构化表述、标注符号锚点、提供增量价值
- **负向 5 条**：禁止"..."占位符、禁止大段原始代码、禁止跳过不确定的内容、禁止将推断包装为事实、禁止猜测业务含义

各阶段 SOP 的产出标准同样适用于对应的任务。
