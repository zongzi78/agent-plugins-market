# AI-SDD 文档腐烂修复

## 定位

⚠️ **依赖 ai-sdd-reverse。** 执行前确保 reverse skill 已加载——rebuild 通过调用 reverse 的 ref 模式来执行重建。

执行文档腐烂修复：

1. 以腐烂的 AI-SDD 文档为 ref（参考材料），以代码为真相源
2. 通过精心设计的提示词，调用 reverse 的 ref 模式执行"重组 + 修正 + 补全 + 保留"四步修复
3. rebuild 不是独立引擎——所有修复通过 reverse 完成

## 前置检查

1. 检查 `.ai/` 是否存在 → 如不存在，提示先运行 `/ai-sdd-init`
2. 检查 `.ai/doc/` 是否存在且有内容 → 如为空，提示"无文档可重建，建议运行 /ai-sdd-reverse 进行首次逆向化"

## 执行流程

### 步骤 1：运行漂移检测

按照 `/ai-sdd-check` 的流程，执行完整的漂移检测。产出诊断报告，包含：
- 哪些文件有漂移、漂移类型、严重程度
- 角色越界（内容出现在错误的文档中）
- 结构退化（追加式更新、内容堆在末尾）

### 步骤 2：展示诊断结果 + 确认重建范围

展示诊断报告摘要，列出建议重建的文件范围，用 AskUserQuestion 让人类确认：

```
=== 文档腐烂诊断摘要 ===

[漂移项列表]

建议重建范围：
  A) 全量重建 — 所有 doc 文件重新从代码生成
  B) 选择性重建 — 仅重建有漂移的文件：[列出]
  C) 取消
```

### 步骤 3：提取需保留的元信息

在重建前，读取旧文档并提取以下元信息：

- `02-决策记录.md` 中 `status=Confirmed` 的决策条目（人类确认过的决策理由）
- `01-行为目录.md` 中人类确认的行为状态裁决
- 各文档 front matter 中的 `last_code_verified` 等时间戳

这些元信息将作为"保留清单"传入 reverse，确保人类的知识积累不丢失。

### 步骤 4：调用 reverse ref 模式

将 `.ai/doc/` 作为 ref 路径传入 reverse。

**Read [references/rebuild-prompt.md](references/rebuild-prompt.md) now and inject it into the reverse call.**

该提示词定义了四个任务：重组 → 修正 → 补全 → 保留，以及与首次 reverse 的关键区别表。

### 步骤 5：输出重建摘要

重建完成后，输出摘要：

```
✅ 文档重建完成。

重建范围：[全量 / 选择性]
重组：[N] 处内容移动到正确文档
修正：[N] 处过时描述已更新
补全：[N] 个新模块/[N] 个新行为/[N] 条新决策
保留：[N] 条人类确认的元信息已保留

⚠️ 待人类处理：
  - [如有标注"待确认"或"待重新确认"的内容，列出]
```

---

## 💡 常见陷阱（Gotchas）

- **人类确认过的元信息提取不完整会导致知识永久丢失**：步骤 3 的元信息提取是单向操作——提取 → 重建 → 元信息注入。如果提取遗漏了某条 Confirmed 决策，它将在重建中消失。
- **check 的诊断报告可能遗漏某些漂移类型**：check 覆盖 S06 七种失效模式中的 4 种。如果漂移属于另外 3 种（模式僵化/AI盲从/橡皮图章），rebuild 不自知。以代码为准的铁律是最后的防线。
- **如果旧文档中的人类确认内容已被代码演进推翻**：不要强行保留。标注 `⚠️ 待重新确认`，而不是假装它还有效。
