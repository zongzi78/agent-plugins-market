# AI-SDD 问题诊断

## 定位

从代码中系统性发现问题，是 `.ai/doc/04-问题与改进.md` 的唯一写入者。

**调用场景**：
- reverse 流程的集中诊断阶段（阶段 6）：拿到完整的 00-03 文档后，一次性逐层扫描
- 独立运行：项目迭代后的定期健康检查
- 分层调用：sdd-archive 前的回归诊断（`layer=D, depth=L1`）
- 深度审计：重大版本前的全层级深度扫描

**与其他 Skill 的分工**：
| | ai-sdd-check | ai-sdd-diagnose（本 Skill） |
|---|---|---|
| 比较对象 | doc ↔ code（文档是否过时） | code ↔ 问题模式（代码有什么隐患） |
| 输出 | 漂移报告（临时） | 04-问题与改进.md（持久） |
| 修改文件 | ❌ 不修改 | ✅ 写入 04 |
| 核心问题 | "文档说谎了吗？" | "代码有什么隐患？" |

---

## 分级体系

### 层级（Layer）—— 决定查什么

| 层级 | 覆盖的 04 子节 | 典型搜索范围 |
|------|---------------|-------------|
| **A** - 架构层 | AV, DD(架构相关) | 循环依赖、God Class、跨层调用、防腐层绕过、单例扩散、模块粒度异常 |
| **R** - 需求层 | DR, FD | 声明-实现不一致、命名-语义偏离、空函数体、空 if/控制流残留、废弃功能痕迹 |
| **T** - 技术层 | VR, TR | 版本漂移、EOL 依赖、取舍迹象（同功能双实现、自实现轮子、硬编码限制） |
| **D** - 设计/代码层 | MT, DD(设计相关), CD | 哨兵值陷阱、双重数据结构、双向映射、weak_ptr 悬空、跨线程无锁、RAII 缺失、注释代码、魔法数字、硬编码路径、未使用声明 |
| **ALL** | 全部子节 | 以上所有 |

### 深度（Depth）—— 决定查多深

| 深度 | 适用场景 | 行为 |
|------|---------|------|
| **L1** - 快速 | sdd-archive 前回归检查、日常快速扫描 | grep 模式匹配（TODO/FIXME/魔数/硬编码路径）；AST 级别结构检测（空函数体、死代码） |
| **L2** - 标准 | reverse 流程集中诊断、独立定期检查（默认） | L1 + 跨文件依赖分析 + 并发路径追踪 + 命名语义分析 |
| **L3** - 深度 | 重大版本审计、安全审查 | L2 + 数据流追踪 + 全量 lint 式模式反检测清单 |

### 调用矩阵

```
                L1    L2    L3
层级 A（架构）   ✓     ✓     ✓
层级 R（需求）   ✓     ✓     ✓
层级 T（技术）   ✓     ✓     ✓
层级 D（设计）   ✓     ✓     ✓
层级 ALL         ✓     ✓     ✓
```

**reverse 流程中的调用**：
- 阶段 6 集中诊断 → `diagnose(layer=ALL, depth=L2)`

**独立调用**：
- `/ai-sdd-diagnose` → `diagnose(layer=ALL, depth=L2)` — 全层级标准扫描
- `/ai-sdd-diagnose --deep` → `diagnose(layer=ALL, depth=L3)` — 深度审计
- `/ai-sdd-diagnose --layer=D` → `diagnose(layer=D, depth=L2)` — 只查设计/代码层

## 执行流程

### 步骤 1：读取已有文档
- 读取 `04-问题与改进.md` 当前内容（用于去重——同文件同行号的同类型发现不重复写入）
- 读取 00-03 全部文档（物化的系统理解，用于关联跨层级根因）
- 读取 `project-log.md` 中的「诊断观察便签」区段（阶段 2-5 的洞察线索）

### 步骤 2：执行搜索
按 `layer × depth` 参数，执行下方「搜索清单」中对应范畴的所有检查项。

### 步骤 3：从高层级到低层级逐层梳理
不按分类平铺，而是以**问题为线索**：
1. 先识别架构层根因（God Class、循环依赖、单例扩散……）
2. 向下追踪每个根因在各层级的表现（AV → DD → MT → CD）
3. 关联为问题组，每个组有一个根因条目 + 各层级的子条目

例：
```
DD-004: 全局消息队列设计缺陷（根因——架构层）
  ├─ AV 视角: extern 全局变量跨 4 个编译单元 → 隐式耦合
  ├─ MT 视角: CMsgThread 跨线程追加，CListModel 读端无锁
  ├─ CD 视角: QMutex 裸 new 无 RAII 包装
  └─ 修复: 将 g_msgVec + g_pMutex + g_stopMsgThread 封装为 MessageQueue 类
```

### 步骤 4：全局严重程度标准化
收集所有发现后，横向对比，统一校准严重程度：
- 跨层级影响 + 无当前规避 → 高
- 单层级影响但有明确的运行时触发场景 → 中
- 代码质量问题、无运行时触发场景 → 低

### 步骤 5：去重与合并
- 按 `(file:line, 问题类型)` 做 key，与 04 已有条目去重
- 与观察便签交叉验证：便签中的线索是否已被覆盖？如有遗漏 → 补充分析

### 步骤 6：写入 04
- 追加新条目到对应子节
- 关联条目在描述中互相引用（如 `相关: MT-002 是本文档 DD-004 在模块层的表现`）
- 更新 front matter（`last_updated`, `confidence`）
- 删除 project-log.md 中的「诊断观察便签」区段

### 步骤 7：返回结构化摘要
```
=== 诊断完成 ===
架构层: 新增 N 项（AV-X, DD-X），去重跳过 M 项
需求层: 新增 N 项（DR-X, FD-X）
技术层: 新增 N 项（VR-X, TR-X）
设计层: 新增 N 项（MT-X, DD-X, CD-X）

待人类确认:
  - UC-001: [具体问题] — 验证方式: [怎么做]
  - UC-002: [具体问题] — 验证方式: [怎么做]
```

## 搜索清单

> 每个范畴定义：检查项 + L1/L2/L3 的检测方法 + 输出目标（04 子节）+ 默认严重程度 + 上调/下调条件。

### 范畴 1：代码残留 → CD

| # | 检查项 | L1 检测 | L2 附加 | L3 附加 | 默认严重程度 |
|---|--------|---------|---------|---------|------------|
| 1.1 | TODO/FIXME/HACK/XXX 注释 | grep 匹配 | 判断是否涉及未完成设计 → 同时标记 DD | 评估遗留期限和影响 | 低（常规 TODO）/ 中（涉及核心功能） |
| 1.2 | 注释掉的代码块（连续 ≥3 行） | grep 连续 `//` 或 `/* */` 注释 | 判断是否有说明注释 | 评估是否应恢复 | 低 |
| 1.3 | 空函数体（`{ }` 或 `{ return; }`） | AST 扫描 | 判断函数名是否暗示应有功能 → 同时标记 DR | 追踪调用链验证是否真的无操作 | 中（函数名暗示功能）/ 低（纯虚函数/默认实现） |
| 1.4 | 未使用的声明 | 声明-引用对比 | 区分预留接口 vs 死代码 | 版本控制历史确认 | 低 |

### 范畴 2：硬编码 → CD

| # | 检查项 | L1 检测 | L2 附加 | L3 附加 | 默认严重程度 |
|---|--------|---------|---------|---------|------------|
| 2.1 | 魔法数字 | 扫描裸数字常量（排除 0, 1, -1 常规用法） | 判断是否跨文件重复出现 | 评估业务含义和合理取值 | 高（仿真参数/超时值）/ 中（UI 尺寸）/ 低（格式化宽度） |
| 2.2 | 硬编码路径 | grep 文件路径模式（含 `/` `\`） | 判断是否绝对路径、是否可配置 | 跨平台兼容性评估 | 中 |
| 2.3 | 硬编码网络地址 | grep IP/URL/host:port 模式 | 区分本地回环 vs 外部地址 | 安全影响评估 | 高（外部地址）/ 低（localhost） |

### 范畴 3：并发安全 → MT

| # | 检查项 | L1 检测 | L2 附加 | L3 附加 | 默认严重程度 |
|---|--------|---------|---------|---------|------------|
| 3.1 | 跨线程无锁访问 | — | 追踪全局容器/变量的所有读写点，标记所在线程 | 评估 Qt 隐式串行化的可靠性 | 高 |
| 3.2 | TryLocker 超时丢消息 | — | 扫描 QMutexTryLocker 使用点，检查超时后行为 | 追踪消息丢失对上游的影响 | 中 |
| 3.3 | weak_ptr 悬空 | — | 扫描 weak_ptr::lock() 调用点，检查返回值是否判空 | 追踪引用对象的生命周期 | 中 |
| 3.4 | 信号槽跨线程连接 | — | 检查 Qt::QueuedConnection 传递的数据是否线程安全 | 检查信号参数的生命周期 | 中 |

### 范畴 4：数据完整性 → MT（陷阱）/ DD（设计债务）

| # | 检查项 | L1 检测 | L2 附加 | L3 附加 | 默认严重程度 |
|---|--------|---------|---------|---------|------------|
| 4.1 | 双重数据结构 | — | 检查同一逻辑数据是否维护在 ≥2 个独立容器中 | 追踪所有修改路径，验证同步完整性 | 高（无同步断言）/ 中（有 assert 保护） |
| 4.2 | 双向映射手动同步 | — | 检查 name↔ID / parent↔child 等映射对是否手动维护 | 统计同步操作点的一致性 | 中 |
| 4.3 | 哨兵值 | 扫描 order=-1 / nullptr / -1 等特殊值的使用 | 判断是否在所有调用方都有检查 | 评估哨兵值滥用风险 | 中 |
| 4.4 | 调用顺序依赖 | — | 检查是否有"必须先调 A 再调 B"的隐式约定 | 是否可通过接口设计消除 | 中 |

### 范畴 5：架构合规 → AV（违规）/ DD（设计债务）

| # | 检查项 | L1 检测 | L2 附加 | L3 附加 | 默认严重程度 |
|---|--------|---------|---------|---------|------------|
| 5.1 | God Class | 扫描单类方法数 >20 且涉及 ≥3 种不同数据域 | 分析职责分组，提出拆分方案 | 评估拆分对调用方的影响 | 高 |
| 5.2 | 循环依赖 | 分析 #include 链路（含 .cpp 隐藏依赖） | 区分直接循环 vs 间接循环 | 提出接口抽象方案 | 高 |
| 5.3 | 跨层直接调用 | 检查表示层是否跳过业务逻辑层直接访问数据层 | 检查 .cpp 中的隐藏跨层 #include | 统计违规次数 | 中 |
| 5.4 | 单例扩散 | 扫描 Singleton 模式使用，统计全局可访问实例数 | 评估是否可通过依赖注入替代 | 对可测试性的影响评估 | 中 |
| 5.5 | extern 全局变量跨模块 | 扫描 extern 声明，统计跨编译单元共享的变量 | 追踪每个全局变量的读写点线程 | 提出封装方案 | 高 |

### 范畴 6：需求一致性 → DR（漂移）/ FD（缺陷）

| # | 检查项 | L1 检测 | L2 附加 | L3 附加 | 输出目标 |
|---|--------|---------|---------|---------|---------|
| 6.1 | 声明-实现不一致 | 对比函数名暗示的语义与实际函数体 | 追踪调用方是否依赖了未实现的功能 | 版本控制历史确认是否曾是功能 | DR（不完整）/ FD（空壳） |
| 6.2 | 命名-语义偏离 | 对比变量/类/宏名与代码中的实际数据域 | 判断是历史遗留还是设计失误 | 评估重命名影响范围 | DR（不一致） |
| 6.3 | 调用-接口不一致 | 检查调用方传参与被调用方实际使用 | — | 评估是否应修改调用方还是接口 | DR |
| 6.4 | 控制流残留 | 扫描空 if/else/while/for 块 | 检查版本控制历史确认是否曾有逻辑 | — | DR（废弃痕迹） |
| 6.5 | 被注释的功能入口 | 检查被注释的实例化/注册/路由代码 | 判断是暂未启用还是已永久废弃 | — | FD |

### 范畴 7：技术风险 → VR

| # | 检查项 | L1 检测 | L2 附加 | L3 附加 | 默认严重程度 |
|---|--------|---------|---------|---------|------------|
| 7.1 | 版本漂移风险 | 从构建文件提取依赖版本，对比最新稳定版 | 评估跨大版本迁移成本 | 检查 CVE 数据库 | 高（EOL 依赖）/ 中（有版本差距）/ 低（当前版本仍受支持） |
| 7.2 | 构建工具过时 | 检查构建工具版本（qmake/CMake/MSVC 等） | 评估迁移到当前工具链的成本 | 检查新工具链的兼容性 | 中 |
| 7.3 | 依赖版本锁定 | 检查是否有不可独立升级的平台锁定依赖 | 评估单点升级的可行性 | 记录实际使用的 API 子集 | 中 |

### 范畴 8：设计取舍 → TR

> TR 不从代码推断取舍理由（不可能），而是识别"遗迹"——代码中暗示此处可能有过权衡选择的特征。

| # | 取舍迹象（触发条件） | 检测方法 | 示例 |
|---|---------------------|---------|------|
| 8.1 | 同功能双实现 | 扫描：同一功能在不同位置以不同方式实现 | 一处用 QHash 一处用 QMap 存同类数据 → 可能的取舍维度：性能 vs 顺序保证 |
| 8.2 | "舍近求远"实现 | 扫描：自实现排序/查找/引用计数而非用标准库/Qt 内置 | 自实现去重逻辑而非用 QSet → 可能的取舍维度：特殊语义 vs 维护成本 |
| 8.3 | 手动内存管理 | 扫描：裸 new/delete 而非 Qt 父子对象机制/智能指针 | 裸 new QMutex 无 delete 配对 → 可能的取舍维度：显式控制 vs RAII 安全 |
| 8.4 | 硬编码系统限制 | 扫描：固定的上限值/超时值/采样数，无配置入口 | 200 条日志上限 → 可能的取舍维度：简单性 vs 可配置性 |
| 8.5 | 轮询替代事件驱动 | 扫描：QTimer 轮询状态而非连接信号 | 定时检查标志位而非响应信号 → 可能的取舍维度：实现简单 vs CPU 开销 |

每条 TR 输出格式：

| 编号 | 观察到的取舍迹象 | 代码证据 | 可能的取舍维度 | 取舍理由（待人类补充） | 标记日期 |
|------|-----------------|----------|---------------|---------------------|----------|

### 范畴 9：模式诊断 → DD（设计模式级）/ CD（代码模式级）

> 本范畴不依赖 grep 搜索，而是逐项审视代码是否违反了特定设计原则。

| # | 检查项 | 检测方法 | 输出目标 | 默认严重程度 |
|---|--------|---------|---------|------------|
| 9.1 | 全局可变状态 | 扫描 g_ 前缀 / extern / static 非 const 变量。跨编译单元共享 → DD；单文件内 → CD | DD/CD | 高（跨线程）/ 中（单线程） |
| 9.2 | 混用编码约定 | 检查新旧语法混用（Qt4/Qt5 connect）、命名风格不一致、中英文混用 | CD | 中 |
| 9.3 | RAII 合规 | 检查裸 new 是否都有对应的 delete 或智能指针包装；手动 lock/unlock 是否有 RAII 封装 | CD | 中 |
| 9.4 | 职责过载 | 单类方法数 >20 且涉及 ≥3 种数据域 → DD；单方法 >80 行且含 ≥3 种操作 → CD | DD/CD | 中 |
| 9.5 | 双重/多重数据结构 | 同一数据维护在 ≥2 个独立容器中，增删需同步（⚠️ 与 4.1 呼应，本项输出设计层面的诊断） | DD | 高 |

## 输出质量规则

### "待确认"标记的使用限制

允许使用"待确认"的条件（必须同时满足）：
1. 已阅读了所有相关的代码路径（调用方、被调用方、错误处理分支）
2. 代码中确实不存在对该场景的处理逻辑
3. 无法从代码推断该场景是否会实际发生

禁止使用"待确认"的情况：
- 只因为没读完相关代码就标注 → 必须先完成代码阅读
- 同一发现中 ≥3 个"待确认" → 降级该发现的整体严重程度

当标注"待确认"时，必须附带：
- 已检查的代码路径（文件:行号列表）
- 为什么代码无法回答这个问题
- 要确认需要什么信息或验证方式

### 与其他 Skill 的协作

- **ai-sdd-check** 发现严重漂移时 → 建议运行本 Skill 深度审计对应层级
- **sdd-archive** 归档前 → 自动调用本 Skill（layer=ALL, depth=L1）检查是否引入新问题
- **ai-sdd-rebuild** 重建后 → 调用本 Skill（layer=ALL, depth=L2）重新诊断

## 独立调用时的前置检查

1. 检查 `.ai/doc/` 是否存在且有内容 → 如无，提示先运行 `/ai-sdd-reverse` 或 `/ai-sdd-init`
2. 检查 00-03 文档是否存在 → 缺少关键文档时，提示该文档缺失可能影响对应层级诊断的准确性
3. 建议先运行 `/ai-sdd-check` 确认文档与代码的一致性 → 如果漂移严重，在过时文档基础上诊断可能产生误导

## 注意事项

- 本 Skill 是 04-问题与改进.md 的**唯一写入者**。其他 Skill（包括 reverse SOP 各阶段）不再直接写入 04
- 观察便签（project-log.md 中的临时区段）在诊断完成后删除
- 独立运行时，如果不存在观察便签，跳过步骤 1 中的便签读取即可
- L3 深度可能耗时较长——大型项目建议在独立会话中运行
