# AI-SDD 问题诊断

## 定位

执行以下诊断工作流：

1. 从代码中系统性发现问题（架构违规、需求漂移、功能缺陷、版本风险、模块陷阱、设计债务、代码债务）
2. 写入 `.ai/doc/04-问题与改进.md`——**其他任何 Skill 不得直接写入或修改此文件**

**调用场景**：
- reverse 流程的集中诊断阶段（阶段 6）
- 独立运行：项目迭代后的定期健康检查
- 分层调用：sdd-archive 前的回归诊断（`layer=D, depth=L1`）
- 深度审计：重大版本前的全层级深度扫描

---

## 分级体系

### 层级（Layer）—— 决定查什么

| 层级 | 覆盖的 04 子节 | 典型搜索范围 |
|------|---------------|-------------|
| **A** - 架构层 | AV, DD(架构相关) | 循环依赖、God Class、跨层调用、防腐层绕过、单例扩散、模块粒度异常 |
| **R** - 需求层 | DR, FD | 声明-实现不一致、命名-语义偏离、空函数体、空 if/控制流残留、废弃功能痕迹 |
| **T** - 技术层 | VR | 版本漂移、EOL 依赖 |
| **D** - 设计/代码层 | MT, DD(设计相关), CD | 哨兵值陷阱、双重数据结构、双向映射、weak_ptr 悬空、跨线程无锁、RAII 缺失、注释代码、魔法数字、硬编码路径、未使用声明 |
| **ALL** | 全部子节 | 以上所有 |

### 深度（Depth）—— 决定查多深

| 深度 | 适用场景 | 行为 |
|------|---------|------|
| **L1** - 快速 | sdd-archive 前回归检查、日常快速扫描 | grep 模式匹配（TODO/FIXME/魔数/硬编码路径）；AST 级别结构检测（空函数体、死代码） |
| **L2** - 标准 | reverse 流程集中诊断、独立定期检查（默认） | L1 + 跨文件依赖分析 + 并发路径追踪 + 命名语义分析 |
| **L3** - 深度 | 重大版本审计、安全审查 | L2 + 数据流追踪 + 全量 lint 式模式反检测清单 |

### 调用矩阵

```
                L1    L2    L3
层级 A（架构）   ✓     ✓     ✓
层级 R（需求）   ✓     ✓     ✓
层级 T（技术）   ✓     ✓     ✓
层级 D（设计）   ✓     ✓     ✓
层级 ALL         ✓     ✓     ✓
```

**reverse 流程中的调用**：
- 阶段 6 集中诊断 → `diagnose(layer=ALL, depth=L2)`

**独立调用**：
- `/ai-sdd-diagnose` → `diagnose(layer=ALL, depth=L2)` — 全层级标准扫描
- `/ai-sdd-diagnose --deep` → `diagnose(layer=ALL, depth=L3)` — 深度审计
- `/ai-sdd-diagnose --layer=D` → `diagnose(layer=D, depth=L2)` — 只查设计/代码层

## 执行流程

### 步骤 1：读取已有文档
- 读取 `04-问题与改进.md` 当前内容（用于去重——同文件同行号的同类型发现不重复写入）
- 读取 00-03 全部文档（物化的系统理解，用于关联跨层级根因）
- 读取 `project-log.md` 中的「诊断观察便签」区段（阶段 2-5 的洞察线索）
- 如果被 reverse 流程的阶段6调用，读取阶段3产出的「行为-代码-问题线索映射」
- 读取 `.ai/supplement-rules.md`（如存在）→ 提取项目特定约束节中 `启用=是` 的 AC/DP/CI 约束，作为判定标准（"尺子"）。ENV 约束不在此检测（由 sdd-apply reviewer 负责）

### 步骤 2：执行搜索

⚠️ **强制步骤**：无论观察便签是否完备，必须独立执行以下全部搜索清单。
观察便签是"提示补充"——告诉你可能遗漏了什么，不是"替代扫描"——不能取代搜索清单。

执行要求：
- 对搜索清单中的每个检查项，即使观察便签中没有对应线索，也必须执行检测
- 每个范畴的 L2 检测必须实际执行（如"跨文件依赖分析"意味着必须打开相关文件追踪，不是仅凭文件名推断）
- 不因范畴靠后（7/8/9）而降低执行标准——所有范畴同等对待
- 不预判"这个项目可能没有这类问题"而跳过检测——检测后没有发现是合法的，不检测就声称没有是违规

按 `layer × depth` 参数，执行下方「搜索清单」中对应范畴的所有检查项。

**完成范畴 1-3 后 — 第一段自检**：
- 是否对代码残留（范畴 1）的每个检查项实际执行了 grep/AST？还是仅凭印象？
- 是否检查了硬编码（范畴 2）的跨文件重复（L2）？
- 是否逐文件追踪了并发安全（范畴 3）的全局变量读写点（L2）？

**完成范畴 4-6 后 — 第二段自检**：
- 是否追踪了数据完整性（范畴 4）的双重数据结构修改路径（L2）？
- 是否对 God Class（范畴 5）做了职责分组分析（L2）？还是仅统计了方法数（L1）？
- 是否追踪了需求一致性（范畴 6）的调用方来验证声明-实现不一致（L2）？

**完成范畴 7-9 后 — 第三段自检**：
- 是否实际查看了技术风险（范畴 7）的依赖版本信息？
- 是否逐文件审视了设计取舍（范畴 8）的迹象？
- 是否逐项审视了模式诊断（范畴 9）的设计原则违反？

如某范畴在 L2 深度下确实无需执行某些检查项（如项目无多线程代码则范畴 3 无法执行），
在步骤 13 摘要中明确说明："范畴 3：N/A — 项目无多线程代码"。
如某范畴因 token 不足未完成 → 在步骤 13 摘要中明确标注 "范畴 N 未完成（token 耗尽于第 X 步）"，
不可用 "未发现明显问题" 掩盖未执行的事实。

### 步骤 3：约束合规检查

对步骤 1 中读取的项目特定约束（AC/DP/CI），逐条在代码中检测违反情况：

1. 对每条约束，在代码中搜索违反模式（CI 类 grep 可自动执行，AC/DP 类需上下文判断）
2. 违反 → 写入 04，格式为：
   ```
   CD-XXX: 违反 [约束编号] — [约束描述]
   位置：[符号锚点]
   修复建议：[具体]
   ```
3. 引用约束编号（如 CI-001），使问题与规则可追溯

> ENV 约束不在此步骤检测——由 sdd-apply reviewer 审查工具调用记录。

### 步骤 4：搜索清单执行自检

全部范畴检测完成后，逐范畴自检：

- [ ] 范畴 1（代码残留）：是否对每个检查项实际执行了 grep/AST？还是仅凭印象？
- [ ] 范畴 2（硬编码）：是否检查了跨文件重复（L2）？
- [ ] 范畴 3（并发安全）：是否逐文件追踪了全局变量的读写点（L2）？——这是最容易被"象征性执行"的范畴
- [ ] 范畴 4（数据完整性）：是否追踪了双重数据结构的修改路径（L2）？
- [ ] 范畴 5（架构合规）：是否对 God Class 做了职责分组分析（L2）？还是仅统计了方法数（L1）？
- [ ] 范畴 6（需求一致性）：是否追踪了调用方来验证声明-实现不一致（L2）？
- [ ] 范畴 7（技术风险）：是否实际查看了依赖版本信息？
- [ ] 范畴 8（设计取舍）：是否逐文件审视了取舍迹象？——这是最容易被整范畴跳过的检查
- [ ] 范畴 9（模式诊断）：是否逐项审视了设计原则违反？

如某范畴在 L2 深度下确实无需执行某些检查项（如项目无多线程代码则范畴 3 无法执行），
在步骤 7 摘要中明确说明："范畴 3：N/A — 项目无多线程代码"。
如某范畴因 token 不足未完成 → 在步骤 7 摘要中明确标注 "范畴 N 未完成（token 耗尽于第 X 步）"，
不可用 "未发现明显问题" 掩盖未执行的事实。

### 步骤 5：从高层级到低层级逐层梳理
不按分类平铺，而是以**问题为线索**：
1. 先识别架构层根因（God Class、循环依赖、单例扩散……）
2. 向下追踪每个根因在各层级的表现（AV → DD → MT → CD）
3. 关联为问题组，每个组有一个根因条目 + 各层级的子条目

例：
```
DD-004: 全局消息队列设计缺陷（根因——架构层）
  ├─ AV 视角: extern 全局变量跨 4 个编译单元 → 隐式耦合
  ├─ MT 视角: CMsgThread 跨线程追加，CListModel 读端无锁
  ├─ CD 视角: QMutex 裸 new 无 RAII 包装
  └─ 修复: 将 g_msgVec + g_pMutex + g_stopMsgThread 封装为 MessageQueue 类
```

### 步骤 6：消费行为-问题线索映射（仅被 reverse 流程的阶段6调用时）

当诊断被 reverse 流程的阶段6调用，且存在「行为-代码-问题线索映射」时：
- 对映射中的每条线索，验证是否已被诊断覆盖：
  - 已覆盖 → 在对应的04条目中追加 `（对应 B-XXX）` 引用
  - 未覆盖 → 补充对应的 FD/DR 条目
- 对映射中未涉及的 B-XXX「不完整」条目（如果映射不全），
  遍历 01-行为目录.md 补全验证

### 步骤 7：全局严重程度标准化
收集所有发现后，横向对比，统一校准严重程度：
- 跨层级影响 + 无当前规避 → 高
- 单层级影响但有明确的运行时触发场景 → 中
- 代码质量问题、无运行时触发场景 → 低

### 步骤 8：缺陷确定性判断

对每个标记为"缺陷"的发现，确定其为"确定缺陷"还是"疑似bug"：

**确定缺陷**（标注为 FD-XXX，不加限定词）：
- 代码逻辑存在明确错误，且更正方案非歧义
- 例：`strAbsExeName = "_d"` 应为 `+= "_d"` — 赋值vs追加是明确的逻辑错误
- 例：空函数体 + 函数名含动作词（pause/stop/delete）— 明确的功能缺失

**疑似bug**（标注为 FD-XXX 并追加 `⚠️ 疑似`）：
- 代码行为与预期不符，但无法从代码确认是故意设计还是无意错误
- 例：函数忽略传入的 num 参数 — 可能是设计意图（"尚未实现分页"），也可能是bug
- 例：轮询替代事件驱动 — 可能是性能优化意图，也可能是未正确使用框架

### 步骤 9：跨条目严重程度一致性检查

收集所有发现后，检查同类问题的严重程度是否一致：

1. 找出所有同类型问题（如：都是"全局可变状态"的条目）
2. 对比它们的严重程度标注
3. 如果有两处严重程度不同 → 必须在备注中说明差异理由
   - 例："DD-002 高 vs CD-010 中 → 差异理由：DD-002 跨4个编译单元共享，
     CD-010 仅单个文件内使用"

如果不写差异理由 → 统一为两者中较高的严重程度

### 步骤 10：优先级计算

**⛔ 强制步骤：现在读取 Priority 计算标准。**

1. 优先读取当前项目中的 `.ai/doc/04-问题与改进.md` 顶部「📋 Priority 计算标准」区块
2. 如项目 04 文件中无此区块 → 读取 `skills/ai-sdd-init/templates/doc/04-问题与改进.md` 模板
3. 如模板也无法访问 → 暂停并报告 "无法获取 Priority 计算标准，诊断无法继续"

该区块是 Priority 的**唯一权威源**，包含：
- 三维度操作化表（严重性 × 修复成本 × 影响范围，各 1-5 级）
- 色标定义（🔴 P>80 / 🟠 P>50 / 🟡 P>20 / 🟢 P≤20）
- 硬约束规则

读取后回到此处，对每个问题条目按该标准评估三个维度，计算 priority。
Priority 在步骤11（去重与合并）之前计算。合并后的条目取各来源的最高 priority。

**Priority 用于排序，不用于筛选。**
所有有效发现（有符号锚点、有具体描述）无论 priority 多低都必须记录。
Priority = 1 的问题排在最后即可，不意味着"不值得记录"。

### 步骤 11：去重与合并
- 按 `(file:line, 问题类型)` 做 key，与 04 已有条目去重
- 与观察便签交叉验证：便签中的线索是否已被覆盖？如有遗漏 → 补充分析

去重后检查：是否存在某范畴被合并到 0 条的情况？
如有 → 检查是否因过度合并导致。列出被合并的原始条目，逐条判断：
  - 合并合理 → 保留合并结果
  - 合并不合理 → 拆分回独立条目
不设硬性下限。问题数量由代码实际决定。

### 步骤 12：写入 04
- 追加新条目到对应子节
- 每个条目标题行包含 priority 标记（如 `### AV-003: 3DES密钥硬编码 🔴 P=75`）
- 各章节（AV/DR/FD/VR/MT/DD/CD）内部，条目按 priority 降序排列（非编号顺序）
- 编号保持不变（AV-001、AV-002...），仅调整出现顺序
- 关联条目在描述中互相引用（如 `相关: MT-002 是本文档 DD-004 在模块层的表现`）
- 更新 front matter（`last_updated`, `status`）
- 删除 project-log.md 中的「诊断观察便签」区段

### 步骤 13：返回结构化摘要
```
=== 诊断完成 ===
架构层: 新增 N 项（AV-X, DD-X），去重跳过 M 项
需求层: 新增 N 项（DR-X, FD-X）
技术层: 新增 N 项（VR-X）
设计层: 新增 N 项（MT-X, DD-X, CD-X）

待人类确认:
  - UC-001: [具体问题] — 验证方式: [怎么做]
  - UC-002: [具体问题] — 验证方式: [怎么做]
```

## 输出质量规则

### "待确认"标记的使用限制

允许使用"待确认"的条件（必须同时满足）：
1. 已阅读了所有相关的代码路径（调用方、被调用方、错误处理分支）
2. 代码中确实不存在对该场景的处理逻辑
3. 无法从代码推断该场景是否会实际发生

禁止使用"待确认"的情况：
- 只因为没读完相关代码就标注 → 必须先完成代码阅读
- 同一发现中如存在多个"待确认" → 检查每个"待确认"是否满足上述三条件。如不满足 → 补充分析后移除"待确认"标记。满足条件即保留，不因数量而调整严重程度

当标注"待确认"时，必须附带：
- 已检查的代码路径（文件:行号列表）
- 为什么代码无法回答这个问题
- 要确认需要什么信息或验证方式

### 与其他 Skill 的协作

- **ai-sdd-check** 发现严重漂移时 → 建议运行深度审计对应层级
- **sdd-archive** 归档前 → 自动调用（layer=ALL, depth=L1）检查是否引入新问题
- **ai-sdd-rebuild** 重建后 → 调用（layer=ALL, depth=L2）重新诊断

## 💡 常见陷阱（Gotchas）

- **质量高的代码诊断输出少是正常的**：不要为了有产出而制造问题。空结果 ≠ 扫描不充分。如果代码质量确实高，04 可以只有少量条目。
- **观察便签只是线索补充，不能替代搜索清单**：如果 diagnose 产出的问题全部来自便签而没有独立发现，视为执行失败。便签告诉你"可能遗漏了什么"，不是"可以跳过搜索清单"。
- **L3 深度在大型项目上可能消耗大量 token**：建议在独立会话中运行 L3 深度扫描，避免因 token 耗尽导致后几个范畴的检查质量下降。
- **范畴 3 的跨线程追踪是最容易被跳过的 L2 检查**：它不产生 grep 命中，需要逐文件理解并发模型。如果时间或 token 有限，优先保证范畴 1/6/7/3。

## 独立调用时的前置检查
